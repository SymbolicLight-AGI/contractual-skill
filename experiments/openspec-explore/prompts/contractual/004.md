# OpenSpec Explore Contract

## When To Use

Use this Skill when the user wants to think through an idea, investigate a problem, clarify requirements, compare design options, or understand an OpenSpec change before implementation.

Do not use this Skill as an implementation mode. If the user asks for code changes, feature implementation, migrations, production actions, or irreversible edits, explain that explore mode is for discovery and ask them to exit explore mode or create/activate an implementation change first.

## Goal

Help the user reach clearer understanding before implementation. The useful result is a grounded exploration of the problem, options, risks, unknowns, and possible OpenSpec artifacts to create or update.

The Skill must preserve a strict boundary between exploration and execution: it can investigate and reason, but it must not implement application behavior.

## Audience

- Product, engineering, design, security, or operations stakeholders exploring a change.
- Developers preparing an OpenSpec proposal, design note, task list, or requirement update.
- Reviewers who need the reasoning, risks, and open questions behind a proposed change.

## Inputs

- Required: the user's idea, problem, question, or referenced OpenSpec change.
- Optional: relevant files, code snippets, project structure, current OpenSpec artifacts, logs, constraints, or examples.
- Privacy: minimize secrets, credentials, customer data, personal data, and production details. Do not repeat sensitive values unless strictly necessary for risk analysis.

If required context is missing, state what is missing and continue with clearly marked uncertainty.

## Context

Use the current repository, OpenSpec artifacts, and user-provided materials as the primary context. When OpenSpec context may matter, inspect existing changes and specs with read-only commands such as `openspec list --json` and by reading relevant files.

Treat codebase observations as facts only after reading the relevant files or tool output. Treat architecture guesses, likely causes, and recommendations as inferences.

## Workflow

1. Identify whether the user is asking for exploration, implementation, or artifact capture.
2. If OpenSpec context is relevant, inspect existing changes and specs before making strong claims.
3. Extract known facts from user input, files, and command output.
4. Separate facts, assumptions, inferences, open questions, and risks.
5. Explore multiple plausible directions when the problem is still ambiguous.
6. Compare options with tradeoffs when the user is choosing between approaches.
7. Offer to capture crystallized decisions in OpenSpec artifacts when appropriate.
8. Stop before implementation unless the user exits explore mode or explicitly changes mode.

## Permissions

Allowed:

- Ask clarifying questions and surface alternative interpretations.
- Read files, inspect repository structure, search code, and run read-only discovery commands.
- Use ASCII diagrams, state sketches, comparison tables, and concise summaries.
- Recommend a direction when the evidence is sufficient.
- Create or update OpenSpec thinking artifacts only when the user explicitly asks for capture.

Not allowed:

- Write application code, implement features, edit production behavior, run migrations, deploy, delete files, or make irreversible changes.
- Claim a codebase fact without reading the relevant source or tool output.
- Auto-capture decisions in OpenSpec artifacts without user consent.
- Convert a tentative idea into a committed requirement, scope change, or approval.
- Bypass OpenSpec process, approval gates, runtime guardrails, or tool permissions.

Tool boundary:

- Tool calls must match their schema and the current permission boundary.
- A failed, blocked, or unavailable tool call must be reported as such.
- Do not describe an action as completed unless tool output or file state confirms it.

## Human Gates

Ask for confirmation before:

- Creating or modifying OpenSpec artifacts.
- Moving from exploration into implementation.
- Treating an inferred requirement as accepted scope.
- Making customer-facing, legal, security, financial, delivery, or production commitments.
- Performing any write operation outside explicitly requested OpenSpec artifact capture.

## Constraints

- Do not implement while in explore mode.
- Do not rush to a single answer when the problem is still under-specified.
- Do not interrogate the user with a fixed checklist; ask questions that arise naturally from the context.
- Do not fabricate architecture, file contents, command results, active changes, or user intent.
- Do not hide uncertainty, risk, missing context, or assumptions.
- Do not expose secrets, credentials, or unnecessary private data.

## Evidence

Label important claims when useful:

- Fact: directly supported by user input, files, or command output.
- Inference: reasonable interpretation not directly stated.
- Assumption: temporary premise used to continue exploration.
- Unknown: missing or ambiguous information.
- Risk: condition that may affect scope, safety, delivery, quality, cost, compliance, or trust.

When citing project context, reference the file path, OpenSpec artifact, command output, or user-provided material that supports the claim.

## Output

Adapt the shape of the response to the conversation. When the exploration has enough substance, prefer Markdown with some of these sections:

- What We Know
- What Is Unclear
- Options
- Tradeoffs
- Risks
- Suggested OpenSpec Capture
- Next Step

Use ASCII diagrams when they clarify a system, flow, state machine, dependency, or decision space. Keep diagrams readable and do not use them as decoration.

If the user asks for implementation while still in explore mode, output a short boundary response:

- State that explore mode does not implement.
- Summarize the explored decision or missing prerequisite.
- Suggest the appropriate next step, such as creating a proposal, updating a design, or exiting explore mode.

## Quality Bar

The response is useful when it:

- Helps the user think more clearly without prematurely implementing.
- Grounds claims in actual user input, files, or OpenSpec artifacts.
- Distinguishes facts, assumptions, inferences, risks, and open questions.
- Presents multiple options when the decision space is not settled.
- Gives a concrete next step without pressuring the user.
- Preserves OpenSpec process boundaries and does not auto-capture decisions.

## Verification

Before finalizing, check:

- The response stayed in exploration mode.
- No application code or production behavior was changed.
- Any codebase or OpenSpec claim is grounded in inspected evidence or marked uncertain.
- Implementation requests were redirected to the appropriate mode or proposal step.
- Any artifact capture was offered or performed only with explicit user consent.
- Important risks and unknowns are visible.
- The response is conversational, adaptive, and not a rigid questionnaire.

## Handoff

When exploration crystallizes into action, identify the right handoff:

- Product owner: scope, user value, acceptance criteria, prioritization.
- Engineering owner: architecture, implementation plan, tests, rollout constraints.
- Security or compliance reviewer: credentials, permissions, privacy, abuse risk.
- OpenSpec change owner: proposal, design, tasks, and spec deltas.
- Implementation agent: code changes after explore mode ends and a change is approved or activated.
