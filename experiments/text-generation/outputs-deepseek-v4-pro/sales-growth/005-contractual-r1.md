# 客户画像
- 客户：某装备制造集团
- 核心需求：售后知识库、工单自动分类、与现有 CRM 和备件系统打通
- 管理层关注点：试点 ROI
- 涉及多方数据权限：集团、分公司、服务商

# 已知事实
- 客户明确提出了三个功能模块：售后知识库、工单自动分类、系统对接（CRM 与备件系统）
- 管理层要求看到试点 ROI 验证
- 客户表达了对系统全面对接的倾向：“先把所有系统都接上，再看效果”
- 客户希望报价中覆盖后续所有二次开发
- 数据权限涉及集团、分公司、服务商三方不同角色

# 推断与不确定性
- 推断：客户可能尚未建立明确的阶段性交付标准，存在一次性大集成交付的预期（基于“全接上看效果”）
- 推断：客户内部决策链可能较长，试点 ROI 可能是争取预算的前提（管理层关注）
- 不确定性：未提供现有 CRM 和备件系统类型、品牌、接口能力，集成可行性与工作量无法判断
- 不确定性：未明确试点范围、试点分公司或服务商，具体数据源与数据量未知
- 不确定性：预算范围、决策人、采购时间表尚未提及
- 不确定性：二次开发的具体边界未定义，是仅限于本次需求范围内的功能补全，还是未来持续变更

# 痛点与机会
- 痛点：售后知识分散，工单处理依赖人工分类，效率低且不一致
- 痛点：CRM 与备件系统未与工单系统联动，信息孤岛影响服务响应
- 机会：通过试点快速验证工单自动分类与知识库调用的价值，用 ROI 推动全面立项
- 机会：三方权限设计可作为行业标杆方案，形成复用资产

# 风险信号
- 风险信号（范围与交付）：客户要求“全系统对接再看效果”，存在范围蔓延和验收标准模糊的高风险，很可能导致项目无法收尾
- 风险信号（商务与成本）：要求报价包含全部二开，存在固定总价下无限开发义务的风险，必须转交风险审查与合同约束
- 风险信号（数据合规与安全）：涉及集团、分公司、服务商三方数据权限，数据访问边界、脱敏、隐私合规未明确，可能涉及法律与客户内部合规审批
- 风险信号（依赖与集成）：未知 CRM 和备件系统接口标准，集成失败或性能问题会直接影响整体方案可行性

# 推荐下一步
- 销售可继续推进：与客户确认试点范围和成功标准，明确告知当前阶段不建议“一次全部打通”，争取以最小可行场景（如某一分公司或一条产品线）开展试点；引导客户区分标准功能与二开清单，并解释标准报价与按需二开的模式。
- 不可由销售直接承诺：任何包含所有系统打通的时间表、保证集成的结果；不得同意无条件包含所有二开的报价方式；不得提前给出数据权限完整解决方案的承诺。

# 交接建议
- 必须交接至 **售前方案**：评估现有 CRM 和备件系统接口技术可行性、需客户提供系统信息；设计试点方案与系统架构，明确集成边界与假设条件；输出阶段交付计划供客户对齐预期。
- 必须交接至 **风险审查**：客户要求的“包所有二开”和“全接上看效果”构成固定范围与验收风险，需在报价和合同条款中设置阶段验收、变更控制机制；数据权限三方模型需评估是否涉及客户内部安全审查或法务要求。
- 建议提前预警 **交付经理**：试点后可能快速扩展，需准备数据迁移和上线验收标准；三方权限在交付阶段需要明确的 UAT 和责任矩阵。
- 不建议在缺少以上确认前，直接由销售出具任何含价格、范围、时间承诺的初步方案书。
