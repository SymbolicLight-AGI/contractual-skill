# 结论摘要

不可直接走合同。当前请求缺失关键验收标准、成本核算、信用审批和分阶段交付条款，无法安全签署覆盖全部阶段和所有系统接口的单份合同。需要售前、交付、财务、法务、风险审查介入并补齐信息后，方可研究合同草案。

# 已知事实

- 项目范围：集团级知识库和多 Agent 工作流，含总部知识库、5 个业务 Agent，对接 OA、CRM、售后工单系统。
- 推广计划：试点后推广到 8 家分公司。
- 客户要求：一份合同覆盖全部阶段，报价包含所有系统接口，验收标准后续再定。

# 缺失信息

- 验收标准：未定义如何验收总部的知识库、5 个 Agent、系统接口以及试点效果。
- 分阶段交付里程碑与付款触发条件：未见试点到推广的关键节点、验收门槛和对应的付款安排。
- 对接系统接口的具体规格、数据准备责任、性能要求、第三方依赖（OA、CRM、工单系统接口所有者及改造成本）。
- 试点与推广的成本估算、价格模型及全量接口报价是否含第三方费用和内部人力成本。
- 客户信用情况和应收款历史，无法评估“一份合同覆盖全部阶段”的信用风险。
- 折扣或特别让步的审批依据（如免费接口、无限制定制等是否存在，未明说但不能排除）。
- 8 家分公司的就绪评估（数据、流程、本地化需求、变更管理责任）。

# 财务与合同风险

- 验收缺失风险：验收标准“后续再定”会导致付款义务和交付完成点悬空，易引发回款纠纷和收入确认延迟。
- 范围蔓延风险：单份合同覆盖所有接口和全部推广，若接口需求变化或分公司需求差异大，成本可能剧增且无对等费用变更条款。
- 分阶段覆盖风险：试点尚未执行就锁定全量推广合同，一旦试点失败或需要重大调整，合同可能无法退出，产生大额履约负债。
- 收款与现金流风险：未知付款里程碑，可能形成“先做后付”，导致现金流承压并增加坏账可能。
- 接口依赖风险：第三方接口可能依赖客户或外部系统供应商，交付延迟或接口变更需明确责任归属和额外收费机制。
- 一次性报价风险：所有接口打包报价，缺失接口具体定义和假设条件，容易亏本。

# 不得直接承诺事项

- 不得同意“验收标准后续再定”作为合同签署条件。
- 不得同意一份合同锁定全部试点和推广工作而不设退出条款。
- 不得同意“所有系统接口”包含在报价内，除非完成接口详细定义、第三方成本及人力成本评估。
- 不得在未获信用审批和收款条件核定前，承诺延后付款或无付款触发条件。
- 不得代替法务批准无限责任、SLA、验收和终止条款。
- 不得在未获得完全成本和风险核算的情况下给出最终报价或折扣承诺。

# 交接建议

- 售前/方案：梳理接口对接的明确假设，给出分阶段交付范围和成本基线，提出试点到推广的合同结构建议，识别工作量和第三方依赖风险。
- 交付 PM：评估第一阶段（总部+5 Agent+接口+试点）工作量、工期、数据准备责任，产出可验收的指标草案，确保应力测试和推广条件可衡量。
- 财务：审核收款节点设计、现金流影响、全部阶段增量成本和定价模型；分析接口打包报价的利润边界；确认是否需单设客户信用审批。

- 风险审查：评估“单份合同覆盖全量”的集中风险、试点失败对合同的影响，提出最大敞口控制和决策触发条件。
