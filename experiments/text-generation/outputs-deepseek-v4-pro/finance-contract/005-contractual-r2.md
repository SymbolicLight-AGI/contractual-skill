# 结论摘要

**不能直接走合同。** 当前请求存在多处关键信息缺失与高风险条款，直接签约会导致范围失控、成本敞口、验收无标准、回款无保障等问题。必须经售前、交付、财务、法务、风险审查复核，补齐缺失项并修订风险条款后方可推进。

# 已知事实

- **项目范围（Known term）**：集团级知识库和多 Agent 工作流，包含集团总部知识库、5 个业务 Agent、对接 OA/CRM/售后工单系统，试点后推广至 8 家分公司。
- **合同模式（Known term）**：客户要求一份合同覆盖全部阶段（试点+推广）。
- **报价范围（Known term）**：客户要求报价包含所有系统接口。
- **验收标准（Known term）**：客户明确“后续再定”，当前无任何验收定义。
- **合同对象（Known term）**：集团客户，涉及多系统、多实体。

# 缺失信息

- **Missing term**：验收标准与判定流程（无验收定义，无法判定交付完成，无法触发付款）。
- **Missing term**：各阶段交付范围、完成定义与推广触发条件（试点成果如何评估、8 家分公司推广是否无条件）。
- **Missing term**：所有系统接口的详细清单、技术边界与第三方依赖（OA/CRM/售后工单的对接深度、衍生开发量、数据准备责任）。
- **Missing term**：8 家分公司的各自业务差异、数据就绪状态、推广成本预估与分阶段报价。
- **Missing term**：付款里程碑、比例、发票节点及与交付/验收的绑定关系。
- **Missing term**：客户信用与内部预算审批状态，以及“先全包后推广”可能导致的应收账款风险。
- **Missing term**：知识产权、数据使用权、SLA 承诺、违约责任与终止条款。

# 财务与合同风险

- **Risk**：验收标准后置——形成“无限交付/无限修改”敞口，无法关闭项目并确认收入，易引发收款纠纷。
- **Risk**：一份合同覆盖试点+多维推广——推广阶段范围未定义、成本未知、各分公司环境不明，报价含所有接口等同于固定总价承担不可预见成本，极易亏损。
- **Risk**：报价包含所有系统接口但未限定接口范围、技术栈、第三方费用归属，可能产生大量外部许可或额外研发成本，直接侵蚀利润。
- **Risk**：未设分期付款、未绑定验收与回款节点，可能导致长期垫资、应收账款高企，现金流风险严重。
- **Risk**：合同条款未定，若先行签约可能被迫接受后续由客户单方面提出的验收、SLA 和赔偿条款，法律与交付责任不可控。

# 不得直接承诺事项

- 不得承诺“所有系统接口全包价”在未见接口清单与技术评估前固定。
- 不得同意验收标准后置，必须明确交付物衡量方式。
- 不得将试点与 8 家分公司推广捆绑在同一固定价格合同内，除非逐阶段明确范围和退出机制。
- 不得在没有信用审核与阶段付款保护的情况下同意“先交付后付款”或长周期回款。
- 不得在未完成数据就绪度调研的情况下承诺推广时间线或包入合同。
- 最终条款须由法务审核，任何折扣、付款周期、SLA 均需经过内部审批链。

# 待确认问题

1. 试点阶段的成功标准与量化验收指标是什么？（交付、法务、风险）
2. 推广到 8 家分公司的条件是什么？是否有分阶段退出或重新议价机制？（售前、交付、风险）
3. OA、CRM、售后工单系统的集成边界、当前 API 状况、第三方额外成本分别由谁承担？（交付、售前、财务）
4. 付款结构如何设计？能否将试点、每家分公司推广设为独立里程碑并挂载付款比例？（财务）
5. 客户内部本次采购是否已获预算批复？信用期和账期政策如何？（财务、风险）
6. SLA 承诺、数据所有权、违约责任、终止权利等法律条款必须先行对齐底线要求。（法务）

# 交接建议

- **售前/解决方案**：重新划范围，拆分为试点合同+推广框架协议/增补采购；明确接口清单与含入条件。
- **交付 PM/技术负责人**：输出试点交付计划、推广调研模板、系统对接技术评估，提供基础成本与工期区间。
- **财务**：审核客户信用，制定付款里程碑、发票计划与应收风险控制方案；拒绝无验收绑定的付款节点。
- **法务**：起草或审查合同框架，确保验收标准、SLA、责任限制、终止条款和对“后续再定”条款的限制性语言，防止开放式义务。
- **风险审查/审批委员会**：对高敞口合同类型（预签全部阶段）进行评审，设定可接受的推广承诺上限和止损条件。
