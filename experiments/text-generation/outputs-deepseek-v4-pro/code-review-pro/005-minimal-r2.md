**Code Review 结果**

### 风险评级与详细说明

| 类别 | 严重度 | 问题 |
|------|--------|------|
| **权限控制** | **严重** | 后端 API 未检查角色权限，仅依赖前端按钮隐藏。任何知晓 API 路径的用户均可发起导出，导致未授权访问全部用户敏感数据。 |
| **隐私合规** | **严重** | 导出字段包含姓名、手机号、邮箱、最近订单金额，属于明确个人信息甚至金融属性信息。缺少导出数量限制、用途声明、数据脱敏选项，极易违反 GDPR/PIPL 等法规。 |
| **审计追溯** | **高** | 完全没有审计日志，无法追踪谁在何时导出了哪些用户数据，事发后不可溯源，违反基本安全运营要求。 |
| **性能和资源** | **中** | 无导出数量限制，恶意或误操作可一次性导出全量用户数据，可能拖垮数据库和服务器，导致拒绝服务。 |
| **可维护性** | **低** | 导出逻辑未考虑分页、异步任务，大数据量下容易超时失败，且无失败重试和通知机制。 |
| **测试覆盖** | **高** | 未见针对权限校验、空数据、大数据量、特殊字符（如 CSV 注入）的测试用例描述，存在数据质量和安全测试缺口。 |

### 需继续处理的人员及事项

1. **安全工程师 / 安全负责人**
   - 确认后端必须实现基于角色的访问控制（RBAC），并在 API 层强制执行。
   - 审查导出的数据分类，评估是否属于个人敏感信息，制定脱敏或审批策略。
   - 要求添加导出操作的审计日志（操作人、时间、导出范围、IP、结果）。
   - 评估 CSV/Excel 导出时的公式注入风险，建议对特殊字符做前缀或转义处理。

2. **后端开发负责人**
   - 实现导出数量上限（每日/每月/单次），并封装为可配置项。
   - 将同步导出改为异步任务（消息队列或轮询），并提供进度反馈和失败重试。
   - 增加请求频率限制 (rate limiting) 以对抗滥用。

3. **产品负责人 / 数据保护官 (DPO)**
   - 明确导出功能是否需要用户授权或告知，并评估法律风险。
   - 确定导出数据保留期限和访问范围（如仅允许管理员或特定角色）。
   - 决议是否需要对手机号、邮箱进行掩码处理（如 138****1234）。

4. **测试工程师**
   - 设计测试用例覆盖：无权限调用 API、超量导出、导出空数据集、特殊字符（`=cmd|、+` 等）注入、大数据量导出时系统稳定性、审计日志完整性。
