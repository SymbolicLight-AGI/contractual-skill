# Code Review: 用户数据导出功能

## 整体判断
❌ **高风险，不可合并。** 存在严重的权限缺失、隐私泄露和审计缺失风险，可能导致大规模用户敏感数据未经授权导出。

## 主要发现与风险

### 1. 权限控制缺失（严重）
- **问题**：后端 API 未检查调用者角色，仅依赖前端按钮是否显示来做管理员判断。任何已认证用户或能够绕过前端限制的攻击者均可直接调用导出接口，获取所有用户的姓名、手机号、邮箱和订单金额。
- **影响**：任意登录用户可批量窃取全量用户隐私数据，违反数据最小权限原则，可能触发数据安全事件及法律合规风险（如个人信息保护法）。
- **修复**：后端接口必须强制校验当前用户具备管理员角色（如 `admin` 或特定导出权限），在业务逻辑层做权限判定，绝不信任前端环境。

### 2. 敏感数据暴露（严重）
- **问题**：导出内容包含手机号、邮箱等个人可识别信息（PII）以及消费金额，且无任何脱敏或最小化处理。若导出文件在传输、存储或使用中失控，将直接泄露海量用户隐私。
- **影响**：隐私泄露可能导致用户信任崩塌、监管处罚及民事索赔。消费金额也可推断用户财力，进一步被用于诈骗或黑市交易。
- **修复**：评估导出场景是否必须包含完整明文 PII。如需导出，应在后端实施字段级权限、按需脱敏（如手机号掩码）、强制添加水印（操作人+时间）并限制可导出字段组合。

### 3. 审计缺失（高风险）
- **问题**：整个导出操作无任何审计日志记录（谁、何时、导出了哪些字段、导出数量）。发生数据泄露时无法追溯，无法满足合规审计要求。
- **影响**：事后无法定位泄露源或责任人，增加法律风险和内部追责难度。
- **修复**：增加操作审计日志，记录操作者 ID、IP、时间戳、导出条件、导出记录数、导出的字段范围等，日志不可篡改且长期保留。

### 4. 导出规模无限制（高风险）
- **问题**：未对单次导出的记录数或时间窗口进行限制，可能一次性导出全量用户数据，造成数据库压力或用作拖库。
- **影响**：数据库负载激增，影响线上服务；大量数据一次性外泄，放大了泄露后的损害规模。
- **修复**：增加单次导出数量上限（如最多 10,000 条）、时间范围强制要求、导出频率限制（如每人每天一次），并对大数据量导出转异步任务并审批。

### 5. 测试覆盖缺失
- **问题**：未提及权限测试、边界测试（空数据、超限、异常字符）、审计验证、隐私字段脱敏验证等。
- **影响**：上述风险可能在测试阶段不被发现，直接发布至生产。
- **建议**：增加自动化测试用例，包括：
  - 非管理员调用导出接口返回 403。
  - 超出数量限制的正确拒绝。
  - 导出 CSV/Excel 字段正确性及未包含额外敏感字段。
  - 审计日志写入验证。
  - 性能测试（模拟大导出量）。
