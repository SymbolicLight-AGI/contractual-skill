## 代码审查风险报告

### 一、风险总结
此 PR 存在严重的安全与合规缺陷，不应合并至主分支。主要问题包括：

1. **权限校验缺失（高危）**  
   - 前端按钮虽做了管理员判断，但攻击者可直接调用后端导出 API。  
   - 后端未对请求角色进行校验，导致任何已认证用户（甚至未认证用户，若认证覆盖不全）均可导出全量用户隐私数据。  
   - **风险**：越权访问、用户数据大规模泄露。

2. **隐私合规风险（高危）**  
   - 导出字段包含姓名、手机号、邮箱、最近订单金额，均属个人身份信息与敏感商业数据。  
   - 无任何脱敏、加密切断或用途声明。  
   - 未提供数据主体同意追认机制，可能违反 GDPR/PIPL 等法规。  
   - 无导出数量限制，可一次性拉取全库用户表，扩大泄露面。

3. **审计能力缺失（中高危）**  
   - 导出操作完全未记录审计日志。  
   - 无法追溯谁、在何时、导出了哪些用户数据。  
   - 使得违规导出不可发现、不可追溯，无法满足审计与问责要求。

4. **测试覆盖不足**  
   - 缺少对权限边界的测试（例如普通用户直接调用 API、未登录调用等）。  
   - 无大数据量导出时的性能与超时测试。  
   - 无脱敏策略验证、导出数量上限验证。

### 二、需要明确责任人及后续事项

**1. 安全工程师 / AppSec**  
- 立即在后端 API 增加角色鉴权（需确认只有管理员可操作）。  
- 审查并加固请求接口，防止 CORS 配置不当导致跨站导出。  
- 确认是否需要增加二次认证（如 MFA）来触发导出操作。

**2. 隐私/法务负责人（DPO/合规）**  
- 确认导出字段的必要性以及是否需要进行脱敏或抑制（例如手机号仅显示后四位、邮箱可否置空等）。  
- 明确导出用途、保留策略与用户告知义务。  
- 定义导出数量上限，以及在超过限额时的审批流程。

**3. 后端开发负责人**  
- 实现审计日志，至少记录操作人 ID、时间、IP、导出参数（筛选条件或全量标记）、导出条数、结果状态。  
- 增加导出数量上限（如单次不超过 1000 条，或限时次数）。  
- 优化大数据量导出时的异步任务与流式导出，避免内存溢出或超时。  
- 联调并增加必要的集成测试、权限测试和性能测试。

**4. 产品负责人**  
- 明确导出功能的交互方案（是否支持筛选、是否生成文件后异步下载等）。  
- 定义导出权限申请的流程和异常处理策略。  
- 完善前端提示文案，确保用户知晓数据的敏感性。

### 建议处理流程
- **阻断合并**，直至上述关键风险全部解决并经过安全 review。  
- 成立专项问题单，由上述负责人各自确认并签字各自条目。  
- 在后续版本中加入数据销毀和定期审计报表，防止未受控的导出行为长期存在。
