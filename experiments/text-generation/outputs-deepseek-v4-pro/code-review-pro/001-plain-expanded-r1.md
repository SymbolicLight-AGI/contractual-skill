# Code Review: get_customer_notes

**Overall Judgment**: 该函数存在严重的安全漏洞，必须立即修复后才能上线。

## Main Findings

### 1. SQL 注入（严重）
- **位置**: `query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"`
- **问题**: `customer_id` 直接拼接到 SQL 语句中，且该值来自 HTTP query 参数（用户可控）。任何请求者都可以构造恶意 `customer_id` 来读取、修改或删除数据库中的数据，例如 `1; DROP TABLE notes--`。
- **影响**: 可能导致所有客户备注数据泄露、数据破坏、整个数据库被攻破。
- **风险**: 信息泄露、完整性破坏、可用性受损，严重危害业务和合规性（如涉及 PII）。

### 2. 缺少输入验证与授权（高）
- `customer_id` 未进行类型、范围或格式校验。即使使用参数化查询，也应确保它是合法的整数或预期的标识符格式。
- 未检查当前请求者是否有权访问该 `customer_id` 的备注。若 CRM 仅有简单认证但无细粒度授权，任意登录用户可遍历所有客户备注。

### 3. 错误处理缺失（中）
- `db.execute(query).fetchall()` 未包裹任何异常处理。数据库连接失败、查询超时、语法错误（目前由注入导致）都会抛出异常，可能向调用方暴露内部堆栈信息，或造成静默失败并返回 5xx。
- 若 `row` 对象不是可映射类型，`dict(row)` 可能抛出异常。依赖库实现细节，不够健壮。

### 4. 可维护性问题（低）
- `SELECT *` 使返回的列与表结构强耦合，表结构变更会静默影响调用方。
- 函数名称 `get_customer_notes` 暗示可能只返回某客户的备注，但使用了 `SELECT *`，无分页或数量限制。如果某客户有大量备注，可能拖垮内存和接口响应。

## Impact
- 严重：攻击者可无限制访问所有备注及可能整个数据库。
- 高：缺少授权可能导致大规模信息泄露。
- 中：异常暴露栈信息增加攻击面。

## Test Advice
- 必须添加单元测试覆盖正常查询、无效 ID（非数字、越权）、数据库错误场景。
- 安全测试：验证 SQL 注入 payload（如 `'; DROP TABLE notes; --`）不会执行恶意 SQL。
- 集成测试：验证使用参数化查询后的实际数据库交互。
- 授权测试：确认当前用户只能访问授权范围内的备注。

## Uncertainty
- 不清楚 `db.execute` 是否支持参数化查询（如使用 `?` 占位符的 `execute(query, params)`）。假设其底层是 SQLite/psycopg2 等类似接口，应改为参数化。
- 调用方的身份认证与授权机制未知，需结合项目实际情况实施访问控制。

## Required Handoffs
- **安全团队/后端负责人**：立即修复 SQL 注入，并审查所有类似拼接 SQL 的代码。
- **产品/架构**：明确 `customer_id` 的白名单格式与权限模型。
- **测试团队**：补齐安全测试用例和异常路径覆盖。

*结论：该代码不可直接部署，必须重写为参数化查询，并加上输入校验和授权。*
