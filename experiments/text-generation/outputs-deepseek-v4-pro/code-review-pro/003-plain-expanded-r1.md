# Code Review – load_orders 吞错修复

## 整体判断
**不认可该修改。** 当前实现存在严重的静默失败风险，绝不能合并。

## 主要发现

1. **严重性：高** – 使用 `except Exception` 捕获所有异常并返回空列表，完全掩盖了上游接口的任何失败。
2. **财务对账风险** – 当 `client.fetch_orders` 因网络中断、超时、认证错误、服务端错误或数据异常而抛出异常时，调用方收到空列表，会导致对账任务错误地判定“没有订单”，从而造成财务数据缺失或错账。
3. **不可观测性** – 异常被丢弃，没有任何日志、告警或指标记录。运维人员无法感知故障，故障会持续累积。
4. **破坏调用契约** – 该函数未明确声明失败模式，调用者无法区分“确实无订单”与“查询失败”，使得上游难以做出正确决策。

## 影响
- 财务对账结果错误，可能导致漏记收入、对不上账、审计问题。
- 问题发现延迟，增加排查和修复成本，甚至引发业务纠纷。

## 测试建议
- **不要为通过测试而压制错误。** 应修改测试用例，覆盖 `client.fetch_orders` 抛出各种异常的场景。
- 测试应验证：
  - 对于预期内可恢复异常（如超时），函数是否可以重试或返回携带错误信息的结果。
  - 对于非预期异常，是否向上传播或触发告警，而非返回空列表。
- 若必须保留空列表作为降级策略（不推荐），需在测试中验证日志/告警的触发。

## 处理建议
1. **明确失败模式** – 与产品/后端协同决定对账任务在查询失败时的业务语义：是重试、暂停对账并告警，还是标记为“查询失败”待人工介入。
2. **改进错误处理** – 至少应区分：
   - 临时性故障（网络超时等）→ 可重试或返回部分结果并告警。
   - 永久性故障（权限错误、数据格式错误）→ 抛出或返回明确错误，阻止对账继续。
   - 示例结构（示意）:
     ```python
     def load_orders(client, user_id):
         try:
             return client.fetch_orders(user_id)
         except TemporaryFailure as e:
             logger.error("transient error fetching orders", user_id=user_id, exc_info=True)
             raise  # 或返回带错误信息的专用类型
         except PermanentFailure as e:
             logger.exception("permanent error fetching orders", user_id=user_id)
             raise
     ```
3. **增加可观测性** – 捕获异常时必须记录日志，包含 `user_id` 和异常详情（不记录敏感信息），并设置监控告警。

## 不确定性
- 未提供 `client.fetch_orders` 可能抛出的具体异常类型，因此无法给出精确的分类处理。需查阅其文档或代码。
- 业务方对“对账无订单”的容忍度未知，需产品确认策略。

## 需转手跟进
- **产品/后端负责人**：明确查询失败时的对账行为与用户体验（是否允许对账跳过、是否要求人工介入）。
- **测试/QA**：确保测试覆盖异常路径，不再让空列表掩盖故障。
- **SRE/运维**：配置相应告警规则，监控该函数的错误率。
