# 代码审查报告

## 总体判断

**高风险发布阻断**  
该导出功能存在严重权限缺失、隐私泄露、审计盲区及滥用风险，在未完成以下多项安全与产品确认前，禁止合并或上线。

## 主要问题

1. **未实施后端授权检查**（发现）  
   前端仅通过按钮显隐进行管理员判断，后端API未校验调用者角色，任何已认证用户（甚至未认证用户若API未强制认证）均可直接调用导出接口，越权获取全量用户的姓名、手机号、邮箱、订单金额。

2. **个人隐私数据未经保护导出**（发现）  
   导出字段包含可直接标识个人的信息（姓名、手机号、邮箱）及财务相关数据（最近订单金额），且无任何脱敏、加密或最小化处理，违反数据最小化原则与隐私合规要求。

3. **无审计日志**（发现）  
   导出操作未记录任何审计日志，无法追踪谁在何时导出了哪些数据，无法满足事后审计、异常检测及合规需求。

4. **无导出数量限制**（发现）  
   接口未设置单次导出条数上限或频率限制，可能被恶意或错误调用用于批量窃取整个用户库数据，造成大规模数据泄露。

5. **缺少必要的安全与产品策略**（不确定性）  
   未提供导出功能的业务目的、导出范围是否经过产品确认、是否需用户同意或通知、数据保留策略等上下文，可能违反GDPR/个人信息保护法带来的合规风险。

## 风险与影响

- **权限绕过**  
  攻击者或低权限用户可遍历导出所有用户隐私与财务数据，导致全量个人信息泄露。
- **隐私泄露与合规风险**  
  导出数据可轻易外泄，触发数据保护法规（如个保法、GDPR）的报告义务和罚款，损害用户信任。
- **滥用与资源耗尽**  
  无限制导出可被用于拖库，或高频调用消耗系统资源，影响线上服务稳定性。
- **不可审计**  
  发生泄露后无法溯源，妨碍事件响应与责任界定。

## 测试建议

- **权限测试**：构造普通用户、未登录请求、不同角色Token调用导出API，验证是否被拒绝（预期403/401）。
- **隐私保护测试**：检查导出文件中是否含明文手机号、邮箱，确认是否需要脱敏或仅提供哈希/部分掩码。
- **审计日志测试**：调用导出接口后查看审计存储是否记录了时间、操作人、IP、导出条数、导出范围等关键字段。
- **限制测试**：请求极大页码或巨大数量参数，验证是否被限制（如最大1000条或分页强制上限），并发测试频率限制。
- **业务场景测试**：与产品确认合法导出场景后，编写端到端测试验证仅导出必要字段和数据范围。

## 不确定性

- 缺少仓库代码上下文，无法确认后端API是否已强制登录；若无强制登录，则风险升级为未认证即可导出。
- 不确定导出功能是否涉及GDPR/个保法中“数据主体的携带权”还是内部运营需求，需产品明确法律依据与同意机制。
- 不清楚前端“管理员判断”的具体实现（基于角色还是简单本地标记），但无论如何均不可依赖前端作为安全控制。
- 未提供是否已有审计基础设施，假设无日志则需从零搭建或接入已有系统。

## 交接建议

- **安全负责人**：  
  - 审查并强制后端实施基于角色的访问控制（RBAC），仅允许经审批的管理员角色访问导出API。  
  - 评估导出字段的必要性，推动敏感字段脱敏（如邮箱掩码、手机号哈希或部分隐藏）或完全移除不必要的PII。  
  - 定义并实施审计日志规范，确保所有导出操作留痕。  
  - 制定并实施速率限制和单次最大导出条数策略，防止批量抽数。  
- **后端负责人**：  
  - 在API层增加授权中间件，移除对前端可控因素的依赖。  
  - 实现导出数量上限及频率限制逻辑。  
  - 接入审计日志记录功能，确保不可篡改。  
- **产品负责人**：  
  - 确认导出功能的用户故事、合法使用场景、导出数据范围和保留期。  
  - 决定是否需要用户同意、导出记录告知或事后通知机制。  
  - 明确数据导出是否符合数据最小化原则，是否可去掉邮箱/手机号等直接标识符。  
- **QA/测试负责人**：  
  - 针对上述测试建议编写安全测试用例和回归测试，并作为发布门禁。
