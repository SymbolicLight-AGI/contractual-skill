## 总体判断
高风险。核心缺陷是 SQL 注入，通过 HTTP 参数直接拼入查询，攻击者可以读/写任意数据。缺少输入验证、错误处理，且返回整表可能暴露多余字段。当前无测试覆盖，无法验证任何行为。

## 主要问题
1. **SQL 注入 (Correctness / Security)**  
   `query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"`  
   `customer_id` 直接源自 HTTP query 参数，未做任何转义或参数化。攻击者可通过构造参数值执行任意 SQL。
2. **返回整行 (`SELECT *`)**  
   暴露 `notes` 表的全部列，可能包含内部标识、时间戳、作者、权限标记等不应返回给调用方或前端的数据。
3. **缺少输入验证**  
   未检查 `customer_id` 是否为预期类型（例如正整数）。接受任意字符串，加剧注入风险。
4. **缺少错误处理**  
   `db.execute(…).fetchall()` 未捕获异常。数据库故障、连接超时或语法错误会导致未处理异常，泄露内部堆栈或造成服务不可用。
5. **无测试**  
   没有任何单元测试或集成测试，无法保证功能正确性或防止回归。

## 风险与影响
- **数据泄露/篡改/删除**：攻击者通过注入修改查询条件或执行联合查询，读取其他客户备注，甚至执行 `UPDATE`/`DELETE`（取决于数据库权限）。
- **信息泄露**：`SELECT *` 可能送出内部审计字段、用户身份、时间戳等，违反最小权限原则。
- **服务中断**：未处理的异常可能导致请求失败、连接泄漏或 500 错误，影响 CRM 可用性。
- **合规风险**：若备注含个人数据，此漏洞可能违反数据保护法规。

## 测试建议
- **参数化查询测试**：验证当 `customer_id=1` 时，SQL 使用占位符，不应拼接值。
- **安全测试**：传入 `1 OR 1=1`、`1; DROP TABLE notes;--`、`1 UNION SELECT ...` 等注入 payload，期望查询不执行恶意操作（参数化时这些 payload 仅视为字符串）。
- **输入验证测试**：传入非数字、负数、超长字符串，期望返回错误或空结果，不导致异常。
- **错误路径测试**：模拟数据库不可用或查询超时，验证函数优雅返回错误或由上层统一处理。
- **返回字段测试**：检查实际返回的字段，确保不包含无关列，建议明确列出所需列。
- **回归测试**：修复后，使用已知 `customer_id` 验证返回记录正确。
