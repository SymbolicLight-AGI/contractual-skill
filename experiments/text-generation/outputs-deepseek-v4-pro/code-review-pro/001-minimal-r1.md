**安全问题（严重）**
- **SQL 注入**：`customer_id` 直接通过 f-string 拼入 SQL 语句，参数来源于 HTTP query 参数，攻击者可构造恶意输入（如 `1 OR 1=1; DROP TABLE notes;--`）执行任意 SQL。必须改用参数化查询（如 `cursor.execute("SELECT * FROM notes WHERE customer_id = ?", (customer_id,))`）。

**正确性风险**
- **类型与格式**：`customer_id` 来自 HTTP 参数，可能是任意字符串，即使无注入也可能因类型不匹配导致数据库错误或意外返回空集。需进行类型校验/转换（如期望整数则 `int(customer_id)` 并捕获异常）。
- **隐式依赖**：`db.execute().fetchall()` 返回的行对象未说明是何种类型（可能为 sqlite3.Row、元组等），在结果处理上须确保 `dict(row)` 可转换，否则会运行时出错。

**可维护性/设计问题**
- 缺少文档字符串，未说明参数类型、返回值结构与异常情况。
- 使用 `SELECT *` 不利于后续字段变更，应显式列出所需列。
- 缺乏输入验证日志或错误处理，内部函数直接暴露原始错误不利于排查。

**测试建议**
- 补全单元测试，至少覆盖以下场景：
  1. **正常整数 customer_id**：验证返回正确的备注列表。
  2. **不存在的 customer_id**：返回空列表，不抛异常。
  3. **恶意 payload**：传入 `"1; DROP TABLE notes;--"` 等，当前代码会执行注入，完成参数化修复后应验证不执行恶意操作且查询正常/空。
  4. **类型错误**：传入非数字字符串，验证服务能返回 400 或合适错误而非 500。
  5. **数据库异常**：模拟 `db.execute` 抛出异常（如连接中断），检查调用方是否妥善处理。
- 建议使用内存数据库或 mock 隔离外部依赖，使测试快速且可重复。

**其他注意事项**
- 若 `customer_id` 预期为整数，应在函数入口进行强制转换和校验，并在上层捕获 `ValueError`，避免将无效输入传给数据库。
