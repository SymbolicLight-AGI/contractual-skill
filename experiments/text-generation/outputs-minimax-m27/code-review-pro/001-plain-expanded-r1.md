# Code Review: `get_customer_notes`

## 总体评价

**严重风险 — 高危缺陷**

该代码存在 **SQL 注入漏洞**，在内部 CRM 系统中处理外部输入时直接拼接到 SQL，属于安全红线问题。

---

## 主要发现

### 1. SQL 注入漏洞 — **严重 (Critical)**

**位置:** 第 2 行

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

**问题:** `customer_id` 来自 HTTP query 参数，直接拼接到 SQL 语句中。攻击者可构造恶意输入如 `1; DROP TABLE notes; --` 执行任意 SQL。

**影响:**
- 数据泄露：攻击者可通过 `UNION` 注入获取其他表数据
- 数据篡改：插入/修改/删除记录
- 数据破坏：`DROP TABLE` 导致永久数据丢失
- 权限提升：若 DB 用户权限过高，危害扩大

**建议修复:** 使用参数化查询（PDO prepared statement）或 ORM 方法。

---

### 2. 输入验证缺失 — **中等 (Medium)**

**位置:** 函数签名

**问题:** `customer_id` 未进行类型校验或范围验证。

**影响:** 传入非整数（如字符串、特殊字符）可能导致数据库错误或异常行为。

**建议:** 添加类型检查和输入验证。

---

### 3. `SELECT *` 过度获取 — **低 (Low)**

**位置:** 第 2 行

**问题:** 返回所有列而非业务所需的具体字段。

**影响:**
- 增加网络传输开销
- 可能暴露额外字段（如内部 ID、时间戳）
- 表结构变更时行为不可预期

**建议:** 明确列出所需字段。

---

### 4. 异常处理缺失 — **低 (Low)**

**位置:** 全函数

**问题:** 无 try-catch，数据库异常直接抛出。

**影响:** 生产环境中可能返回未处理的堆栈信息，泄露数据库结构细节。

---

## 测试建议

### 高优先级

1. **安全测试 — 必须覆盖**
   - SQL 注入 payloads 测试：
     ```python
     test_inputs = [
         "1; DROP TABLE notes; --",
         "1 OR 1=1",
         "1 UNION SELECT * FROM users--",
         "' OR '1'='1"
     ]
     ```
   - 验证所有输入均被正确转义或拒绝

2. **边界条件测试**
   - `customer_id = 0`
   - `customer_id = -1`
   - `customer_id = None`
   - `customer_id = ""`
   - `customer_id = float`（如 `1.5`）
   - 极大整数（如超出 INT 范围）

3. **权限测试**
   - 验证用户只能访问自己有权限的客户数据
   - 测试越权访问场景

### 中优先级

4. **正常流程测试**
   - 有效的 `customer_id` 返回正确笔记
   - 返回数据结构验证（字段名、类型）
   - 空结果集处理

5. **性能测试**
   - 大数据量场景响应时间
   - 数据库连接池压力测试

## 不确定性

1. **业务权限模型未知** — 是否需要验证 `customer_id` 属于当前登录用户？
2. **数据库类型未知** — 不同数据库参数化查询语法可能不同
3. **`db` 对象实现未知** — 是否已有内置 SQL 注入防护机制？
4. **返回字段范围** — 不确定哪些字段属于客户备注的业务语义
