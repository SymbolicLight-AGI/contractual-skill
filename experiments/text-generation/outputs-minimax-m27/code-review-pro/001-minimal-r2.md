## Code Review: `get_customer_notes` 函数

### 严重风险 (Critical)

| 风险 | 描述 | 影响 |
|------|------|------|
| **SQL 注入漏洞** | 直接使用 f-string 拼接 `customer_id` 到 SQL 中 | 攻击者可通过构造恶意输入执行任意 SQL 语句，可能导致数据泄露、数据篡改或数据库破坏 |
| **信息泄露风险** | `SELECT *` 返回所有字段 | 可能暴露内部字段或敏感元数据 |

### 高风险 (High)

| 风险 | 描述 |
|------|------|
| **输入验证缺失** | 未验证 `customer_id` 类型是否为整数，未处理 None 或负数 |
| **错误处理缺失** | 无 try-except，数据库异常直接暴露给用户 |
| **无权限校验** | 未检查当前用户是否有权访问该客户的备注 |

### 中风险 (Medium)

| 风险 | 描述 |
|------|------|
| **性能问题** | `SELECT *` 检索不必要的字段，增加网络和内存开销 |
| **可维护性差** | SQL 硬编码，难以重构或添加新过滤条件 |

## 测试建议

### 必须覆盖的测试用例

```text
1. 正常场景
   - 有效的 customer_id 返回对应的备注列表

2. 安全测试
   - SQL 注入尝试: customer_id = "1; DROP TABLE notes; --"
   - 注入字符串: customer_id = "' OR '1'='1"

3. 边界条件
   - customer_id = 0
   - customer_id = -1
   - customer_id = None
   - customer_id = "invalid"
   - customer_id = 999999999 (不存在的客户)

4. 错误处理
   - 数据库连接失败
   - 查询超时

5. 权限测试
   - 跨用户访问 (用户 A 尝试访问用户 B 的备注)
```

### 建议测试方法

| 测试类型 | 建议工具 |
|----------|----------|
| 单元测试 | pytest + unittest.mock (mock db) |
| SQL 注入测试 | 参数化查询重构后验证注入无效 |
| 集成测试 | 使用测试数据库 |

---

## 总结

**最高优先级**: 修复 SQL 注入漏洞 — 必须改用参数化查询 (prepared statement)。
