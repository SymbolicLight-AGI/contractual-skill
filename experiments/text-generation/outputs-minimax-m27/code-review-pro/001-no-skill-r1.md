# 代码审查报告

## 主要问题

### 1. 严重安全风险：SQL 注入漏洞

**问题位置**: 第 2 行

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

**风险描述**: 
- `customer_id` 直接拼接至 SQL 语句，完全未做转义或校验
- 攻击者可构造恶意输入，如：`1; DROP TABLE notes;--`
- 内部 CRM 系统若暴露于网络，将成为严重攻击面

**建议**: 使用参数化查询（Prepared Statements）替代字符串拼接。

---

### 2. 正确性问题

| 问题 | 说明 |
|------|------|
| 使用 `SELECT *` | 应明确指定返回字段，避免 schema 变化导致问题 |
| 无类型校验 | `customer_id` 可能为字符串/None，需显式类型转换 |
| 缺少错误处理 | 数据库异常时无降级或日志记录 |

---

### 3. 可维护性问题

- SQL 语句分散在业务逻辑中，难以复用和维护

---

## 测试建议

### 必须覆盖的测试用例

| 测试场景 | 预期行为 |
|----------|----------|
| 合法 `customer_id` (如 `123`) | 返回该客户的备注列表 |
| 非数字输入 (`"abc"`) | 抛出异常或返回空列表 |
| SQL 注入尝试 (`"1; DROP TABLE notes;--"` ) | 参数化查询应将其作为字面值处理，不执行注入代码 |
| `customer_id` 为 `None` | 抛出异常或返回空列表 |
| 不存在的客户 ID | 返回空列表 |
| 数据库连接失败 | 抛出预期异常，不返回裸错误 |

### 推荐测试策略

1. **安全测试**：使用常见注入 payload 验证防护有效性
2. **边界测试**：极大数值、负数、空字符串
3. **集成测试**：验证实际 SQL 执行结果（使用测试数据库）
