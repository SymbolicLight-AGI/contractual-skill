## 代码审查结果

### 主要问题

**1. 严重安全风险 — SQL 注入漏洞**

```
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

直接拼接用户输入到 SQL 语句，攻击者可通过构造 `customer_id=1 OR 1=1--` 等payload实现未授权数据访问或数据破坏。

**2. 缺乏输入验证**

- 未校验 `customer_id` 类型（期望整数）
- 未处理空值或异常值情况

**3. 异常处理缺失**

- 数据库连接失败、查询超时等场景无降级处理
- 调用方无法区分错误类型

**4. 使用 SELECT \***

- 返回字段不可控，可能暴露敏感列
- 性能隐患（未来表结构变更影响功能）

---

### 测试建议

| 测试场景 | 预期结果 |
|---------|---------|
| 正常 `customer_id`（整数） | 返回匹配的 notes 列表 |
| SQL 注入尝试（如 `1 OR 1=1`） | 抛出异常或返回空，而非全部记录 |
| 非整数输入（如 `"abc"`） | 抛出 TypeError 或 ValidationError |
| 不存在的 `customer_id` | 返回空列表 |
| 数据库连接超时 | 抛出可捕获的数据库异常 |
| `customer_id=None` | 抛出异常或参数错误响应 |

**建议补充**：参数化查询重构后，需验证参数化查询实际生效（可检查生成的 SQL 不含字符串拼接）。
