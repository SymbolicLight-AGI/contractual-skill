## 主要问题

### 1. 密钥泄露 (Severity: Critical)

- **Finding**: `api_key: "FAKE_PAYMENT_TOKEN_FOR_SYNTHETIC_TEST_ONLY"` 是支付平台的 live 密钥，直接明文写在配置文件中并提交到仓库。
- **Risk**: 密钥一旦进入 git 历史，即使后续删除仍可被任何人通过 `git log` 或第三方平台获取，导致支付账户被非法使用或产生财务损失。
- **Impact**: 该密钥在 `git history` 中永久留存，需立即轮换。

### 2. 缺少密钥管理机制

- **Finding**: 未使用环境变量、 Vault 或其他密钥管理方案存储敏感信息。
- **Risk**: 密钥硬编码增加了泄露面，无法配合 CI/CD 密钥注入流程。

### 3. 缺少 Secret 扫描

- **Finding**: 项目声明无 secret 扫描工具（如 GitGuardian、Trufflehog、gitleaks 等）。
- **Risk**: 即使此密钥被轮换，未来新引入的密钥或凭证仍可能泄露。

## 风险与影响

| 影响面 | 具体风险 |
|--------|----------|
| **安全** | 密钥泄露可能导致未授权支付操作、财务损失 |
| **合规** | 可能违反 PCI DSS 等支付安全标准 |
| **运营** | 需紧急轮换密钥，影响服务连续性 |
| **供应链** | 密钥存于代码中，可能被 Fork 项目的用户利用 |

## 测试建议

1. **静态扫描测试**：集成 `gitleaks` 或 `trufflehog` 到 pre-commit / CI，扫描每次提交是否包含新密钥。
2. **配置验证测试**：验证配置加载时 `api_key` 字段必须从环境变量读取，直接硬编码应导致测试失败。
3. **回归测试**：修复后确认 `git log` 中历史密钥记录已被清理或确认无法访问。
