## Code Review Report

### 总体判断

**严重程度：高**  
该代码存在**明确的 SQL 注入漏洞**，直接使用字符串拼接构造 SQL 查询，完全不可接受。

---

### 主要问题

**1. SQL 注入漏洞（严重）**
- **Finding**: `f"SELECT * FROM notes WHERE customer_id = {customer_id}"` 直接将 `customer_id` 插入 SQL 字符串
- **风险**: 攻击者可通过 `customer_id` 参数注入任意 SQL，如：`1 OR 1=1` 获取所有客户数据，或更恶意操作
- **根因**: 未使用参数化查询

**2. 缺少输入验证**
- **Finding**: `customer_id` 未校验类型和范围（应为整数）
- **风险**: 非整数值导致数据库查询异常，可能暴露内部错误信息

**3. 返回结构不一致**
- **Finding**: 返回 `[dict(row) for row in rows]` 依赖数据库返回类型
- **风险**: 不同数据库驱动返回行格式可能不同，导致调用方处理失败

---

### 风险与影响

| 维度 | 风险等级 | 说明 |
|------|----------|------|
| 安全 | 🔴 严重 | SQL 注入可直接获取/篡改数据库内容 |
| 数据完整性 | 🔴 严重 | 攻击者可能执行 `DROP TABLE` 等破坏性操作 |
| 隐私 | 🟠 高 | 泄露任意客户备注数据，违反数据保护要求 |
| 可用性 | 🟡 中 | 缺少参数校验导致查询异常 |

### 测试建议

**安全测试（必须）**
```
1. SQL 注入测试用例
   - customer_id = "1 OR 1=1"     → 应返回空或报错，不泄露其他数据
   - customer_id = "1; DROP TABLE notes;--" → 应被拒绝
   - customer_id = "' OR '1'='1" → 应被拒绝

2. 参数类型测试

   - customer_id = "-1"          → 应返回空结果
   - customer_id = "0"           → 应返回空结果
   - customer_id = "999999999"   → 应返回空结果（不存在时）

3. 边界测试
   - customer_id = ""            → 应返回 400 错误
   - customer_id = None          → 应返回 400 错误
```

   - 验证调用方是否有权限访问该客户数据
