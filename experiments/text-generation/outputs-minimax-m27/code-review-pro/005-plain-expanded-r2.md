# Code Review: PR #005 - 用户导出功能

## 总体判定

**风险等级：高**

该 PR 存在严重的安全漏洞和合规问题，不建议合并。

---

## 主要发现

### 1. [严重] 后端 API 缺失权限验证

**问题**：后端 API `/export/users` 未实现角色检查，仅依赖前端按钮的可见性控制。

**影响**：任何已认证用户（即使非管理员）可直接调用 API 导出全部用户敏感数据。

**证据**：
```
变更摘要明确指出："只有前端按钮做了管理员判断，后端 API 未检查角色。"
```

**建议**：后端必须添加角色验证中间件，检查请求者是否具有 `ROLE_ADMIN` 权限。

---

### 2. [严重] 隐私数据暴露

**问题**：导出内容包含手机号、邮箱等 PII（个人身份信息），且无加密或访问控制。

**影响**：
- 违反 GDPR/《个人信息保护法》等法规
- 数据一旦泄露可直接识别用户身份
- 手机号可用于社会工程攻击

**建议**：
- 添加数据访问审批流程
- 导出的敏感字段应加密存储
- 明确告知用户数据用途并获取同意

---

### 3. [高] 缺失审计日志

**问题**：用户导出操作无任何审计记录。

**影响**：
- 无法追溯谁在何时导出了哪些数据
- 违反数据合规要求
- 出问题后难以进行事后分析

**建议**：记录以下审计信息：
```
timestamp, user_id, requestor_role, export_filters, record_count, destination_ip
```

---

### 4. [中] 无导出数量限制

**问题**：未对单次导出量设置上限。

**影响**：
- 可能被恶意用户耗尽系统资源
- 大量数据导出增加泄露风险面

**建议**：添加分页或数量限制（如单次最多 1000 条），并记录导出原因。

---

## 测试建议

| 测试场景 | 预期行为 |
|---------|---------|
| 非管理员用户调用导出 API | 返回 403 Forbidden |
| 管理员导出数据 | 成功，且生成审计日志 |
| 导出 1001 条数据 | 提示分批导出或拒绝 |
| 审计日志内容 | 包含操作者、时间、记录数 |

---

## 不确定性

以下信息未在变更摘要中提供，需要确认：

- 后端 API 路由和具体实现代码
- 是否有现成的权限中间件可复用
- 数据导出后存储位置（临时文件、云存储、S3？）
- 是否已有审计日志框架
