## Code Review: `get_customer_notes`

### 🔴 严重问题

**1. SQL 注入漏洞（安全-critical）**

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```
- `customer_id` 直接拼接进 SQL 语句
- 来源为 HTTP query 参数（用户输入）
- 攻击者可构造恶意输入执行任意 SQL

示例攻击：`customer_id = "1 OR 1=1 --"`

**2. 缺失授权检查（安全-high）**

- 无权限验证：任何请求者均可查询任意客户备注
- 可能导致跨客户数据泄露

### 🟡 中等问题

**3. 无输入校验**
- 未验证 `customer_id` 类型（应为整数）
- 未校验边界值或空值情况

**4. 无错误处理**
- 数据库连接失败、查询异常未捕获
- 可能导致服务直接崩溃

### ⚪ 代码质量问题

**5. 使用 `SELECT *`**
- 维护性风险：返回字段不确定，schema 变更可能破坏代码
- 性能浪费：获取不必要的字段

## 测试建议

| 测试类型 | 覆盖内容 |
|---------|---------|
| **安全测试** | SQL 注入尝试（如 `' OR 1=1 --`） |
| **边界测试** | 无效ID (0, 负数, 字符串, 空), 非存在客户 |
| **权限测试** | 未登录/无权限用户访问应拒绝 |
| **异常测试** | 数据库连接超时、查询失败场景 |
| **正确性测试** | 正常查询返回预期数据结构和内容 |

---

## 总结

**最高优先修复**：SQL 注入漏洞 → 使用参数化查询或 ORM 替代字符串拼接。
