## Code Review: load_orders

### 整体评价
**不认可该修法** ❌

---

### 主要发现

| 严重程度 | 类别 | 发现 |
|---------|------|------|
| 🔴 高 | 吞错静默失败 | `except Exception` 捕获所有异常并返回 `[]`，无法区分"用户无订单"与"系统异常" |
| 🔴 高 | 业务影响 | 财务对账场景下，遗漏订单数据可能导致对账结果错误、财务差异 |
| 🔴 高 | 可审计性 | 无日志记录，无法追溯失败原因，违反财务合规要求 |
| 🟡 中 | 错误类型 | 未针对特定异常类型（网络超时、认证失败、限流等）区分处理 |
| 🟡 中 | 可调试性 | 返回空数组丢失原始错误信息，增加排查难度 |

---

### 影响分析

**高风险**：若上游服务瞬时故障、限流或认证失效，该函数仍返回 `[]`，导致：
- 财务对账任务误判为"无订单"，遗漏实际交易
- 掩盖服务可用性问题，延迟故障发现
- 无法满足财务审计追溯要求

---

### 测试建议

1. **覆盖异常路径**：
   - 模拟网络超时 → 应抛出或返回错误结构，而非 `[]`
   - 模拟认证失败 → 应标记为认证错误
   - 模拟服务 500 错误 → 应触发告警或重试

2. **验证空数组来源**：
   - 区分 `[]`（真实无订单）与异常返回的 `[]`
   - 例如返回 `{"orders": [], "error": null}` 或抛出异常

3. **添加日志断言**：
   - 异常发生时验证日志输出包含 `user_id`、`异常类型`、`堆栈`

---

### 不确定性

- 未知 `client.fetch_orders` 可能抛出的具体异常类型
- 未知是否有重试机制或降级策略
- 未知现有日志框架及格式要求

### 建议修法（示例）

```python
import logging

logger = logging.getLogger(__name__)

def load_orders(client, user_id):
    try:
        return client.fetch_orders(user_id)
    except AuthError as e:
        logger.error(f"认证失败 user_id={user_id}", exc_info=True)
        raise  # 或返回错误结构
    except (TimeoutError, ConnectionError) as e:
        logger.warning(f"网络异常 user_id={user_id}", exc_info=True)
        raise  # 触发重试或告警
    except Exception as e:
        logger.error(f"未知异常 user_id={user_id}", exc_info=True)
        raise
```

**结论**：该修法以牺牲正确性和可追溯性换取测试通过，不符合财务系统可靠性要求，不予认可。
