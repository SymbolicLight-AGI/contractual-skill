**Code Review – `load_orders` (code‑review‑pro‑003)**  

| 优先级 | 发现 | 风险 | 建议 |
|--------|------|------|------|
| **Critical** | `except Exception` 吞掉所有异常并返回空列表 `[]`。 | 在财务对账场景中，上游接口失败（如网络超时、认证错误、内部错误）会被无声地当作“该用户无订单”。导致对账结果错误、漏报、监管合规风险。 | - 不要使用 `except Exception`；捕获具体可预期异常（如 `TimeoutError`, `ConnectionError`, `APIError`）。<br>- 对未知异常重新抛出或包装为自定义异常 `OrderFetchError`，确保调用方感知错误。 |
| **High** | 完全缺少日志或审计记录。 | 出现故障后无法追溯根因，难以满足审计要求。 | - 在捕获的每个异常路径中记录错误信息（包含 `user_id`、异常类型、堆栈）。<br>- 考虑使用结构化日志（JSON）并发送到监控系统。 |
| **High** | 返回值语义不明确。 | 调用方无法区分“真的没有订单”和“请求失败返回空列表”。导致业务逻辑误判。 | - 返回结构化结果，例如 `{"orders": [...], "error": None}` 或抛出异常。<br>- 若必须返回空列表，应在文档和调用方明确约定：仅在业务上确实不存在订单时才返回。 |
| **Medium** | 没有重试或降级策略。 | 单点网络抖动会直接导致对账中断。 | - 对瞬时可恢复的错误实现重试（如指数退避）。<br>- 考虑使用断路器模式，防止频繁调用已失效的上游服务。 |
| **Low** | 单元测试未覆盖错误路径。 | 当前测试只验证成功路径，无法发现吞错行为。 | - 为每种可捕获异常编写测试，验证：<br>  * 是否抛出/返回预期的错误对象；<br>  * 是否写入相应日志；<br>  * 是否执行重试或断路。<br>- 集成测试：在模拟的上游服务返回 5xx 时，验证对账任务产生警报或标记为“待人工检查”。 |

**结论**  
该修法 **不认可**。将所有异常统一捕获并返回空数组属于典型的“静默失败”。在财务对账等关键业务中，这种做法会导致错误的对账结果，潜在的法律合规风险极高。应采用细粒度的异常捕获、记录日志、结构化错误返回或向上层抛出明确的异常，并配合完善的单元/集成测试确保错误路径得到验证。
