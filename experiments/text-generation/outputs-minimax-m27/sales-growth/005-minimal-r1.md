| 项目 | 内容 |
|------|------|

| **组织架构** | 集团—分公司—服务商三级体系 |

## 销售可继续推进事项

| 事项 | 说明 |
|------|------|
| **业务现状调研** | 深入了解现有售后流程、工单量级、人员配置 |
| **关键干系人确认** | 明确各层级（集团/分公司/服务商）的业务负责人 |
| **试点范围初步建议** | 与客户协商选定1-2个场景作为首期试点 |
| **ROI评估标准沟通** | 与管理层对齐"试点ROI"的衡量指标和时间窗口 |
| **决策链梳理** | 确认最终采购决策人和项目拍板人 |

## 必须移交的事项

### 移交售前方案

| 移交事项 | 原因 |
|----------|------|
| **技术集成方案** | "所有系统都接上"涉及与CRM、备件系统的深度集成，需评估技术可行性和资源投入 |
| **系统架构设计** | 数据权限涉及集团、分公司、服务商三方，需明确数据流向和权限模型 |
| **实施路径规划** | 需制定分阶段集成策略，而非一次性全量对接 |

### 移交风险审查

| 移交事项 | 原因 |
|----------|------|
| **范围蔓延风险** | "先把所有系统都接上"属于固定低价全包诉求，需评估项目失控可能性 |
| **二开定价风险** | "报价包含后续所有二开"属于未定义范围的无限承诺，需审查合同条款 |
| **交付承诺审查** | 需评估现有资源能否支撑多系统集成+快速上线的组合要求 |

### 移交交付经理

| 移交事项 | 原因 |
|----------|------|
| **交付计划制定** | 多方系统集成需明确实施里程碑和资源调配 |
| **试点ROI跟踪机制** | 需建立可量化的试点效果评估体系 |
| **服务商角色定义** | 需明确服务商在系统使用和数据权限中的定位 |

