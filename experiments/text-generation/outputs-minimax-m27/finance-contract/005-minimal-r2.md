# 项目走合同可行性判断

## 结论

**不能直接走合同。** 当前材料存在多项未明确的关键事项，必须完成复核后方可进入合同流程。

---

## 必须交由各角色复核的事项清单

### 1. 售前 / 销售负责人

| 复核事项 | 具体问题 |
|----------|----------|
| 范围细化 | ① 集团总部知识库需明确：内容结构、用户数、存储容量；② 5个业务Agent的具体业务场景与功能边界；③ 各系统接口的对接范围（单向/双向、数据字段、频次） |
| 报价完整性 | ① 试点阶段的接口数量与复杂度是否已全含；② 推广至8家分公司是否在报价内，若不在，分公司单价或打包价如何设定 |
| 验收标准来源 | "后续再定"由谁定、何时定、双方何种机制确认——需客户书面确认流程 |

### 2. 交付 / 项目负责人

| 复核事项 | 具体问题 |
|----------|----------|
| 技术可行性 | 各系统接口对接的工作量与依赖关系，是否存在无法集成的风险 |
| 分阶段交付 | 试点与推广是否可分阶段定义里程碑，还是必须一次交付 |
| 验收判定 | 无确定验收标准时，交付团队如何自证完成 |

### 3. 财务 / 商务负责人

| 复核事项 | 具体问题 |
|----------|----------|
| 全阶段报价逻辑 | 试点+推广全含的定价依据，若后期范围扩大是否设预留调价机制 |
| 付款节点 | 各阶段的付款触发条件，与验收标准挂钩的方式 |
| 成本敞口 | 接口范围"全含"可能导致成本超出报价 |

### 4. 法务 / 合同审查人

| 复核事项 | 具体问题 |
|----------|----------|
| 合同结构 | 一份合同覆盖全阶段，需明确：试点验收通过是否为推广启动的前置条件 |
| 变更机制 | 范围模糊条款下，是否设立变更/调价触发阈值（如接口超过N个需重新议价） |
| 违约风险 | 验收标准缺失时，甲乙双方的履约边界如何界定 |

### 5. 风险 / 项目管理办公室（PMO）

| 复核事项 | 具体问题 |
|----------|----------|
| 范围蔓延风险 | 高（"全含"承诺缺乏边界定义） |
| 验收争议风险 | 高（无确定标准，客户随时可主张未达验收） |
| 建议处置 | ① 要求客户在合同签署前书面锁定验收标准；② 或改用分阶段合同（试点合同+推广意向函），降低全阶段绑定风险 |

---

## 建议下一步

1. **向售前返回**补充范围定义（至少完成上表左侧栏的①②③细化），再进入合同流程。
2. **财务+法务联合**制定"范围不确定"条款下的调价与变更机制。
3. **风险确认**后再由销售推进合同审批。
