### 🔴 严重 (Critical) - 安全风险：敏感凭证硬编码
- **发现**：配置文件中直接写入了生产环境支付 API 密钥（`SYNTHETIC_PAYMENT_SECRET_PLACEHOLDER`）。
- **风险**：结合背景中“项目未启用 Secret 扫描”的情况，该提交一旦合并，密钥将永久留存于 Git 历史中。攻击者可通过仓库克隆或历史遍历直接获取凭证，导致未授权支付调用、资金损失或数据泄露。
- **建议**：**立即阻断本次合并**。必须在支付平台侧**轮换/吊销该密钥**。后续所有敏感凭证必须通过环境变量、CI/CD 密钥库或专用密钥管理服务注入，严禁明文提交至代码仓库。

### 🟡 中等 (Medium) - 可维护性风险：配置与环境耦合
- **发现**：`api_url` 与 `api_key` 等环境强相关参数直接写死在静态配置中。
- **风险**：多环境（Dev/Test/Prod）部署时需手动修改文件，易引发配置漂移、误覆盖或部署失败。
- **建议**：采用环境分离策略，将动态/敏感配置抽离至 `.env` 文件或基础设施配置中心，代码库仅保留默认值或占位符。

### 🟢 低 (Low) - 测试与流程缺失
- **发现**：缺乏自动化凭证检测与配置校验机制。
- **风险**：依赖人工 Review 难以持续拦截类似泄露，且无法验证配置加载的正确性。
- **建议**：
  1. 在 CI/CD 流水线中集成 Secret 扫描工具（如 gitleaks、truffleHog 或平台原生安全扫描）。
  2. 补充配置加载单元测试，验证必填字段存在性、格式合法性，并确保测试环境使用 Mock 密钥而非真实凭证。
