## 跟进人判定与交接材料

---

### 一、跟进人判定

| 事项 | 负责角色 | 状态 |
|:---|:---|:---|
| 商务关系维护、合同框架谈判、ROI 试点模式共识 | **销售** | 继续推进 |
| 技术架构方案、系统集成方案、数据权限设计 | **售前方案** | 必须交接 |
| 报价范围界定、二开边界与知识产权条款 | **风险审查/法务** | 必须交接 |
| 交付可行性评估、实施周期与资源排期 | **交付经理** | 必须交接 |

---

### 二、销售可继续推进事项

| 事项 | 具体动作 |
|:---|:---|
| 试点 ROI 共识 | 引导客户接受"有限范围试点→验证 ROI→逐步扩展"模式，对抗"全接上看效果" |
| 决策链梳理 | 明确管理层、IT、分公司、服务商各方的决策权和影响力 |
| 预算周期确认 | 了解客户财年、预算窗口及采购流程 |
| 竞品信息收集 | 客户是否接触其他厂商，评估标准为何 |
| 合同框架预沟通 | 明确"订阅/授权+实施"还是"项目制"，为报价结构铺垫 |

---

### 三、必须交接事项及材料

#### 1. 售前方案（技术架构与集成）

| 交接材料 | 关键问题 |
|:---|:---|
| 现有 CRM 与备件系统清单 | 品牌、版本、接口文档、数据量级 |
| 三方数据权限架构图需求 | 集团-分公司-服务商的权限模型（组织隔离、数据脱敏、审批流） |
| 售后知识库知识来源与结构化方案 | 历史工单/手册/专家经验？是否需要 NLP 训练？ |
| 工单自动分类准确率基线 | 现有分类体系、标注数据量、预期准确率 |
| 集成技术路线 | 中间件/API/定制开发？是否需驻场？ |

#### 2. 风险审查（报价与法律）

| 交接材料 | 关键问题 |
|:---|:---|
| "后续所有二开"范围界定 | 需明确：功能增强/接口适配/报表定制/系统升级是否均包含？ |
| 知识产权归属 | 二开代码、定制模型、知识库内容归属 |
| 验收标准与退出机制 | "效果"如何量化？未达效果时的权责 |
| 数据安全合规 | 装备行业是否涉及工控数据、跨境传输、等保要求 |

#### 3. 交付经理（实施可行性）

| 交接材料 | 关键问题 |
|:---|:---|
| 客户方 IT 配合度与能力 | 是否有专人对接？接口开发由谁承担？ |
| 多方协调机制 | 集团 IT、分公司、服务商的协作流程与历史矛盾 |
| 试点范围建议 | 最小可行试点（如：单事业部、单产品线） |
| 实施周期与里程碑 | 全量集成 vs 试点上线的资源差异 |

---

### 四、风险信号应对建议（供销售参考）

| 风险信号 | 销售应对策略 |
|:---|:---|
| "先把所有系统都接上，再看效果" | 以行业案例说明：全量集成周期过长、风险不可控，建议"试点验证价值→复制推广" |
| 报价含"所有二开" | 不承诺无限范围，引导客户接受"试点包+扩展包"的阶梯报价 |
| 三方数据权限复杂 | 强调需技术团队评估后给出方案，避免销售擅自承诺 |
