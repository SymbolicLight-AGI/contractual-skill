## 总体判断

**高风险 PR，不建议直接合并。** 存在权限绕过、敏感数据泄露、缺乏审计追踪等核心安全问题。

---

## 主要问题

| 优先级 | 类别 | 发现 |
|:---|:---|:---|
| **P0** | 权限绕过 | 后端 API 未校验管理员角色，非管理员可直接调用导出接口 |
| **P0** | 敏感数据泄露 | 导出字段包含手机号、邮箱等 PII，无脱敏或最小化控制 |
| **P1** | 审计缺失 | 无操作日志记录谁、何时、导出了哪些数据 |
| **P1** | 性能/滥用 | 无导出数量限制，存在全量拖库风险 |
| **P2** | 前端越权假象 | 前端按钮隐藏但接口暴露，造成"安全已做"的误导 |

---

## 风险与影响

| 风险 | 具体影响 |
|:---|:---|
| **大规模 PII 泄露** | 攻击者遍历用户 ID 即可批量获取姓名、手机号、邮箱、消费金额 |
| **合规违规** | 违反《个人信息保护法》最小必要原则，可能触发监管处罚 |
| **内部威胁** | 无审计日志导致数据泄露后无法追溯责任人 |
| **竞争对手情报获取** | 订单金额数据可用于商业分析 |

---

## 测试建议

1. **越权测试**：用普通用户 Token 直接调用导出 API，验证是否被拒绝
2. **字段级校验：确认非管理员请求时返回 403，且错误信息不暴露数据存在性**
3. **审计验证：检查数据库/日志系统是否记录 `operator_id`, `timestamp`, `export_scope`, `row_count`**
4. **压力测试：验证 10万+ 用户导出时的内存占用和响应时间**
5. **数据范围测试：验证只能导出授权范围内的用户（如本租户/本组织）**

## 交接建议

| 负责人 | 必须确认的事项 | 原因 |
|:---|:---|:---|
| **🔒 Security** | 1. 后端 API 强制校验管理员角色<br>2. 导出字段最小化评审（手机号/邮箱是否必须明文）<br>3. 敏感操作审计标准 | 权限绕过 + PII 泄露属于安全红线 |
| **🏗️ Backend** | 1. 接入现有权限中间件或新增 `@require_admin` 装饰器<br>2. 增加 `limit`/`offset` 或流式导出控制<br>3. 数据范围校验（防水平越权） | 技术实现主体，需保证数据隔离 |
| **📋 Product** | 1. 确认导出字段清单（尤其手机号/邮箱的明文/脱敏策略）<br>2. 确认导出场景：管理员单次/批量/全量？<br>3. 用户告知与授权（隐私政策更新） | 业务合规边界，影响功能设计 |
