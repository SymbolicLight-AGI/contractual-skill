## Review: get_customer_notes

### 主要发现

| 优先级 | 类别 | 问题 |
|--------|------|------|
| **P0 - 严重** | **SQL 注入** | 字符串拼接 `customer_id` 构成 SQL 注入漏洞。攻击者可构造 `customer_id=1 OR 1=1` 或更恶意 payload，导致未授权数据访问、数据泄露或数据破坏。 |
| **P1 - 高** | **权限绕过 / 越权访问** | 无租户隔离或用户权限校验，任何持有 `customer_id` 的用户可查询任意客户备注，存在水平越权风险。 |
| **P2 - 中** | **数据暴露** | `SELECT *` 可能返回敏感字段（如内部审核标记、PII），超出最小必要原则。 |
| **P2 - 中** | **错误处理缺失** | 无异常处理：数据库连接失败、查询错误时直接抛异常，可能泄露堆栈信息或导致服务中断。 |
| **P3 - 低** | **可维护性** | 返回裸 `dict` 无 schema 约束，下游消费方难以依赖；变更表结构时易隐性破坏。 |

---

### 影响评估

- **安全**：SQL 注入可直接导致全表数据泄露或篡改；越权访问使内部 CRM 数据对任意已知 ID 的用户可见。
- **业务**：客户隐私数据（备注常含敏感信息）暴露可能触发合规风险（如 GDPR、个人信息保护法）。
- **运维**：异常未处理可能引发 500 错误风暴，缺乏可观测性。

---

### 测试建议

| 测试类型 | 具体建议 |
|----------|----------|
| **安全测试** | 使用参数化查询后，用 fuzzing 测试注入 payload（如 `' OR '1'='1`、`; DROP TABLE` 变体）验证无注入。 |
| **权限测试** | 模拟未授权用户请求他人 `customer_id`，断言返回 403 或空结果。 |
| **边界测试** | 测试 `customer_id` 为 `None`、空字符串、超长字符串、非预期类型（如 list）时的行为。 |
| **数据测试** | 验证返回字段是否严格限定，无多余敏感列。 |
| **异常测试** | Mock `db.execute` 抛出 `OperationalError`、`InterfaceError`，验证错误处理与日志记录，不泄露敏感信息。 |

---

### 不确定性

- 不确定 `db` 的具体类型（SQLAlchemy Core/ORM、raw DB-API、或其他），影响参数化查询的具体写法建议。
- 不确定 `customer_id` 是否经过上游校验/清洗，但即使有过滤也不应依赖其作为安全边界。
