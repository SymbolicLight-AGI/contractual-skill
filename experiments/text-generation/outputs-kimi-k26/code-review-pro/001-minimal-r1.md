## Code Review 结果

### 🔴 严重：SQL 注入漏洞

**风险**：`customer_id` 直接来自 HTTP query 参数，通过 f-string 拼接进 SQL 语句，存在 SQL 注入攻击风险。

```python
# 攻击示例（customer_id 参数）：
# 123 OR 1=1; DROP TABLE notes; --
```

**影响**：数据泄露、数据篡改或删除、未授权访问。

---

### 🟠 高：缺乏输入验证

**风险**：`customer_id` 未验证类型和格式。可能传入非预期值（字符串、负数、UUID 格式错误等），导致查询异常或逻辑错误。

---

### 🟡 中：错误处理缺失

**风险**：未处理以下异常场景：
- 数据库连接失败
- `customer_id` 不存在（返回空列表 vs 报错边界模糊）
- SQL 语法错误（拼接失败时）

---

### 🟡 中：返回字段不可控

**风险**：`SELECT *` 可能随表结构变更返回意外字段，存在 Schema 变更风险。调用方可能依赖不稳定字段。

---

### 🟢 低：无资源释放确认

**风险**：`fetchall()` 在数据量大时可能内存溢出。依赖外部 `db` 对象的生命周期管理，存在连接泄漏隐患。

---

## 测试建议

| 测试类型 | 具体建议 |
|---------|---------|
| **安全测试** | 使用参数化查询后，验证 SQL 注入 payload（`' OR '1'='1`、`; DROP TABLE --` 等）被安全处理 |
| **输入验证测试** | 覆盖边界值：负数、0、超大整数、字符串、None、空值、特殊字符 |
| **异常测试** | 模拟数据库异常（连接断开、超时）、模拟不存在的 `customer_id` |
| **Schema 测试** | 断言返回字段集合，确保 `SELECT *` 变更时测试能捕获 |
| **性能测试** | 大数据量场景下验证内存使用，或确认使用流式/分页查询 |
