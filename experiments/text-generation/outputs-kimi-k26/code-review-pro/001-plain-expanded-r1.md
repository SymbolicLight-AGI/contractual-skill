## Review: get_customer_notes

### 发现项

| # | 问题 | 严重性 | 类别 |
|---|------|--------|------|
| 1 | **SQL 注入漏洞**：`customer_id` 直接字符串拼接进 SQL 语句，未参数化 | **Critical** | 安全 |
| 2 | **无输入验证**：`customer_id` 类型和格式未校验，可能传入非预期值 | High | 正确性 |
| 3 | **SELECT * 反模式**：返回所有列，存在 schema 变更后隐式行为变化风险 | Medium | 可维护性 |
| 4 | **无结果集大小限制**：可能返回大量数据导致 OOM 或慢查询 | Medium | 性能/运维 |
| 5 | **异常处理缺失**：数据库操作异常未捕获，可能暴露内部信息或导致 500 | Medium | 运维 |

---

### 风险影响

- **SQL 注入**：攻击者可构造 `customer_id=1 OR 1=1` 或 UNION 查询，导致全表数据泄露、数据篡改或删除
- **信息泄露**：数据库异常原样抛出可能暴露 schema、表结构等敏感信息
- **可用性风险**：无 LIMIT 的大查询在客户备注多时可能拖垮数据库

---

### 测试建议

| 优先级 | 测试项 |
|--------|--------|
| P0 | 参数化查询验证：确认 `?` 占位符或 ORM 参数绑定生效 |
| P0 | 注入攻击测试：传入 `' OR '1'='1`、`1; DROP TABLE` 等 payload |
| P1 | 非法输入测试：`None`、空字符串、超长字符串、非数字字符串 |
| P1 | 大数据量测试：模拟万级/百万级返回，验证 LIMIT 和分页 |
| P2 | Schema 兼容性测试：模拟 `notes` 表增删列，验证行为稳定性 |

---

### 不确定性

- 使用的数据库驱动类型未知（sqlite3/psycopg2/pymysql 等），参数化语法细节不确定
- `db` 对象是否为连接池、是否有内置查询超时机制未知
- 该函数是否有调用方做前置校验未知
