## Code Review 结果

### 严重风险 (Critical)

| 项目 | 问题 | 影响 |
|:---|:---|:---|
| **垂直权限绕过** | 后端 API 未校验管理员角色，仅依赖前端按钮隐藏 | 普通用户可直接调用 API 导出全量用户数据 |
| **敏感数据泄露** | 导出字段包含手机号、邮箱等 PII，无脱敏或最小化原则 | 违反 GDPR/《个人信息保护法》，面临合规处罚 |
| **无访问控制** | 缺乏基于角色的访问控制 (RBAC) | 任何获取 API 端点的用户均可批量窃取数据 |

### 高风险 (High)

| 项目 | 问题 | 影响 |
|:---|:---|:---|
| **无审计日志** | 无法追踪谁、何时、导出了多少数据 | 数据泄露后无法溯源，无法满足合规审计要求 |
| **无导出限制** | 未限制单次导出数量/频率 | 可导致全库拖库，服务器资源耗尽 |
| **无速率限制** | API 缺乏限流 | 易被自动化工具批量爬取 |

### 中风险 (Medium)

| 项目 | 问题 |
|:---|:---|
| **数据范围未限定** | 可导出"所有用户"而非按需范围，违反最小必要原则 |
| **传输安全存疑** | 未确认导出文件是否加密传输/存储 |
| **缺少测试覆盖** | 无权限绕过测试、无数据边界测试、无审计验证测试 |

## 需各负责人确认事项

| 负责人 | 必须确认事项 |
|:---|:---|
| **安全负责人** | 1. 是否需立即下线该 API 或加紧急 WAF 规则<br>2. 数据泄露事件响应预案是否需启动（已存在暴露风险）<br>3. 是否需要引入数据分类分级（DLP）管控 |
| **后端负责人** | 1. RBAC 中间件补全方案及排期<br>2. 审计日志表设计（操作人、时间、IP、数据范围、记录数）<br>3. 导出数量上限/分页/异步队列实现<br>4. 接口速率限制（如 10次/分钟） |
| **产品负责人** | 1. 导出字段是否需缩减（如去掉手机号，仅保留脱敏后信息）<br>2. 导出场景是否需用户授权同意（合规要求）<br>3. 是否需"导出申请-审批"流程而非直接导出<br>4. 数据保留期限及删除策略 |

---

## 测试建议

- **安全测试**：未授权访问、水平/垂直越权、SQL 注入（若存在动态拼接）
- **性能测试**：大数据量导出时的内存占用、超时、数据库连接池耗尽
- **合规测试**：审计日志完整性校验、敏感操作告警触发

---

**结论**：该 PR **不可直接合并**。存在可导致大规模数据泄露的严重安全缺陷，需安全与后端优先修复权限与审计问题，产品确认数据范围后，方可进入测试环节。
