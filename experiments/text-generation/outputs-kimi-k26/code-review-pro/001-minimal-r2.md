## Code Review Findings

### 🔴 Critical: SQL Injection

**Location:** `get_customer_notes` 函数第 2 行

**问题：** 使用 f-string 拼接 SQL 查询，存在 SQL 注入漏洞。

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

**风险：** `customer_id` 来自 HTTP query 参数，攻击者可构造恶意输入（如 `1 OR 1=1` 、`1; DROP TABLE notes;--` 等）读取或破坏数据。

**修复方向：** 使用参数化查询，如 `db.execute("SELECT * FROM notes WHERE customer_id = ?", (customer_id,))`。（具体语法取决于数据库驱动）

---

### 🟡 Medium: 缺少输入校验

**问题：** 未验证 `customer_id` 的类型和格式。

**风险：**
- 非预期输入（如空字符串、负数、UUID 格式字符串）可能导致查询异常或逻辑错误
- 数据库层错误暴露到应用层

**修复方向：** 校验 `customer_id` 为预期类型（如正整数），提前拒绝非法输入。

---

### 🟡 Medium: 异常处理缺失

**问题：** 未处理 `db.execute()` 可能抛出的异常。

**风险：**
- 数据库连接失败、查询超时等异常直接向上传播
- 可能暴露内部错误信息（如数据库结构、连接字符串）

**修复方向：** 添加适当的异常捕获和日志记录，对外返回统一错误响应。

---

### 🟢 Low: `SELECT *` 的维护性问题

**问题：** 使用 `SELECT *` 查询所有列。

**风险：**
- 表结构变更时可能返回意外数据
- 不必要的 I/O 和内存开销（若 notes 表包含大字段）

**修复方向：** 明确指定所需列名。

---

### 🟢 Low: 返回数据未脱敏

**问题：** 直接返回原始数据库行数据。

**风险：** 若 `notes` 表包含敏感字段（如内部备注、PII），可能泄露给未授权调用方。

**修复方向：** 明确选择返回字段，或增加数据访问控制层。

---

## 测试建议

| 测试类型 | 具体内容 |
|---------|---------|
| **安全测试** | 验证 SQL 注入攻击被防御（输入 `1' OR '1'='1` 等） |
| **输入校验测试** | 边界值：负数、0、超大整数、非数字字符串、空值 |
| **异常场景测试** | 模拟数据库连接失败、查询超时、表不存在等情况 |
| **功能测试** | 正常查询、无结果返回空列表、多行结果正确序列化 |
| **性能测试** | 大数据量下的响应时间和内存占用 |
| **权限测试** | 验证只能访问当前租户/用户的客户备注（若涉及多租户） |
