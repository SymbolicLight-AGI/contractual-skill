## Code Review: `get_customer_notes`

### 安全问题

| 问题 | 严重程度 | 说明 |
|:---|:---|:---|
| **SQL 注入** | 🔴 严重 | 字符串拼接 `customer_id` 导致注入漏洞。`customer_id` 来自 HTTP 参数，攻击者可构造恶意输入（如 `1 OR 1=1` 或 `1; DROP TABLE notes--`） |
| 权限绕过风险 | 🟡 中 | `SELECT *` 可能返回敏感字段；若 `notes` 表后续增加敏感列，存在信息泄露风险 |

### 正确性问题

| 问题 | 说明 |
|:---|:---|
| **缺少参数化查询** | 未使用占位符（`?` / `%s` / `:customer_id`），既引发安全问题，也可能导致类型处理错误 |
| **无输入校验** | 未验证 `customer_id` 类型/格式（如应为整数或 UUID），非法输入可能引发非预期行为 |
| **无结果为空处理** | `fetchall()` 返回空列表是正常行为，但需确认调用方是否区分"无备注"与"客户不存在" |
| **数据库连接异常未处理** | `db.execute()` 失败时异常直接抛出，可能暴露内部信息或导致服务中断 |

### 测试风险

| 风险 | 说明 |
|:---|:---|
| **无现有测试** | 变更缺乏回归保护，重构或修复时易引入回归缺陷 |
| **测试难以覆盖注入场景** | 若测试仅 mock `db.execute`，无法验证实际 SQL 安全性；需集成测试或 SQL 审计 |
| **边界值未验证** | 需测试：`customer_id` 为 `None`、空字符串、超大整数、特殊字符、SQL 关键字等 |
| **ORM/驱动差异** | `dict(row)` 依赖行对象的 `__iter__` 或 `keys()` 实现，不同数据库驱动（sqlite3/psycopg2/sqlalchemy）行为可能不同 |

### 测试建议

1. **安全测试**：使用参数化查询后，用 `pytest` + `unittest.mock` 验证 `execute` 调用参数，确保用户输入被作为参数绑定而非拼接
2. **集成测试**：针对真实数据库驱动验证 `dict(row)` 兼容性
3. **输入校验测试**：覆盖合法 ID、非法字符串、空值、极值等场景
4. **异常测试**：模拟数据库连接失败、`execute` 抛出异常，验证错误处理逻辑
