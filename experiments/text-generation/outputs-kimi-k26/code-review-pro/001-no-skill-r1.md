## Code Review 结果

### 安全问题

| 项目 | 说明 |
|:---|:---|
| **SQL 注入** | 使用 f-string 拼接 SQL，`customer_id` 来自 HTTP 参数，攻击者可构造 `1 OR 1=1` 等恶意输入 |
| 风险等级 | **严重** — 可导致数据泄露、未授权访问 |

### 正确性问题

| 项目 | 说明 |
|:---|:---|
| **缺少参数化查询** | 应使用占位符（如 `?` / `%s`）替代字符串拼接 |
| **未处理 `customer_id` 类型** | 未验证/转换输入为整数，可能传入非法值 |
| **未处理空结果/异常** | `db.execute()` 可能抛异常，无 `try/except`；数据库连接失败场景未覆盖 |
| **`SELECT *` 隐患** | 返回全部列，若表结构变更可能影响下游逻辑；建议显式指定所需列 |

### 测试风险

| 风险点 | 说明 |
|:---|:---|
| **无测试覆盖** | 当前零测试，变更无回归保障 |
| **数据库依赖** | 直接依赖真实 `db` 对象，难以单元测试；需引入 mock 或依赖注入 |
| **输入边界未验证** | 缺少对 `None`、空字符串、负数、超大数值、非字符串类型的测试 |
| **SQL 执行结果未断言** | 返回的 `dict` 结构、字段完整性未验证 |

### 测试建议

1. **安全测试**：验证 SQL 注入 payload（如 `' OR '1'='1`）被正确转义或拒绝
2. **单元测试**：mock `db.execute` 返回固定结果，断言查询语句和参数
3. **集成测试**：使用临时/内存数据库验证端到端查询
4. **输入验证测试**：覆盖非法 `customer_id` 的异常处理路径
