**代码审查发现与风险评估（按严重程度排序）**

**1. [严重] 安全风险：SQL 注入 (SQL Injection)**
*   **问题描述**：代码直接使用 f-string 将来自 HTTP query 参数的 `customer_id` 拼接到 SQL 查询语句中。由于 HTTP 参数通常是不可信的用户输入，攻击者可以传入恶意字符串（例如 `1 OR 1=1` 或 `1; DROP TABLE notes`），从而绕过查询限制、窃取数据或破坏数据库。
*   **修复建议**：必须废弃字符串拼接的做法。应使用底层数据库驱动提供的“参数化查询（Parameterized Queries）”或预编译语句来传递 `customer_id`。

**2. [中等] 维护性与正确性风险：使用 `SELECT *`**
*   **问题描述**：使用 `SELECT *` 会返回表中的所有列。如果未来 `notes` 表的结构发生变化（例如添加了敏感字段或大文本字段），该函数会隐式地将这些多余数据暴露给上层调用者，增加网络和内存开销，且容易引发不可预期的错误。
*   **修复建议**：明确列出需要查询的字段名称（例如 `SELECT id, content, created_at FROM notes...`）。

**3. [低] 性能风险：无限制的 `fetchall()`**
*   **问题描述**：由于缺乏数据量上下文，如果某个客户拥有海量备注，直接使用 `fetchall()` 会将所有数据一次性加载到内存中，可能导致内存溢出或响应缓慢。
*   **修复建议**：评估业务需求，考虑是否需要在 SQL 查询中引入分页机制（`LIMIT` 和 `OFFSET`）。
