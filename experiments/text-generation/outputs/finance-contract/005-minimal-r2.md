## 主要风险判断

1. **范围过大且边界不清**
   - 集团级知识库、5 个业务 Agent、3 类系统对接、试点后推广 8 家分公司，属于多阶段、多系统、多组织范围。
   - “一份合同覆盖全部阶段”会导致后续推广范围、接口数量、实施工作量难以控制。

2. **接口报价风险高**
   - 客户要求“报价包含所有系统接口”，但目前只列出 OA、CRM、售后工单系统，未明确：
     - 接口数量
     - 接口方式
     - 数据字段
     - 第三方配合责任
     - 是否包含二次开发、联调、变更和异常处理
   - 容易形成无限接口责任。

3. **验收标准缺失**
   - “验收标准后续再定”不可直接进入合同。
   - 缺少验收标准会影响交付边界、付款节点、争议处理和收入确认。

4. **推广阶段风险不明确**
   - 试点后推广到 8 家分公司，但未明确推广条件、各分公司差异、部署方式、培训范围、上线支持、验收方式。
   - 不宜一次性承诺全部推广交付。

5. **付款与里程碑信息缺失**
   - 未提供合同金额、报价明细、付款节点、验收付款比例、质保或运维范围。
   - 无法判断现金流、回款风险和收入确认风险。

## 必须复核角色及事项

### 1. 售前必须复核

需确认并输出正式方案边界：

- 集团总部知识库的建设范围、数据来源、数据量、权限体系。
- 5 个业务 Agent 的具体业务场景、功能边界、输出形式和成功标准。
- OA、CRM、售后工单系统的接口清单、接口方式、接口数量、数据字段和依赖条件。
- 试点范围与推广范围是否拆分。
- 8 家分公司的差异化需求是否包含在当前报价内。
- 是否存在客户期望的“全部系统、全部接口、全部阶段打包交付”风险。

### 2. 交付必须复核

需评估可交付性和资源风险：

- 项目阶段划分：调研、设计、开发、集成、测试、试点、推广、验收。
- 每阶段交付物、交付周期、人员投入和客户配合要求。
- 多系统联调责任边界，尤其是客户 OA、CRM、售后工单系统厂商配合。
- 试点到 8 家分公司推广的实施工作量、上线支持和培训范围。
- 是否需要设置变更管理机制，避免后续范围无限扩大。
- 验收标准必须在合同前确定，至少需明确试点验收和推广验收规则。

### 3. 财务必须复核

需评估报价、付款和收入风险：

- 当前报价是否覆盖：
  - 5 个业务 Agent
  - 3 类系统接口
  - 8 家分公司推广
  - 项目管理、培训、测试、上线支持
  - 后续运维或质保
- 是否存在漏报、低估、一次性打包导致毛利不足风险。
- 付款节点是否与交付里程碑匹配。
- 是否存在验收后置、付款延迟、尾款比例过高风险。
- 在验收标准未定的情况下，收入确认和回款风险较高，需财务确认。

### 4. 法务必须复核

需审查合同结构和责任边界：

- 是否适合“一份合同覆盖全部阶段”，或应拆分为主合同+阶段订单/补充协议。
- 合同中必须明确范围边界、接口边界、验收标准、客户配合义务、变更机制。
- 避免出现“包含所有接口”“满足全部业务需求”等无限责任表述。
- 明确第三方系统、客户数据、权限、安全合规责任。
- 明确试点失败、推广条件不满足、客户未配合时的责任处理。
- 验收标准不能留到后续再定，否则合同争议风险高。

### 5. 风险审查必须介入

该项目建议进入风险审查，原因包括：

- 集团级项目，覆盖总部和 8 家分公司。
- 多 Agent、多系统集成，交付复杂度高。
- 客户要求一份合同覆盖全部阶段。
- 报价要求包含所有系统接口，存在无限范围风险。
- 验收标准未定，存在重大回款和争议风险。
- 涉及知识库、业务数据、OA/CRM/售后工单系统，可能涉及数据安全和权限风险。

## 建议下一步

1. 暂不直接走合同。
2. 先由售前和交付联合完成《范围说明书》或《实施边界说明》。
3. 明确接口清单、Agent 清单、试点范围、推广范围和客户配合责任。
4. 财务重新核算报价和付款里程碑。
5. 法务评估合同结构，建议采用阶段化合同或主合同加阶段订单机制。
6. 在验收标准、付款节点、变更机制明确前，不建议提交正式合同审批。
