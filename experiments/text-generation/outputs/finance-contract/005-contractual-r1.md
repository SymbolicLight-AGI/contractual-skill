## 结论摘要

**结论：不建议直接走合同。**  
该项目存在跨阶段覆盖、系统接口范围、验收标准未定义、推广范围较大等关键不确定项。当前材料不足以形成安全合同承诺，应先完成售前、交付、财务、法务和风险审查复核后，再决定是否进入合同流程。

## 财务与合同风险

- **Risk：一份合同覆盖全部阶段，范围过大**
  - 试点和 8 家分公司推广属于不同交付阶段，若全部纳入一份合同但缺少阶段边界，可能导致交付义务扩大、成本失控和回款节点不清。

- **Risk：报价包含所有系统接口但接口范围未定义**
  - “所有系统接口”容易被解释为无限接口或包含未来新增接口，可能产生额外开发、联调、第三方配合和运维成本。

- **Risk：验收标准后续再定**
  - 验收标准未定会导致收入确认、回款触发、交付完成认定和争议处理风险。

- **Risk：推广到 8 家分公司存在复制成本与差异化需求**
  - 分公司可能存在流程、权限、数据、系统版本和业务场景差异，不能默认与总部试点完全一致。

- **Risk：多 Agent 工作流可能涉及效果边界**
  - 5 个业务 Agent 的能力边界、准确率、人工复核责任、异常处理机制未定义，可能引发客户对效果的超预期要求。

- **Risk：现金流与应收风险未评估**
  - 未见付款节点、开票条件、回款条件和客户信用信息，不能支持“先覆盖全部阶段再付款”或不清晰付款安排。

- **Risk：法律责任与系统集成责任不清**
  - OA、CRM、售后工单系统可能涉及第三方系统、数据安全、权限、接口稳定性和客户配合义务，需明确责任边界。

## 不得直接承诺事项

- 不得直接承诺“一份合同无条件覆盖试点和 8 家分公司全部推广”。
- 不得承诺“报价包含所有系统接口”，除非接口清单、数量、边界和变更机制已明确并获审批。
- 不得接受“验收标准后续再定”作为直接签约条件。
- 不得承诺无限接口、无限定制、无限培训、无限优化或无限推广支持。
- 不得承诺未审批的免费接口、免费分公司推广、免费 Agent 扩展或免费系统改造。
- 不得承诺未复核的 SLA、响应时效、效果指标、准确率指标或法律责任条款。
- 不得在缺少成本、信用、付款和审批检查的情况下支持“先做后定验收 / 后付款”。

## 待确认问题

### 售前需确认

- 5 个业务 Agent 的具体业务场景、功能边界、输入输出、工作流节点是什么？
- 集团总部知识库包含哪些数据来源、数据量、格式、权限模型和更新机制？

- 是否存在客户口头要求但未写入材料的额外功能、接口、培训或运维承诺？

### 交付需确认

- 试点阶段交付物、周期、资源投入和客户配合事项是什么？
- 分公司推广的实施方式、批次、每家分公司的上线条件和验收方式是什么？
- 数据清洗、知识库构建、权限配置、系统联调、测试环境和生产环境由谁负责？
- Agent 效果如何评估？是否需要人工复核、灰度发布和回滚机制？
- 是否需要单独评估接口开发、联调、部署、培训和运维工时？

### 财务需确认

- 报价金额、折扣、毛利、成本测算是否已完成？
- 接口开发、分公司推广、培训、运维、项目管理是否已纳入成本？
- 付款里程碑是否与阶段交付和验收绑定？
- 是否存在延期付款、低首付、尾款比例过高或验收后集中付款风险？
- 是否需要信用评估或应收账款风险审查？

### 法务需确认

- 一份合同覆盖全部阶段是否可行，是否应拆分为主合同 + 阶段订单 / SOW？
- 接口范围、第三方系统责任、客户配合义务、数据安全责任如何表述？
- 验收标准、验收逾期视同通过、争议处理和终止条款是否需要明确？
- 是否涉及 SLA、知识产权、数据合规、保密和责任限制条款？
- “验收标准后续再定”是否存在重大合同不确定性。

### 风险审查需确认

- 是否构成跨阶段、跨组织、范围不清的高风险合同？
- “所有系统接口”是否形成无限范围承诺？
- “8 家分公司推广”是否存在不可控复制风险？
- 是否存在未审批让利、免费定制、无限服务或责任边界不清问题？
- 是否需要设定阶段性决策门槛，试点通过后再启动推广。

## 交接建议

- **Recommendation：总体处理**
  - 当前不建议直接进入合同签署，应先完成范围拆解、成本测算、验收标准和付款节点确认。

- **Recommendation：给售前**
  - 输出明确的方案范围说明，包括 5 个 Agent、知识库、3 类系统接口、总部试点和 8 家分公司推广边界。
  - 将“所有系统接口”改为“经双方确认的接口清单范围内”，并列明超出范围需走变更。

- **Recommendation：给交付**
  - 评估总部试点和 8 家分公司推广的工作量、实施条件、客户配合要求、交付风险和验收方式。
  - 拆分试点交付物与推广交付物，避免默认一次性覆盖全部实施义务。

- **Recommendation：给财务**
  - 复核报价、成本、毛利、付款里程碑、开票节点和现金流风险。
  - 对接口开发、分公司推广、运维支持和培训是否已计价进行确认。

- **Recommendation：给法务**
  - 审查是否采用“主合同 + 阶段性工作说明书 / 订单”的结构。
  - 明确验收标准不得后置为空白，至少应有阶段验收机制、验收材料、客户反馈期限和争议处理方式。

- **Recommendation：给风险审查**
  - 对“一份合同覆盖全部阶段”“所有系统接口”“验收后续再定”进行专项风险评估。
  - 要求设置范围上限、阶段门槛、变更机制和审批记录。

- **Draft language：仅供内部讨论，非最终合同条款**
  - “本项目范围以双方确认的阶段性工作说明书、接口清单及验收标准为准；未列明的新增接口、分公司差异化需求、额外 Agent、额外培训或持续优化服务，应另行评估工作量、费用及交付周期，并经双方书面确认后实施。”
  - “试点阶段完成并通过双方确认的验收标准后，方可启动后续分公司推广；推广范围、批次、费用、周期和验收方式应在启动前确认。”
