不能直接走合同。

该项目范围大、阶段多、接口多，且客户要求“验收标准后续再定”，存在明显的范围、交付、成本、验收和合同风险。建议先完成方案、范围、报价、验收和风险评审后，再进入合同签署流程。

## 必须复核事项

### 1. 售前复核

需售前确认以下事项：

- 项目整体方案是否已明确：
  - 集团级知识库建设范围；
  - 5 个业务 Agent 的具体功能、边界和交互流程；
  - OA、CRM、售后工单系统的对接方式；
  - 试点与 8 家分公司推广的实施路径。
- “一份合同覆盖全部阶段”是否可拆分为明确阶段：
  - 总部试点阶段；
  - 系统集成阶段；
  - 业务 Agent 建设阶段；
  - 分公司推广阶段。
- 报价是否覆盖全部范围：
  - 5 个 Agent；
  - 3 类系统接口；
  - 8 家分公司推广；
  - 知识库初始化、配置、训练、测试、上线支持等工作。
- 客户提出“报价包含所有系统接口”，需明确接口数量、接口复杂度、第三方配合条件和变更边界。
- 验收标准不能后续再定，至少需形成初版验收口径。

### 2. 交付复核

需交付团队评估以下事项：

- 当前范围是否具备可交付性。
- OA、CRM、售后工单系统是否已有明确接口文档、测试环境、账号权限和客户侧负责人。
- 5 个业务 Agent 的需求是否足够清晰，是否存在后续大量定制风险。
- 集团总部与 8 家分公司的组织、数据、流程是否一致；如不一致，推广工作量可能显著增加。
- 知识库数据来源、数据质量、清洗规则、权限规则是否明确。
- 项目里程碑、交付物、上线标准、试点成功标准是否明确。
- 验收标准后续再定不可接受，需在合同或附件中明确：
  - 功能验收；
  - 接口验收；
  - Agent 效果验收；
  - 知识库准确性或可用性验收；
  - 试点验收；
  - 分公司推广验收。

### 3. 财务复核

需财务确认以下事项：

- 报价是否覆盖完整成本，包括：
  - 产品/平台费用；
  - Agent 配置或开发费用；
  - 系统接口开发费用；
  - 数据处理费用；
  - 项目实施费用；
  - 分公司推广费用；
  - 运维支持费用；
  - 差旅及驻场成本，如有。
- “所有系统接口”是否可能导致成本不可控。
- 一份合同覆盖全部阶段时，付款节点是否与交付阶段匹配。
- 是否需要按阶段收款，避免全部风险集中在最终验收。
- 是否存在低价包干风险。
- 如验收标准未定，收入确认和回款风险较高，需财务重点审查。

### 4. 法务复核

需法务审查以下事项：

- 合同是否允许“验收标准后续再定”；建议不接受该条款。
- 必须在合同或附件中明确：
  - 项目范围；
  - 阶段划分；
  - 交付物；
  - 验收标准；
  - 客户配合义务；
  - 变更机制；
  - 接口对接边界；
  - 延期责任；
  - 付款条件；
  - 知识产权归属；
  - 数据安全与保密责任。
- 对接 OA、CRM、售后工单系统涉及客户系统和数据，应明确数据使用、权限、保密和安全责任。
- 对 8 家分公司推广，应明确是否属于同一合同主体授权范围，避免合同主体和服务对象不一致。
- 若第三方系统接口不开放或客户未配合，需明确我方免责或工期顺延机制。
- 对 Agent 输出结果的准确性、建议性质、责任边界需明确。

### 5. 风险审查

建议进入风险审查，重点关注：

- 范围不清风险：集团知识库、5 个 Agent、3 类系统接口、8 家分公司推广范围较大。
- 验收不清风险：客户要求验收标准后续再定，不建议接受。
- 成本失控风险：报价包含所有系统接口，但接口数量、复杂度和第三方配合情况未知。
- 交付延期风险：涉及多个系统、多个业务 Agent、多个分公司。
- 客户配合风险：数据、接口、账号、流程、测试环境均依赖客户。
- 推广复制风险：8 家分公司的系统、流程、数据权限可能不一致。
- 数据安全风险：知识库和 Agent 可能涉及集团内部业务数据、客户数据或工单数据。
- 合同履约风险：一份合同覆盖全部阶段，若前期范围不清，后期争议可能较大。

## 结论

不建议直接走合同。

应先由售前、交付、财务、法务和风险团队共同复核，至少补齐以下内容后再签约：

- 明确项目范围和阶段；
- 明确 5 个 Agent 的功能边界；
- 明确接口清单、接口责任和变更机制；
- 明确总部试点和 8 家分公司推广范围；
- 明确交付物和验收标准；
- 明确付款节点；
- 明确客户配合义务；
- 明确数据安全、知识产权和责任边界。
