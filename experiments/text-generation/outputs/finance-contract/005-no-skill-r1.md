不能直接走合同。

该项目范围大、阶段多、接口多，且客户要求“验收标准后续再定”，存在明显的交付、成本、商务和法律风险。必须先完成复核与边界确认后，才能进入合同签署流程。

## 必须复核事项

| 事项 | 必须复核角色 | 原因 |
|---|---|---|
| 一份合同覆盖全部阶段 | 售前、交付、法务、风险 | 涉及试点、推广到 8 家分公司，阶段边界、里程碑、变更机制需明确 |
| 集团总部知识库建设范围 | 售前、交付 | 需确认数据范围、知识来源、清洗整理责任、权限体系、更新机制 |
| 5 个业务 Agent 的功能边界 | 售前、交付 | 需明确每个 Agent 的业务场景、能力范围、输入输出、限制条件 |
| 对接 OA、CRM、售后工单系统 | 售前、交付、财务、法务、风险 | 接口数量、复杂度、第三方配合、数据安全、费用和责任边界均需明确 |
| 报价包含所有系统接口 | 售前、交付、财务 | “所有接口”表述过宽，需拆分接口清单、工作量、假设条件和超出范围计费方式 |
| 试点后推广到 8 家分公司 | 售前、交付、财务 | 需确认推广范围、部署方式、差异化配置、培训支持、实施成本 |
| 验收标准后续再定 | 售前、交付、法务、风险 | 不建议接受。验收标准必须在合同或附件中明确，否则存在无法验收、回款延迟和争议风险 |
| 项目总价与付款节点 | 财务、售前、法务 | 需匹配阶段交付、验收节点和现金流风险 |
| 变更管理机制 | 售前、交付、法务、风险 | 防止客户后续扩大范围但不增加费用或周期 |
| 数据安全、系统权限、客户数据使用 | 法务、风险、交付 | 涉及集团知识库、多系统数据接入和多分公司推广，需明确数据责任和合规要求 |
| 第三方系统配合责任 | 法务、交付、售前 | OA、CRM、售后工单系统如由客户或第三方提供，需明确接口资料、账号、测试环境和配合义务 |
| 交付周期与资源投入 | 交付、售前、财务 | 当前范围较复杂，需评估实施排期、人力成本和可交付性 |
| 售后支持与运维边界 | 售前、交付、财务、法务 | 需明确免费支持周期、服务范围、响应时间、后续运维收费 |

## 结论

该项目不能在当前条件下直接走合同。  
至少需要先完成以下前置确认：

1. 明确项目阶段、范围和交付物。  
2. 明确 5 个 Agent 的具体功能和验收标准。  
3. 明确所有系统接口清单、责任边界和费用假设。  
4. 明确试点与 8 家分公司推广的实施边界。  
5. 将验收标准、付款节点、变更机制写入合同或合同附件。  
6. 经售前、交付、财务、法务、风险审查通过后，再进入合同流程。
