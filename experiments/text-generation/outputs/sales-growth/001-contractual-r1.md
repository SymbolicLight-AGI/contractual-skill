## 客户画像

- **客户类型**：华东某离散制造企业。
- **当前关注方向**：正在评估企业知识库和售后服务 Agent。
- **参会角色**：
  - 信息化负责人
  - 售后负责人
  - 生产计划经理
- **初步需求特征**：
  - 售后知识沉淀与统一管理。
  - 标准问答、工单经验复用。
  - 新员工培训效率提升。
  - 数据安全、权限隔离、系统集成风险控制。

- **已知事实**：售后知识分散在微信群、Excel 和老系统中。
- **已知事实**：重复问答较多。
- **已知事实**：新员工培训周期长。
- **已知事实**：客户希望把标准问答和工单经验沉淀下来。
- **已知事实**：客户对数据安全和权限隔离比较敏感。

- **缺失信息**：决策时间表未明确。
- **缺失信息**：是否已经比较竞品未明确。
- **缺失信息**：最终决策人或审批链路未明确。
- **缺失信息**：现有数据规模、数据质量、数据来源细节未明确。
- **缺失信息**：老系统类型、接口能力、集成范围未明确。
- **缺失信息**：期望上线节奏、试点范围、验收标准未明确。

### 我方观察

- **已知事实**：售后负责人比较积极。
- **已知事实**：信息化负责人关注系统集成风险。

## 推断与不确定性

- **推断**：售后负责人可能是业务侧推动者或核心使用方，但是否具备采购决策权不确定。
- **推断**：信息化负责人可能会重点评估数据安全、权限体系、系统对接复杂度和后续运维风险。
- **推断**：客户存在知识管理和售后服务效率提升机会，但当前不能判断其为已确认采购意向。
- **推断**：生产计划经理参会可能说明客户希望知识库或 Agent 后续与生产、计划、工单、备件等流程有关联，但客户未明确表达，需进一步确认。
- **不确定性**：客户是否已有内部立项、预算池或年度数字化规划尚不清楚。
- **不确定性**：客户是否接受先做小范围试点、再扩展到更多业务场景尚不清楚。
- **不确定性**：客户对 Agent 自动回复、辅助坐席、工单推荐、知识检索等具体形态的优先级尚不清楚。

## 痛点与机会

### 痛点

- **已知事实**：知识分散在微信群、Excel 和老系统中，检索和复用效率可能较低。
- **已知事实**：重复问答多，可能造成售后人员时间浪费和响应不一致。
- **已知事实**：新员工培训周期长，知识传承依赖人工经验。
- **已知事实**：数据安全和权限隔离敏感，说明客户对系统可控性要求较高。

### 机会

- **推断机会**：可围绕“售后知识库统一沉淀”设计初步方案，包括标准问答、故障处理经验、工单经验、常见问题分类等。
- **推断机会**：可围绕“售后服务 Agent 辅助问答”探索场景，例如内部售后人员检索、坐席辅助、工单处理建议等。
- **推断机会**：可将“新员工培训”作为价值场景，展示如何通过知识库降低学习成本，但不能承诺具体培训周期缩短结果。
- **推断机会**：若客户老系统中已有大量历史工单，可进一步评估数据治理、知识抽取和问答效果。
- **推断机会**：信息化负责人关注集成风险，若能提前明确边界、接口、权限和部署方式，有助于降低推进阻力。

## 风险信号

- **风险信号**：客户对数据安全和权限隔离敏感，若方案涉及敏感售后数据、客户数据、工单数据，需要明确权限模型、数据边界和访问控制。
- **风险信号**：知识分散在微信群、Excel 和老系统中，数据质量、格式一致性、可迁移性和清洗成本存在不确定性。
- **风险信号**：信息化负责人关注系统集成风险，老系统接口能力、账号体系、权限体系、工单系统对接方式均可能影响范围、成本和周期。
- **风险信号**：预算未明确，不能直接推进报价或商务承诺。
- **风险信号**：决策时间表未明确，项目紧迫性和采购节奏不清晰。
- **风险信号**：竞品比较情况不明，可能存在隐性竞争或客户仍处于早期调研阶段。
- **风险信号**：若客户期望“快速上线、低价固定范围、全系统打通、无限定制”，需要进入风险评审，避免过度承诺。
- **风险信号**：当前不能将售后负责人积极理解为客户已确认购买意向。

## 推荐下一步

1. **补齐业务场景信息**
   - 约一次售后业务专题沟通，聚焦：
     - 重复问答的主要类型。
     - 标准问答数量和来源。
     - 历史工单量、工单字段、典型问题分类。
     - 新员工培训当前流程和主要卡点。
     - 首期最希望解决的 1-2 个高频场景。

2. **补齐技术与数据边界**
   - 与信息化负责人单独或联合沟通：
     - 老系统名称、接口能力、数据导出方式。
     - 是否需要与工单系统、账号体系、权限体系集成。
     - 数据是否可出域、是否有私有化或本地化要求。
     - 不同角色的权限隔离要求。
     - 日志审计、知识审批、敏感内容管控要求。

3. **确认项目推进条件**
   - 明确是否已有内部立项。
   - 明确预算来源或预算区间是否存在。
   - 明确决策链路，包括业务负责人、信息化负责人、采购、财务、最终审批人。
   - 明确期望试点时间窗口和采购节奏。
   - 明确是否正在比较竞品，以及客户主要评估维度。

4. **建议采用“试点范围”推进**
   - 可建议客户先选择一个边界清晰的售后知识场景做试点，例如：
     - 高频售后 FAQ。
     - 某一类产品线的常见故障知识。
     - 历史工单经验检索。
   - 注意：不得承诺最终上线时间、固定交付范围或具体业务结果，需在售前评估后确认。

5. **准备下一次会议材料**
   - 准备面向业务侧的场景梳理模板。
   - 准备面向信息化侧的技术问题清单。
   - 准备数据安全、权限隔离、知识审批流程的通用说明。
   - 准备可讨论的试点方案框架，但不包含最终价格、折扣、交付周期承诺。

## 交接建议

### 需要交接给售前 / Presales

- **交接事项**：
  - 梳理企业知识库与售后服务 Agent 的初步方案边界。
  - 评估标准问答、工单经验、老系统数据的知识沉淀方式。
  - 评估权限隔离、知识审批、角色访问控制的方案可行性。
  - 评估与老系统、工单系统、账号体系的集成可行性。
- **原因**：客户关注方案能力、数据安全和集成风险，需要售前参与明确技术假设。

### 需要交接给交付 PM / Delivery PM

- **交接事项**：
  - 后续如进入试点，需要评估数据准备、知识清洗、测试验收、上线范围。
  - 明确客户侧配合事项，例如数据提供、业务专家标注、权限规则确认。
- **原因**：知识分散在多处，数据治理和试点验收可能影响交付可行性。

### 需要交接给风险评审 / Risk Review

- **交接触发条件**：
  - 客户要求全系统打通。
  - 客户要求快速上线但数据和接口条件不清。
  - 客户要求低价固定范围但包含大量不确定集成或定制。
  - 客户要求承诺问答准确率、降本比例、培训周期缩短比例等结果。
- **原因**：以上内容可能涉及范围、交付、结果承诺风险。

### 暂不建议直接交接给财务合同 / Finance Contract

- **原因**：
  - 预算未明确。
  - 采购节奏未明确。
  - 方案范围和集成边界未明确。
- **后续触发条件**：
  - 客户确认预算或报价需求。
  - 试点范围明确。
  - 付款条款、折扣、合同语言进入讨论阶段。
