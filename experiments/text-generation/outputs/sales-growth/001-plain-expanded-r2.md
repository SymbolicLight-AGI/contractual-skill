## 3. 推断机会

以下为基于拜访记录的销售推断，不等同于客户已确认需求：

- **售后知识库建设机会**  
  客户存在知识分散、重复问答多的问题，可能适合推动售后知识统一归集、分类、检索和维护机制建设。

- **售后服务 Agent 机会**  
  客户正在评估售后服务 Agent，且有标准问答和工单经验沉淀诉求，后续可探索 Agent 在售后问答、工单辅助、知识推荐等场景的适配度。

- **新员工培训场景机会**  
  客户提到培训周期长，可进一步挖掘是否需要基于知识库的培训问答、案例学习、标准流程查询等能力。

- **数据治理与权限方案机会**  
  客户对安全和权限敏感，若能提供清晰的权限隔离、数据访问控制、知识范围管理方案，有助于降低信息化负责人的顾虑。

- **老系统集成机会**  
  售后知识存在于老系统中，后续可能涉及数据迁移、接口集成或知识同步，但需由售前进一步评估可行性和范围。

- **预算不明确**  
  目前无法判断客户是否具备立项预算或采购可能性。

- **决策时间表不明确**  
  尚不清楚客户是早期调研、方案比较，还是近期有明确采购计划。

- **竞品情况不明确**  
  未知客户是否已接触或比较其他厂商，需防止进入被动陪标或仅做方案参考。

- **信息化负责人关注集成风险**  
  若涉及老系统、工单系统、权限体系、企业内部账号体系等集成，可能带来交付复杂度和项目周期不确定性。

  需要谨慎回应，不应直接承诺完全满足所有安全要求，应交由售前或安全方案团队评估。

- **需求范围尚未固化**  
  当前仅能确认方向，不能将“知识库 + Agent + 集成 + 培训”直接视为已确定项目范围。

## 5. 推荐下一步动作

### 5.1 销售跟进动作

1. **确认客户项目阶段**
   - 了解当前是内部调研、方案评估、预算申请，还是已进入采购流程。
   - 确认是否已有明确牵头部门和项目负责人。

2. **补齐商业信息**
   - 询问预算是否已规划。
   - 了解决策链条：信息化、售后、生产、采购、管理层分别扮演什么角色。
   - 了解预期决策时间和可能上线窗口，但不承诺交付周期。

3. **推动一次售前需求澄清会**
   - 重点围绕售后知识来源、权限模型、老系统情况、Agent 使用场景进行梳理。
   - 建议邀请信息化负责人和售后负责人同时参加，平衡业务价值和技术风险。

4. **围绕售后负责人建立业务价值共识**
   - 聚焦重复问答减少、知识沉淀、新员工培训提效等业务痛点。
   - 收集具体案例，例如高频问题类型、典型工单、培训资料、现有知识分布情况。

5. **围绕信息化负责人降低技术顾虑**
   - 准备数据安全、权限隔离、部署方式、系统集成边界等说明材料。
   - 对集成范围保持审慎，不提前承诺可集成所有系统。

## 6. 建议询问客户的问题

### 业务需求类

- 当前售后重复问答主要集中在哪些产品线或问题类型？
- 新员工培训目前通常需要多久？主要卡点是什么？
- 标准问答和工单经验目前是否已有沉淀规范？
- 售后负责人期望优先解决的是知识检索、自动问答，还是工单辅助？

### 数据与系统类

- 售后知识目前分别存放在哪些系统或文件中？
- 老系统是否支持接口、导出或数据同步？
- 是否已有统一账号体系或权限管理体系？
- 权限隔离需要按部门、角色、产品线、客户，还是其他维度？

### 商务与决策类

- 该项目是否已有预算规划？
- 是否已有明确采购时间表？
- 当前是否已接触或比较其他厂商？
- 内部决策流程涉及哪些部门和关键角色？
