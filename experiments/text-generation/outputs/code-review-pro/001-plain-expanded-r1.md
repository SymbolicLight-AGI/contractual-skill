## 主要发现（按严重程度）

### 高危：SQL 注入风险
`customer_id` 来自 HTTP query 参数，但被直接拼接进 SQL：

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

攻击者可以构造恶意参数改变 SQL 语义，例如读取其他客户备注、绕过条件，甚至在特定数据库/驱动配置下执行破坏性语句。

**影响：**
- 可能泄露 CRM 客户备注，属于敏感业务数据。
- 可能访问非授权客户数据。
- 可能导致数据篡改或删除，具体取决于数据库权限和驱动行为。
- 内部系统也不能假设输入可信，HTTP query 参数应视为不可信输入。

**建议：**
- 使用数据库驱动提供的参数化查询。
- 不要用 f-string、字符串拼接或 `%` 格式化构造 SQL。
- 对 `customer_id` 做类型校验和边界校验，例如必须为合法整数。

---

### 高危：缺少授权/访问控制上下文
函数只根据 `customer_id` 查询备注，没有体现当前用户是否有权限访问该客户。

**影响：**
- 如果上层接口没有严格鉴权和客户范围校验，用户可能通过修改 query 参数读取任意客户备注。
- CRM 备注通常可能包含隐私、销售、合同或支持信息，越权读取风险较高。

**不确定性：**
- 代码片段未展示调用方是否已经完成认证、授权和租户/组织隔离校验，因此不能确定当前系统一定存在越权漏洞。
- 但该函数本身没有防护，依赖上层时应明确约定并测试。

**建议：**
- 确认调用方在进入该函数前已校验当前用户对 `customer_id` 的访问权限。
- 如系统支持多租户或组织隔离，应确保查询条件或调用路径包含租户/组织范围约束。

---

### 中危：输入类型和错误处理不明确
`customer_id` 来自 HTTP query 参数，通常是字符串。当前代码没有验证其是否为整数。

**影响：**
- 非数字输入可能导致 SQL 语法错误。
- 空字符串、特殊字符、超大数字等输入可能导致异常或异常查询。
- 错误行为依赖数据库，可能表现不一致。
- 如果上层将异常直接返回，可能暴露数据库错误信息。

**建议：**
- 在进入查询前明确校验 `customer_id`。
- 对非法参数返回明确的客户端错误，例如 400。
- 不建议用宽泛的 `except Exception` 静默吞掉错误并返回空列表，因为这会掩盖真实问题并导致误判。

---

### 中危：`SELECT *` 带来数据暴露和维护风险
当前查询返回 notes 表所有字段：

```sql
SELECT * FROM notes
```

**影响：**
- 如果 notes 表包含内部字段、审计字段、删除标记、敏感元数据或未来新增字段，可能被无意返回给调用方。
- 表结构变化会影响 API 返回内容，降低可维护性。
- 可能增加不必要的数据传输和序列化成本。

**建议：**
- 明确列出业务需要返回的字段。
- 确认是否需要过滤软删除、归档、内部备注等字段。
- 如果备注按时间展示，建议明确排序规则。

---

### 中危：结果集可能无限制返回
函数没有分页、数量限制或排序。

**影响：**
- 某些客户备注数量较多时可能导致慢查询、高内存占用或接口超时。
- 返回顺序不稳定，影响前端展示和测试稳定性。
- 可能造成内部服务的性能风险。

**建议：**
- 根据业务需求增加分页、limit 和稳定排序。
- 确认是否需要按创建时间倒序或正序返回。

---

## 测试建议

### 安全测试
- 使用恶意 `customer_id` 输入验证不会发生 SQL 注入，例如包含 SQL 运算符、注释符、引号、分号等字符。
- 验证参数化查询后，恶意输入被当作普通值处理，而不是改变 SQL 结构。
- 验证用户无法通过修改 `customer_id` 读取无权限客户的备注。
- 如果系统有租户/组织隔离，增加跨租户访问测试。

### 正确性测试
- 合法 customer_id 能返回对应客户的备注。
- 不存在的 customer_id 返回空列表。
- 非法输入，例如空字符串、非数字、负数、超大数字，返回预期错误或被明确拒绝。
- 验证返回字段只包含业务允许暴露的字段。
- 如果业务要求排序，测试返回顺序稳定。

### 性能和可靠性测试
- 对备注数量较多的客户测试接口性能。
- 测试分页或 limit 行为。
- 测试数据库异常时不会泄露底层 SQL 错误细节。
- 避免通过吞异常返回空列表掩盖数据库故障。

### 回归测试
- 增加单元测试覆盖 SQL 参数绑定行为。
- 增加集成测试覆盖真实数据库查询路径。
- 增加 API 层测试覆盖 HTTP query 参数到数据库查询的完整路径。
