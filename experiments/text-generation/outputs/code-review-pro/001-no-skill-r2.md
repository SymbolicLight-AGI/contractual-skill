## 主要问题

### 1. 安全风险：SQL 注入

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

`customer_id` 来自 HTTP query 参数，直接拼接进 SQL。攻击者可以构造恶意输入，例如：

```text
1 OR 1=1
```

导致查询返回所有客户备注，造成敏感数据泄露。

更严重时，如果数据库/驱动允许多语句执行，可能进一步造成数据破坏。

这是当前代码中最主要的问题。

---

### 2. 数据越权风险

函数只根据 `customer_id` 查询备注，没有体现任何权限校验或租户隔离逻辑。

如果调用方没有严格校验当前用户是否有权限访问该客户，攻击者可能通过修改 query 参数访问其他客户的备注。

虽然权限校验可能在其他层完成，但从该函数和背景信息看，这一点存在不确定性，需要确认。

---

### 3. 正确性风险：未校验 `customer_id` 类型

`customer_id` 来自 HTTP query 参数，通常是字符串。

当前代码没有校验它是否为合法的整数 ID。可能出现：

- 空字符串
- 非数字字符串
- 负数
- 超大数字
- 带 SQL 片段的字符串
- `None`

这些输入可能导致 SQL 报错、异常传播，或触发安全问题。

---

### 4. 正确性风险：`SELECT *` 不稳定

使用 `SELECT *` 会让返回结果依赖表结构。

如果 `notes` 表新增、删除或调整字段，返回内容可能发生变化，影响调用方逻辑或暴露不应返回的字段。

在 CRM 备注场景下，如果表中存在内部字段、审计字段或敏感字段，`SELECT *` 也可能带来数据泄露风险。

---

### 5. 错误处理不明确

当前函数没有处理数据库异常，例如：

- 连接失败
- SQL 执行失败
- `customer_id` 非法导致 SQL 错误
- `fetchall()` 失败
- `dict(row)` 转换失败

如果异常直接向上抛出，可能导致 API 返回 500，甚至泄露内部错误信息。是否由上层统一处理目前不明确。

---

### 6. `dict(row)` 兼容性风险

```python
return [dict(row) for row in rows]
```

这依赖于 `row` 对象支持直接转换为字典。

不同数据库驱动或 cursor 配置下，`row` 可能是 tuple、Row 对象或其他结构。当前代码可能在某些环境下失败。

---

### 7. 无排序导致结果不稳定

查询没有 `ORDER BY`：

```sql
SELECT * FROM notes WHERE customer_id = ...
```

数据库不保证返回顺序。备注列表如果展示给 CRM 用户，顺序可能在不同执行中变化，导致体验或测试不稳定。

---

### 8. 性能与分页风险

当前直接 `fetchall()` 获取所有备注。

如果某个客户备注很多，可能导致：

- 单次查询返回大量数据
- 内存占用升高
- API 响应变慢
- 潜在 DoS 风险

是否需要分页、限制条数或按时间排序，需要结合业务需求确认。

---

## 测试建议

### 1. SQL 注入相关测试

需要覆盖恶意 `customer_id` 输入，例如：

- `"1 OR 1=1"`
- `"1; DROP TABLE notes; --"`
- `"abc"`
- `"1 --"`

预期应确认不会返回其他客户备注，也不会执行非预期 SQL。

当前实现大概率无法通过这类安全测试。

---

### 2. 合法输入测试

覆盖正常客户 ID：

- 存在备注的客户 ID
- 没有备注的客户 ID

验证返回结果是否正确，尤其是：

- 只返回对应客户的备注
- 无备注时返回空列表
- 返回结构是否符合 API 预期

---

### 3. 非法输入测试

覆盖来自 HTTP query 参数的异常值：

- 空字符串
- `None`
- 非数字字符串
- 负数
- 小数
- 超大整数
- 带空格的字符串

验证系统行为是否明确，例如返回参数错误，而不是数据库异常或 500。

---

### 4. 越权访问测试

模拟用户访问不属于自己的 `customer_id`：

- 普通用户访问其他客户备注
- 不同租户之间访问客户备注
- 无权限用户访问备注

需要验证不会因为仅修改 query 参数而读取他人数据。

如果权限控制在上层，应增加集成测试覆盖该链路。

---

### 5. 返回字段测试

由于当前使用 `SELECT *`，建议测试返回字段是否包含敏感或不应暴露字段，例如：

- 内部标记
- 删除标记
- 审计字段
- 仅内部可见备注
- 创建人/修改人内部信息

如果 API 有固定响应契约，应验证返回字段稳定。

---

### 6. 排序测试

如果备注展示需要按创建时间或更新时间排序，应增加测试确认排序规则稳定。

当前代码没有排序，测试结果可能不稳定。

---

### 7. 数据库异常测试

模拟数据库执行失败、连接失败或查询异常，验证：

- 是否有统一错误处理
- 是否不会泄露 SQL 或内部堆栈信息
- API 返回是否符合预期错误格式

---

### 8. 大数据量测试

构造某客户拥有大量备注的场景，验证：

- 响应时间
- 内存使用
- 是否需要分页或限制返回数量

当前 `fetchall()` 在大数据量下存在风险。

---

## 总结

这段变更的核心风险是：`customer_id` 来自 HTTP query 参数，却被直接拼接进 SQL，存在明显 SQL 注入风险。同时还存在参数校验不足、潜在越权访问、`SELECT *` 暴露字段、结果顺序不稳定、无分页和缺少测试覆盖等问题。

建议优先补充安全测试、参数校验测试、权限访问测试和正常查询测试。
