## 主要发现与严重性

### Critical：SQL 注入风险
`customer_id` 来自 HTTP query 参数，但直接通过 f-string 拼接进 SQL：

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

攻击者可以构造恶意参数改变 SQL 语义，例如读取其他客户备注、绕过过滤条件，甚至在某些数据库/驱动配置下造成数据破坏。

**影响：**
- 可能泄露 CRM 中其他客户的备注信息。
- 可能导致越权访问敏感业务数据。
- 如果数据库权限过大，存在进一步修改或删除数据的风险。

**建议方向：**
- 使用数据库参数化查询。
- 对 `customer_id` 做类型校验，确保为预期的整数或 UUID 等格式。
- 不应通过简单字符串过滤来“修补”注入问题。

---

### High：缺少授权/访问控制上下文
函数只根据 `customer_id` 查询备注，代码中没有体现当前用户是否有权限访问该客户备注。

**影响：**
- 即使修复 SQL 注入，攻击者或内部用户仍可能通过修改 `customer_id` 查询不属于自己权限范围内的客户备注。
- CRM 备注通常可能包含敏感信息，越权读取风险较高。

**不确定性：**
- 目前无法确认授权是否在调用方、路由层或中间件中完成。
- 如果授权在外层已完成，应在接口契约或代码注释中明确。

**建议方向：**
- 后端/安全负责人确认访问控制边界。
- 测试应覆盖“无权限用户不能读取其他客户备注”的场景。

---

### Medium：输入类型和边界处理不明确
`customer_id` 来自 HTTP query 参数，通常是字符串。当前函数没有做类型转换或校验。

**影响：**
- 非数字、空字符串、特殊字符会导致 SQL 错误或触发注入风险。
- 错误行为可能依赖数据库实现，造成不一致结果。
- 无法区分“参数非法”和“客户不存在”。

**建议方向：**
- 明确 `customer_id` 的合法格式。
- 非法输入应返回明确错误，而不是直接进入查询。
- 测试应覆盖空值、非数字、负数、超大值、特殊字符等情况。

---

### Medium：`SELECT *` 可能导致不必要的数据暴露和维护风险
函数返回 `notes` 表中所有字段：

```python
SELECT * FROM notes
```

并直接转成字典返回。

**影响：**
- 如果表中包含内部字段、审计字段、删除标记、敏感元数据，可能被上层接口意外暴露。
- 表结构变化会影响接口返回内容，造成兼容性风险。
- 查询不必要字段也会增加数据传输和处理成本。

**建议方向：**
- 明确接口需要返回哪些字段。
- 避免直接返回整行数据库记录。
- 产品/后端确认备注接口的返回字段契约。

---

### Medium：缺少错误处理策略
当前数据库执行失败会直接抛出异常。是否合适取决于调用层是否有统一错误处理。

**影响：**
- 如果上层没有统一处理，可能导致 500 错误或暴露数据库错误信息。
- 如果后续添加错误处理，不应使用宽泛 `except Exception` 后返回空列表，因为这会掩盖真实故障，让调用方误以为客户没有备注。

**不确定性：**
- 当前无法确认调用栈是否已有统一异常处理和日志记录。

**建议方向：**
- 明确数据库异常由哪一层处理。
- 错误应记录并返回合适的错误响应，不应静默返回成功样式结果。

---

## 测试建议

### 安全测试
- SQL 注入 payload 测试，例如构造会改变查询条件的 `customer_id`。
- 验证恶意输入不会返回其他客户备注。
- 验证恶意输入不会导致数据库结构或数据被修改。
- 越权访问测试：用户 A 请求用户 B 或无权限客户的 `customer_id`，应被拒绝。

### 正确性测试
- 合法 `customer_id` 返回对应客户备注。
- 不存在的 `customer_id` 返回空结果或约定响应。
- 非法输入测试：
  - 空字符串
  - 缺失参数
  - 字母字符串
  - 特殊字符
  - 负数
  - 超大数字
- 数据库异常时是否按系统约定返回错误，而不是误报为空列表。

### 数据契约测试
- 返回字段应符合接口契约，不应因为 `notes` 表新增字段而自动暴露。
- 确认返回结果排序是否有要求；当前 SQL 没有 `ORDER BY`，如果业务要求按时间排序，需要测试并明确。

### 回归测试
- 参数化查询修复后，确保正常客户备注查询仍可用。
- 确认修复不会破坏现有调用方对返回格式的预期。

## 需要跟进的负责人

- **后端负责人**：修复 SQL 拼接、输入校验、返回字段契约和错误处理策略。
- **安全负责人**：确认 CRM 备注访问控制要求，补充注入和越权访问测试。
- **测试负责人**：补充安全、正确性、异常和数据契约测试。
- **产品/业务负责人**：确认客户备注中哪些字段允许返回，以及备注排序规则是否有业务要求。

## 总体判断

当前变更存在高风险安全问题，尤其是 HTTP 参数直接拼接 SQL，不能上线。需要先修复 SQL 注入风险，并确认授权、输入校验、返回字段和测试覆盖后再考虑合入。
