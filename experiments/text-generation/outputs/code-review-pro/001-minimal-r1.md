## 主要问题

### 高严重性：SQL 注入风险

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

`customer_id` 来自 HTTP query 参数，直接拼接进 SQL。攻击者可以传入恶意内容改变查询语义，例如绕过条件、读取其他客户备注，甚至在部分数据库/驱动配置下执行破坏性语句。

这是当前代码中最主要的安全问题。

风险包括：

- 查询非目标客户的备注；
- 泄露 CRM 中敏感客户信息；
- 破坏或篡改数据，具体取决于数据库驱动和权限配置；
- 内部系统如果缺少边界校验，仍然可能被滥用，不应因为是“内部 CRM”而忽略注入风险。

---

### 高严重性：缺少输入校验

`customer_id` 来自 HTTP query 参数，通常是字符串。当前代码没有校验它是否为合法客户 ID。

可能问题：

- 非数字输入导致 SQL 错误；
- 空值、负数、超大数字等异常值未处理；
- 特殊字符可触发 SQL 注入；
- 类型不一致可能导致数据库隐式转换，产生不可预期行为。

需要明确 `customer_id` 的合法格式，例如是否必须是正整数。

---

### 中严重性：可能缺少访问控制校验

函数只根据 `customer_id` 查询备注，没有体现当前用户是否有权限访问该客户备注。

由于背景中说明这是 CRM 客户备注，通常属于敏感信息。即使 SQL 注入被修复，如果上层接口允许任意传入 `customer_id`，用户也可能通过枚举 ID 读取其他客户备注。

由于片段中没有展示调用方逻辑，不能确定是否已在上层做了权限控制，但这是需要重点确认的风险。

---

### 中严重性：使用 `SELECT *` 可维护性和数据暴露风险

```python
SELECT * FROM notes
```

`SELECT *` 会返回所有列，存在几个问题：

- 未来表结构新增敏感字段时，可能被该接口意外返回；
- 调用方依赖隐式字段集合，可维护性差；
- 可能返回不必要的数据，增加传输和内存开销；
- 字段顺序或字段名变化可能影响 `dict(row)` 的行为，取决于数据库驱动。

建议明确需要返回的字段，但 review 中不直接修改代码。

---

### 中严重性：错误处理不明确

当前代码没有处理数据库异常、非法输入、查询失败等情况。

例如：

- 数据库连接失败；
- SQL 执行异常；
- `customer_id` 不合法；
- `row` 不能直接转换为 `dict`。

这些异常会如何暴露给 HTTP 客户端，取决于上层框架。如果未统一处理，可能导致：

- 返回 500；
- 泄露 SQL 或数据库错误信息；
- 日志中记录敏感数据；
- 用户体验不稳定。

---

### 低到中严重性：`dict(row)` 依赖数据库返回对象行为

```python
return [dict(row) for row in rows]
```

这依赖 `row` 对象支持直接转换为字典。不同数据库库或 cursor 配置下行为可能不同。

如果 `row` 是 tuple，`dict(row)` 可能报错。若当前项目中 db row 已配置为 mapping row，则没问题，但片段中无法确认。

---

### 低到中严重性：性能风险不明确

该函数没有分页、排序或限制返回数量：

```python
fetchall()
```

如果某个客户有大量备注，可能一次性加载过多数据，导致接口响应慢或内存占用高。

此外，`customer_id` 字段是否有索引未知。如果没有索引，查询可能随数据量增长变慢。

由于缺少表结构和业务约束，这一点需要进一步确认。
