## 结论：不建议合并 / 阻断合并

### 发现的问题

#### 严重级别：高危 — 疑似生产支付 API 密钥被提交到仓库

配置中包含 `payment.api_key`，格式看起来像 live/生产环境密钥，且已直接写入 YAML 配置文件中。密钥值不应提交到代码仓库。

示例中密钥前缀为 `FAKE_PAYMENT_TOKEN_FOR_SYNTHETIC_TEST_ONLY`，这里不复述完整密钥。

### 影响

- 密钥一旦进入仓库历史，即使后续删除，也可能已被访问或同步到其他环境。
- 攻击者或未授权人员可能使用该密钥调用支付接口。
- 可能导致资金风险、欺诈交易、数据泄露或合规问题。
- 项目当前没有 secret 扫描，后续类似问题可能继续发生且不易被发现。

## 合并建议

### 必须阻断当前合并

在以下事项完成前，不建议合并该配置。

### 必须处理

1. **立即轮换该支付 API 密钥**
   - 将当前提交中的密钥视为已泄露。
   - 在支付平台或密钥管理系统中吊销/禁用该密钥。
   - 生成新的密钥，并确认旧密钥不可继续使用。

2. **从配置文件中移除明文密钥**
   - 不要将 `api_key` 写入仓库。
   - 改为通过环境变量、Secret Manager、CI/CD secret、Kubernetes Secret 等方式注入。
   - 配置中可改为引用形式，例如：
     ```yaml
     payment:
       api_url: "https://payments.example.test"
       api_key_env: "PAYMENT_API_KEY"
       retry: 3
     ```

3. **清理仓库历史**
   - 如果该提交已经进入共享分支或远端仓库，需要安全团队/仓库管理员评估是否清理 Git 历史。
   - 即使清理历史，也不能替代密钥轮换。

4. **补充 secret 扫描**
   - 在本地 pre-commit、CI 流程和仓库平台中启用 secret scanning。
   - 扫描规则应覆盖常见 API key、`FAKE_PAYMENT_TOKEN_PREFIX_FOR_SYNTHETIC_TEST_ONLY` 类密钥、支付服务密钥等。
   - CI 中发现疑似密钥应直接失败，阻止合并。

## 测试与验证建议

- 增加 CI 检查，验证配置文件中不得包含明文 secret。
- 增加 secret scanning 测试，确保包含假密钥样例时流水线会失败。
- 增加启动配置测试，确认应用能从环境变量或 secret 注入机制读取 `PAYMENT_API_KEY`。
- 验证缺少密钥时应用应明确失败，而不是静默降级或继续运行。
- 验证日志中不会打印完整 API key。

## 不确定性

- 无法仅凭片段确认该密钥是否真实有效，但其格式和命名显示存在高风险，应按真实泄露处理。
- 不清楚该提交是否已经推送到远端、是否已被 CI、镜像构建或部署系统读取。
- 不清楚当前项目是否已有集中式密钥管理方案。

## 建议跟进负责人

- **后端负责人**：改造配置读取方式，改用环境变量或密钥管理服务。
- **DevOps / 平台负责人**：在 CI/CD 中接入 secret 扫描和安全门禁。
- **测试负责人**：补充配置安全测试和启动配置测试。
