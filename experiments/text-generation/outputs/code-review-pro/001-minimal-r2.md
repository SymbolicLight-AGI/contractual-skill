## 主要问题（按严重程度）

### 高危：SQL 注入风险

```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```

`customer_id` 来自 HTTP query 参数，直接拼接进 SQL。攻击者可以传入类似：

```text
1 OR 1=1
```

或更恶意的 SQL 片段，导致越权读取、数据泄露，甚至在某些数据库配置下造成数据破坏。

**风险：**

- 读取其他客户备注
- 绕过查询条件
- 泄露 CRM 内部敏感信息
- 如果数据库/驱动允许多语句，可能进一步扩大影响

**建议：**

- 使用数据库驱动提供的参数化查询机制
- 不要通过字符串拼接构造 SQL
- 对 `customer_id` 做明确的类型校验和格式校验

---

### 高危：缺少访问控制上下文

函数只根据 `customer_id` 查询备注，没有体现当前用户是否有权限访问该客户备注。

虽然权限控制可能在调用层完成，但从当前代码片段无法确认。

**风险：**

- 已登录用户通过修改 query 参数访问其他客户备注
- 内部 CRM 中不同角色之间的数据隔离失效

**建议：**

- 确认调用方是否校验当前用户对 `customer_id` 的访问权限
- 如果该函数处于数据访问层，应明确由上层传入已授权的客户 ID，或在服务层统一做权限检查
- 测试中应覆盖越权访问场景

---

### 中危：`customer_id` 类型和格式未校验

背景说明 `customer_id` 来自 HTTP query 参数，通常是字符串。当前函数没有验证它是否为合法 ID。

**风险：**

- 非数字输入导致 SQL 错误
- 空字符串、`None`、负数、超大数字等边界输入行为不明确
- 错误信息可能暴露数据库细节

**建议：**

- 在进入数据库查询前校验 `customer_id`
- 明确非法输入时的行为，例如返回 400、抛出业务异常或拒绝查询
- 不应通过拼接 SQL 来“隐式处理”类型

---

### 中危：使用 `SELECT *` 可维护性和数据暴露风险

```python
SELECT * FROM notes
```

`SELECT *` 会返回表中的所有字段。

**风险：**

- 表结构变化后，函数返回内容可能无意变化
- 如果 notes 表未来新增敏感字段，可能被直接暴露给 API 调用方
- 增加不必要的数据传输和处理成本

**建议：**

- 明确列出需要返回的字段
- 保持 API 返回结构稳定
- 对敏感字段做显式排除或脱敏

---

### 中危：结果顺序未定义

当前查询没有 `ORDER BY`。

**风险：**

- 不同数据库执行计划下返回顺序不稳定
- 前端或调用方如果依赖备注顺序，可能出现间歇性问题
- 测试结果可能不稳定

**建议：**

- 明确备注排序规则，例如按创建时间倒序或正序
- 测试中不要依赖未定义排序

---

### 低到中危：异常处理行为不明确

```python
rows = db.execute(query).fetchall()
```

数据库连接错误、SQL 错误、类型错误等异常未处理。

**风险：**

- HTTP 层可能返回 500
- 错误信息可能泄露内部 SQL 或数据库结构
- 调用方无法区分非法输入、无数据、数据库异常等情况

**建议：**

- 明确异常处理边界：该函数是否只负责查询，还是需要转换为业务错误
- 确保上层不会把数据库异常细节直接返回给用户
- 测试数据库异常时的行为

---

### 低危：`dict(row)` 依赖数据库 row 对象行为

```python
return [dict(row) for row in rows]
```

这依赖 `row` 对象支持直接转换为 dict。不同数据库驱动或 row factory 配置下行为可能不同。

**风险：**

- 更换数据库驱动或配置后函数失败
- 测试环境和生产环境行为不一致

**建议：**

- 明确 db 返回的 row 类型
- 在测试中覆盖正常行转换行为
- 如果项目中已有统一的数据访问封装，应复用一致的转换方式

---

## 测试建议

当前没有测试，建议至少补充以下测试类别：

### 1. SQL 注入防护测试

覆盖恶意 `customer_id` 输入，例如：

- `"1 OR 1=1"`
- `"1; DROP TABLE notes; --"`
- `"' OR '1'='1"`
- `"abc"`

期望结果应是拒绝非法输入或安全地作为参数处理，而不是扩大查询范围或执行异常 SQL。

---

### 2. 合法输入测试

覆盖正常客户 ID：

- 存在备注的客户 ID，返回备注列表
- 没有备注的客户 ID，返回空列表
- 多条备注时，确认返回字段和排序符合预期

---

### 3. 类型和边界测试

覆盖：

- `customer_id=None`
- 空字符串 `""`
- 空白字符串 `"   "`
- 负数 `"-1"`
- 浮点数形式 `"1.5"`
- 超大数字
- 非数字字符串 `"abc"`

需要明确这些输入应返回错误、空结果，还是被拒绝。

---

### 4. 权限/越权测试

如果该函数由 HTTP 接口调用，应测试：

- 用户只能访问自己有权限的客户备注
- 修改 query 参数为其他客户 ID 时不能返回数据
- 管理员、普通销售、客服等不同角色访问结果是否符合权限模型

如果权限不在该函数中实现，也应在接口或服务层测试中覆盖。

---

### 5. 返回字段测试

确保返回结果只包含预期字段，例如：

- note 内容
- 创建时间
- 创建人
- 备注 ID

同时确认不会返回内部字段或敏感字段，例如：

- 删除标记
- 内部审计字段
- 其他不应暴露的客户信息

---

### 6. 异常路径测试

模拟：

- 数据库连接失败
- SQL 执行失败
- row 转 dict 失败
- 数据库超时

确认系统不会向 HTTP 响应中暴露 SQL、堆栈、连接字符串等内部信息。

---

## 总体结论

当前代码的主要风险是：`customer_id` 来自 HTTP query 参数却直接拼接 SQL，存在严重 SQL 注入风险。同时缺少输入校验、权限确认、明确返回字段和测试覆盖。建议优先修复查询构造方式和访问控制验证，并补充安全、正确性和异常路径测试。
