## PR 风险 Review 结果

### 总体结论

该 PR 存在高风险问题，不建议合并。  
新增“用户导出功能”涉及姓名、手机号、邮箱、最近订单金额等敏感或隐私数据，但当前权限控制、审计、导出限制均不足，存在越权导出和大规模数据泄露风险。

---

## 高优先级风险

### 1. 权限控制风险：后端 API 未校验管理员角色

**严重级别：高 / 阻塞合并**

当前仅前端按钮做管理员判断，后端 API 未检查角色。

风险：

- 非管理员用户可能通过直接调用 API 导出用户数据。
- 前端权限判断可以被绕过，不能作为安全边界。
- 如果接口被脚本、抓包工具或内部误用调用，可能造成批量数据泄露。

建议：

- 后端必须进行鉴权和角色校验。
- 导出接口应明确限制为具备相应权限的管理员或特定角色。
- 不应依赖前端按钮隐藏来实现访问控制。
- 对无权限访问应返回明确的拒绝结果，并记录安全事件。

需要交给：

- **后端负责人**：补充 API 级别权限校验。
- **安全负责人**：确认权限模型、越权风险和访问控制策略。

---

### 2. 隐私数据泄露风险：导出字段包含敏感信息

**严重级别：高 / 阻塞合并**

导出内容包括：

- 姓名
- 手机号
- 邮箱
- 最近订单金额

这些字段涉及个人身份信息和交易相关信息，属于敏感或隐私数据。

风险：

- 未经授权导出可能导致个人信息泄露。
- 导出的文件可能被转发、误发或长期保留。
- 最近订单金额属于交易/消费相关信息，可能需要更严格的访问控制和合规确认。
- 如果没有脱敏或字段最小化策略，导出范围可能超过业务必要性。

建议：

- 明确导出目的和使用场景。
- 确认每个字段是否必要，遵循最小化原则。
- 评估是否需要对手机号、邮箱进行脱敏。
- 确认导出文件的保存、下载、过期、传输安全策略。
- 明确是否需要用户数据处理合规审批。

需要交给：

- **产品负责人**：确认导出字段的业务必要性和使用范围。
- **安全负责人**：确认隐私保护、脱敏、合规和数据处理要求。
- **后端负责人**：根据确认结果实现字段控制或脱敏逻辑。

---

### 3. 缺少审计日志

**严重级别：高**

当前没有审计日志。

风险：

- 无法追踪谁导出了用户数据。
- 发生数据泄露时无法定位责任人和导出范围。
- 无法满足敏感操作的审计要求。
- 无法发现异常导出行为，例如频繁导出、大量导出、非工作时间导出等。

建议审计日志至少记录：

- 操作人 ID / 角色
- 导出时间
- 导出条件
- 导出字段
- 导出数量
- 请求来源 IP / 设备信息，如系统已有能力
- 导出是否成功
- 文件标识或任务 ID，如有异步导出机制

需要交给：

- **后端负责人**：补充审计日志实现。
- **安全负责人**：确认审计字段、留存周期和告警策略。
- **产品负责人**：确认审计行为是否影响管理员使用流程。

---

### 4. 没有导出数量限制

**严重级别：高**

当前未设置导出数量限制。

风险：

- 管理员或被盗账号可以一次性导出大量用户数据。
- 接口可能被滥用导致数据库压力、服务性能下降。
- 批量导出会扩大数据泄露影响面。
- 没有分页、异步任务或限流机制时，可能引发超时、内存占用过高、服务不可用等问题。

建议：

- 设置单次导出最大数量。
- 设置导出频率限制。
- 对大数据量导出使用异步任务。
- 对导出任务设置权限、审批或二次确认。
- 必要时增加导出前的统计预览，例如“本次将导出 N 条记录”。
- 对异常导出行为增加告警。

需要交给：

- **产品负责人**：确认合理的单次导出上限、审批流程和使用场景。
- **后端负责人**：实现分页、限流、异步导出或数量限制。
- **安全负责人**：确认大规模导出的安全策略和告警要求。

---

## 中优先级风险

### 5. 前后端权限逻辑不一致，易造成维护问题

**严重级别：中**

当前前端有管理员判断，后端没有相应判断，说明权限逻辑分散且不一致。

风险：

- 后续维护者可能误以为接口已经安全。
- 前端权限变更和后端权限变更可能不同步。
- 新客户端、脚本或第三方入口调用 API 时会绕过前端限制。

建议：

- 后端作为最终权限控制点。
- 前端仅用于控制展示和交互体验。
- 权限规则应有统一定义或清晰文档。

需要交给：

- **后端负责人**：保证服务端权限闭环。
- **产品负责人**：确认哪些角色可以看到导出入口。
- **安全负责人**：确认权限设计是否符合最小权限原则。

---

### 6. 缺少异常场景和边界处理说明

**严重级别：中**

变更摘要未说明导出失败、无数据、超大数据量、部分字段为空等情况的处理方式。

风险：

- 导出结果不一致或不可用。
- 异常时可能暴露内部错误信息。
- 大数据导出可能导致接口超时或服务异常。

建议：

- 明确导出失败时的用户提示。
- 明确空结果导出的行为。
- 明确字段为空、格式异常时的处理方式。
- 避免返回敏感错误信息。

需要交给：

- **后端负责人**：处理导出边界和异常。
- **产品负责人**：确认用户体验和错误提示。

## 测试风险与建议

### 必须补充的测试

#### 权限测试

- 非登录用户调用导出 API，应被拒绝。
- 普通用户直接调用导出 API，应被拒绝。
- 非管理员但能看到或猜到接口地址的用户，应无法导出。
- 管理员用户调用导出 API，应允许。
- 权限失效、账号禁用、角色变更后，应无法继续导出。

#### 隐私与字段测试

- 验证导出字段是否仅包含产品确认过的字段。
- 验证手机号、邮箱是否按安全要求脱敏。
- 验证最近订单金额是否只在授权场景下导出。
- 验证无关字段不会被导出。

#### 审计日志测试

- 成功导出后应产生审计日志。
- 失败导出也应产生必要日志，尤其是无权限访问。
- 审计日志应包含操作人、时间、导出数量、导出条件等关键字段。
- 普通用户越权调用导出接口时应记录安全事件。

#### 数量限制与性能测试

- 超过单次导出上限时应被拒绝或转为受控流程。
- 高频导出应触发限流或告警。
- 大数据量导出不应导致接口超时、内存溢出或数据库压力异常。
- 并发导出场景下系统应稳定。

#### 安全测试

- 直接调用后端 API，确认前端限制不可绕过。
- 验证是否存在越权导出其他范围用户数据的问题。
- 验证导出接口是否受 CSRF、重放、脚本批量调用等风险影响；具体防护需结合现有系统上下文确认。
- 验证导出文件下载地址是否有权限校验和过期机制；如果当前实现没有文件下载环节，则该项需根据实际实现确认。

## 需要继续确认和处理的负责人

### 安全负责人需要确认

- 用户导出功能是否符合公司隐私和安全要求。
- 导出字段是否需要脱敏。
- 最近订单金额是否属于更高敏感级别数据。
- 导出接口的权限模型是否合理。
- 是否需要审批、二次确认、告警或风控策略。
- 审计日志字段和留存周期。
- 大规模导出限制和异常行为告警策略。

### 后端负责人需要处理

- 在后端 API 增加角色和权限校验。
- 不允许仅依赖前端按钮控制访问。
- 增加审计日志。
- 增加导出数量限制、频率限制或异步导出机制。
- 处理导出失败、空数据、大数据量、字段为空等边界情况。
- 确保导出文件或响应不会暴露额外敏感字段。
- 补充权限、安全、性能和异常场景测试。

### 产品负责人需要确认

- 哪些角色允许导出用户数据。
- 导出功能的具体业务场景和必要性。
- 是否必须导出姓名、手机号、邮箱、最近订单金额这些字段。
- 是否需要字段脱敏。
- 单次导出数量上限和导出频率限制。
- 是否需要审批流程、二次确认或导出前风险提示。
- 管理员导出后的数据使用和保存规范。

## 合并建议

当前 PR 不建议合并。

至少需要在合并前完成：

1. 后端 API 权限校验。
2. 敏感字段导出范围确认。
3. 审计日志。
4. 导出数量或频率限制。
5. 权限、隐私、审计和大数据量相关测试。

在安全负责人、后端负责人、产品负责人完成确认并处理上述问题前，该功能应保持阻塞状态。
