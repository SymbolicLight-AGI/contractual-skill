## 总体判断

该变更存在高优先级安全与数据访问风险。`customer_id` 来自 HTTP query 参数，但直接拼接进 SQL 查询，可能导致 SQL 注入；同时函数未体现客户备注访问权限校验、输入校验、字段最小化返回和失败路径处理。当前没有测试，无法验证安全性、正确性和回归行为。

## 主要问题

1. **Finding：SQL 查询直接拼接 HTTP 参数**
   ```python
   query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
   ```
   `customer_id` 来自 HTTP query 参数，未做参数化查询或严格校验，直接进入 SQL 字符串。

2. **Finding：缺少输入类型与格式校验**
   - 代码假设 `customer_id` 可安全作为 SQL 条件使用。
   - 未验证是否为合法客户 ID，例如整数、正数、存在的客户 ID 等。

3. **Finding：使用 `SELECT *` 返回所有备注字段**
   - 函数会返回 `notes` 表的全部字段。

## 风险与影响

1. **Risk：SQL 注入导致数据泄露或篡改**
   - 攻击者可能通过 HTTP query 参数构造恶意 `customer_id`，绕过条件查询、读取其他客户备注，甚至执行破坏性 SQL，具体影响取决于数据库驱动和执行权限。

2. **Risk：越权访问客户备注**
   - 如果上层没有严格授权控制，用户可能通过修改 `customer_id` 查询不属于自己权限范围的客户备注。
   - 客户备注可能包含个人信息、商业信息或内部沟通记录，存在隐私与合规风险。

3. **Risk：返回字段过多导致敏感信息暴露**
   - `SELECT *` 可能返回内部字段、审计字段、隐藏字段或未来新增的敏感列。
   - 表结构变化时，该函数的响应内容也会隐式变化，增加维护和合规风险。

4. **Risk：错误输入导致运行时异常或异常 SQL 行为**
   - 非数字、空字符串、特殊字符等输入可能导致 SQL 语法错误。
   - 如果异常没有在上层妥善处理，可能造成 500 错误或泄露错误信息。

5. **Risk：缺少测试导致安全回归不可见**
   - 当前没有测试，后续修改可能再次引入注入、越权或字段暴露问题而无法及时发现。

## 测试建议

1. **Test advice：SQL 注入防护测试**
   - 使用包含 SQL 控制字符或布尔条件的 `customer_id` 输入，验证不会返回非目标客户备注。
   - 验证恶意输入不会改变查询语义，也不会造成额外 SQL 执行。

2. **Test advice：输入校验测试**
   - 覆盖以下输入：
     - 合法整数客户 ID；
     - 非数字字符串；
     - 空值；
     - 负数；
     - 小数；
     - 超大数；
     - 特殊字符。
   - 验证非法输入被拒绝或返回明确错误，而不是进入 SQL 查询。

3. **Test advice：权限边界测试**
   - 模拟有权限用户查询自己可访问客户的备注。
   - 模拟无权限用户查询其他客户备注，确认被拒绝。
   - 如果权限校验在上层完成，应增加集成测试验证上层确实阻止越权访问。

4. **Test advice：字段返回测试**
   - 验证返回结果只包含业务需要的字段。
   - 如果当前设计确实需要全部字段，应由产品或数据负责人确认字段范围，并为新增敏感列建立回归测试。

5. **Test advice：空结果与异常路径测试**
   - 客户存在但没有备注时，应返回空列表或预期结果。
   - 数据库执行失败时，应验证错误处理方式符合 API 约定，不返回成功样式的错误结果，也不泄露内部 SQL 细节。

6. **Test advice：数据隔离测试**
   - 准备多个客户的备注数据，确认查询某个 `customer_id` 时只返回该客户的备注。
   - 覆盖相邻客户 ID、无效客户 ID、不存在客户 ID。
