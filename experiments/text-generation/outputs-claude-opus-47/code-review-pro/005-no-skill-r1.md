# PR Review：用户导出功能

## 风险评估

### 1. 权限风险（严重）
- **后端 API 未校验管理员角色**：前端按钮的可见性判断只是 UI 层防护，任何用户只要拿到接口地址即可直接调用，等同于越权导出全量用户数据。
- 前端校验 ≠ 安全控制，必须在服务端强制鉴权。

### 2. 隐私风险（严重）
- 导出字段包含 **姓名、手机号、邮箱、最近订单金额**，属于个人信息 + 交易信息，涉及个保法/GDPR 合规要求。
- 缺少数据脱敏选项（如手机号掩码）。
- 未说明导出文件的存储、传输加密与销毁策略。
- 未评估是否需要用户告知或合规审查。

### 3. 审计风险（高）
- 无审计日志，无法追溯"谁、何时、导出了哪些数据、导出了多少条"。
- 一旦发生数据泄露，无证据链可追查。

### 4. 滥用与性能风险（中高）
- 无导出数量上限，可能一次性导出全量用户，造成：
  - 数据库压力 / 接口超时
  - 大规模数据外泄风险被放大
- 无频率限制（rate limit），可被脚本反复调用。

### 5. 测试风险（高）
- 未见权限相关单测/集成测试（非管理员调用应 403）。
- 未见大数据量导出的性能测试。
- 未见审计日志的验证用例。
- 未见字段正确性与脱敏逻辑测试。

---

## 处理分派

**🔴 阻塞合并**，需以下负责人确认后才能继续：

### 安全负责人需确认
- 后端鉴权方案（角色校验、权限粒度）
- 审计日志规范（字段、留存周期、不可篡改性）
- 导出文件的传输/存储加密与销毁策略
- 是否需要二次认证（如管理员二次验证、审批流）

### 后端负责人需确认
- API 层强制角色校验的实现
- 导出数量上限与分页/异步导出方案
- 接口限流策略
- 补齐权限、审计、性能相关测试

### 产品负责人需确认
- 导出字段范围是否符合最小必要原则（手机号、订单金额是否必须）
- 使用场景与目标用户（是否仅限超管？是否需要审批流？）
- 合规与法务是否已评估（个人信息导出告知义务）
- 是否需要对外用户告知或在隐私政策中体现

---

## 不确定项
- 当前系统是否已有统一的审计日志组件与限流中间件，需结合代码库确认（PR 摘要未提供）。
- 适用的合规框架（个保法/GDPR/其他）取决于业务地域，需法务确认。
