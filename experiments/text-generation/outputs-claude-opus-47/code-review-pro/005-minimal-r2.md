## 风险清单（按严重度排序）

### 🔴 严重 - 权限控制缺陷
- **后端 API 未校验角色**：仅靠前端按钮隐藏判断管理员，攻击者可直接调用接口绕过，导致**任意已登录用户可导出全量用户数据**。这是典型的 Broken Access Control（OWASP A01）。
- 前端权限判断**永远不能作为安全边界**，必须在后端鉴权层（中间件 / 注解 / 守卫）强制校验角色或权限位。

### 🔴 严重 - 隐私与合规风险
- 导出字段包含 **姓名、手机号、邮箱、订单金额**，属于 PII + 财务敏感数据。
- 缺乏以下控制：
  - 是否符合 GDPR / 个人信息保护法（PIPL）下的最小必要原则？
  - 手机号/邮箱是否需要脱敏导出（如 `138****1234`）？
  - 是否需要数据主体同意或合法性基础（legitimate basis）？
  - 导出文件的存储、传输（是否 HTTPS+加密）、留存周期？

### 🟠 高 - 缺失审计日志
- 无审计日志意味着：
  - 发生数据泄露时无法溯源（谁、何时、导出了哪些数据、多少条）。
  - 不满足等保 / SOC2 / ISO27001 的可追溯性要求。
- 建议记录：操作者 ID、时间戳、IP、导出条件（过滤参数）、记录数量、文件哈希。

### 🟠 高 - 无导出数量限制
- 未限制单次/单日导出条数，存在风险：
  - **数据爬取 / 内部数据外泄**（恶意管理员或被盗号账户一次拖库）。
  - **性能与 DoS**：大批量查询可能压垮数据库或内存（一次性 load 全表）。
- 建议：分页 / 速率限制 / 单次上限 / 异步任务 + 审批流。

### 🟡 中 - 测试覆盖缺失（推断，需确认）
摘要未提及测试。重点缺口可能包括：
- 未登录用户调用接口 → 应 401
- 普通用户调用接口 → 应 403
- 管理员调用 → 200 且字段正确
- 超过数量限制 → 应拒绝
- 审计日志写入断言
- PII 脱敏逻辑单测
- 并发/大数据量场景

### 🟡 中 - 其他可能风险（不确定，需查看代码）
- SQL 注入 / 过滤参数是否参数化？
- 导出文件命名是否可枚举或路径穿越？
- CSV 注入（字段以 `=`、`+`、`-`、`@` 开头时）？
- 响应是否设置正确的 `Content-Disposition` 与 MIME？
- 是否走 CSRF 防护？
