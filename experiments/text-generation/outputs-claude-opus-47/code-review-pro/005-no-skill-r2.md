# PR Review：用户导出功能

## 风险评估

### 🔴 权限风险（严重）
- **后端 API 未校验角色**：前端管理员判断可被绕过（直接调用接口、改前端状态、Postman 重放），任何已登录用户均可调用导出接口。
- 前后端权限不一致，属于典型的**水平/垂直越权**漏洞。

### 🔴 隐私风险（严重）
- 导出字段包含 **手机号、邮箱、订单金额**，属于个人敏感信息（PII）。
- 未见脱敏、加密或最小化字段控制。
- 未明确合规依据（用户授权 / 内部数据处理协议），存在违反《个人信息保护法》、GDPR 等法规风险。
- 缺少导出文件的存储、传输、过期清理策略（不确定，未在变更中体现）。

### 🟠 审计风险（高）
- **无审计日志**：谁、何时、导出了多少条、为何导出，均无法追溯。
- 一旦发生数据泄露，无法定位责任人与影响范围。

### 🟠 滥用风险（高）
- **无导出数量限制 / 频率限制**：可一次性拉取全量用户数据。
- 无速率限制、无任务异步化，存在被批量爬取或拖库风险。

### 🟡 测试风险
- 未见权限相关的单元/集成测试（未授权访问、普通用户访问、越权场景）。
- 未见对大数据量导出的性能与超时测试。
- 未见对敏感字段输出的快照测试。

---

## 移交与确认事项

### 交给 **安全负责人** 确认
1. 后端鉴权方案（中间件统一拦截 vs 接口内校验）。
2. PII 字段是否需要脱敏（如手机号中间四位掩码）。
3. 审计日志规范：字段、留存周期、告警阈值。
4. 是否需要二次认证（如管理员重新输入密码 / MFA）后才能导出。

### 交给 **后端负责人** 处理
1. 在 API 层补齐角色校验（建议放统一权限中间件）。
2. 增加审计日志写入。
3. 增加导出数量上限、速率限制，大数据量改异步任务 + 下载链接。
4. 补充权限、越权、限流相关的自动化测试用例。

### 交给 **产品负责人** 确认
1. 导出功能的业务必要性与使用场景（谁用、用来做什么）。
2. 字段范围是否可裁剪（最小必要原则，是否真的需要订单金额）。
3. 是否需要用户侧告知 / 合规审批流程。
4. 上线节奏：建议**安全与审计未补齐前不予合并发布**。

---

## 结论
**当前 PR 不建议合入主干 / 上线**，存在后端越权与 PII 泄露的高危风险。需安全 + 后端补齐鉴权、审计、限流后再次 review，并由产品确认字段范围与合规要求。

> 不确定项：仓库中是否已有统一鉴权中间件、是否已有审计日志基础设施、导出产物的存储介质与访问控制策略，均未在变更摘要中体现，需在 review 会议中确认。
