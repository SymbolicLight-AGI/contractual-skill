# Code Review: `get_customer_notes`

## 主要问题

### 1. 安全：SQL 注入（严重）
- `customer_id` 通过 f-string 直接拼接到 SQL 中，且来源是 HTTP query 参数（外部不可信输入）。
- 攻击者可构造类似 `1 OR 1=1`、`1; DROP TABLE notes--` 等 payload，造成数据泄露、篡改或删除。
- 即便上层做了类型校验，也不应依赖；应使用参数化查询（占位符 / bind variables）。

### 2. 安全：缺少授权与租户隔离
- 函数只根据 `customer_id` 取数据，未见调用方是否校验"当前用户是否有权访问该 customer 的备注"。
- 在 CRM 场景下，这容易演变为 IDOR（越权访问其他客户的备注）。
- 备注内容可能包含 PII / 敏感信息，应纳入访问控制与审计日志。

### 3. 正确性：`SELECT *` 的脆弱性
- `SELECT *` 会随表结构变更隐式扩散字段（包括可能不应返回给前端的内部列，如软删标记、内部评分等）。
- 建议显式列出需要的字段。

### 4. 正确性：输入类型与校验缺失
- 未校验 `customer_id` 是否为整数 / UUID 等预期类型；非法输入会直接抛 DB 错误，错误信息可能泄露。
- 未处理 `customer_id` 为 `None` / 空串的情况。

### 5. 正确性：无分页与排序
- 一个客户备注可能很多，全量返回会导致内存与响应体膨胀，并阻塞 DB 连接。
- 缺少 `ORDER BY`，返回顺序不确定，前端展示不稳定。

### 6. 正确性 / 健壮性：异常处理与资源管理
- 未捕获 DB 异常；连接、游标的生命周期不可见（取决于 `db` 抽象，但函数未做防御）。
- 没有日志（请求 id、耗时、命中行数），出问题难以排查。

### 7. 数据形态：`dict(row)` 的可移植性
- 是否能 `dict(row)` 取决于驱动（如 sqlite3.Row、SQLAlchemy Row、psycopg2 默认 cursor 不支持）。
- 在未明确 `db.execute` 返回类型时，这是一个隐藏耦合点。

### 8. 性能
- 未见 `customer_id` 上是否有索引；高频接口需确认。
- 返回 `list[dict]`，若数据量大可考虑流式 / 分页。

## 测试建议

### 单元测试（使用内存 DB 或 mock）
1. **SQL 注入回归测试**：传入 `"1 OR 1=1"`、`"1; DROP TABLE notes--"`，断言：
   - 不返回其他客户的备注；
   - 不发生破坏性副作用；
   - 抛出明确的校验错误（修复后预期行为）。
2. **类型校验**：`None`、空字符串、负数、超大整数、非数字字符串 → 预期拒绝或返回空。
3. **正常路径**：插入 N 条 notes，断言返回数量、字段、顺序（按修复后定义的 `ORDER BY`）。
4. **空结果**：不存在的 `customer_id` → 返回 `[]`，而非异常。
5. **字段白名单**：断言返回 dict 的 key 集合与预期一致（防止 `SELECT *` 回潮）。

### 集成 / 安全测试
6. **越权 (IDOR)**：用户 A 请求用户 B 的 `customer_id`，断言被拒绝（403）。
7. **大数据量**：插入上万条 notes，验证分页或限流生效，响应时间在阈值内。
8. **DB 异常注入**：mock `db.execute` 抛错，断言函数返回受控错误并记录日志，不泄露堆栈到响应。

### 静态 / 流程层面
9. 接入 SAST（如 Bandit）对 f-string SQL 模式告警。
10. 在 PR 模板加入"是否使用参数化查询 / 是否做了授权校验"两项 checklist。
