# 总体判断

该函数存在**严重 SQL 注入漏洞**，因 `customer_id` 来自 HTTP query 参数并被直接字符串拼接进 SQL。必须在合入前修复，并补充测试与输入校验。其余问题（`SELECT *`、缺少错误处理、缺少授权校验）属次要但应一并处理。

# 主要问题

1. **SQL 注入（严重 / Finding）**
   - 证据：`query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"`，`customer_id` 来源于 HTTP query 参数，未做参数化或类型校验。
   - 影响：攻击者可通过构造 `customer_id`（如 `1 OR 1=1`、`1; DROP TABLE notes--`，或基于联合查询/盲注的 payload）读取或破坏数据库内容。
   - 高层建议：使用参数化查询（占位符绑定），不要做字符串拼接；同时在入口层将 `customer_id` 强制转换为整数或 UUID 并校验。

2. **缺少授权 / 访问控制（高 / Risk，需上下文确认）**
   - Finding：函数对任何传入的 `customer_id` 都直接返回 notes，未见调用方是否检查请求者是否有权查看该客户。
   - Risk：在 CRM 中可能导致 IDOR（越权访问其他客户备注，潜在涉及个人数据）。
   - 建议：在 API 层强制基于当前会话的权限校验，或在查询条件中加入 `tenant_id`/`owner_id` 约束。

3. **`SELECT *` 与返回结构不稳定（中 / Finding）**
   - Risk：列增减会改变接口契约；可能无意暴露敏感字段（例如内部状态、PII 字段）。
   - 建议：显式列出字段，并在 API 层做字段白名单过滤。

4. **缺少错误处理与空结果语义（中 / Finding）**
   - Risk：DB 异常会直接冒泡，可能暴露堆栈或内部 SQL 信息；调用方无法区分“客户不存在”和“客户存在但无 notes”。
   - 建议：捕获预期 DB 异常并转换为受控错误；明确空列表语义。

5. **可维护性 / 日志（低-中 / Risk）**
   - 若有日志记录原始 query，可能把注入 payload 或客户标识写入日志，引入二次风险。建议日志中避免拼接 SQL，仅记录绑定参数与脱敏后的 ID。

6. **性能（低 / Uncertainty）**
   - 未见 `notes.customer_id` 是否有索引，也未做分页；大客户的 notes 量可能很大。建议加分页/`LIMIT` 与索引校验。

# 风险与影响

- 数据机密性：SQL 注入可导致整库读取，含潜在 PII。
- 数据完整性：注入可执行写/删操作，破坏 notes 数据。
- 合规：客户备注可能含个人数据，未授权访问触发隐私合规风险。
- 可用性：恶意 payload 可能拖垮 DB（重型联合查询）。

# 测试建议

- **安全测试（必需）**
  - 注入用例：`customer_id` = `"1 OR 1=1"`、`"1; DROP TABLE notes--"`、`"1 UNION SELECT ..."`、引号/分号/注释符；断言：要么 400 拒绝，要么参数化后作为字面值匹配且无附加行返回。
  - 类型校验：传入非数字（字母、空、超长字符串、负数、浮点、Unicode）应返回 400 而非 500。
- **授权测试**
  - 越权：用户 A 请求用户 B 的 `customer_id`，应 403/404。
  - 跨租户：多租户场景下跨 tenant 的 ID 应不可见。
- **功能测试**
  - 存在客户且有 notes：返回正确字段集合与顺序。
  - 存在客户但无 notes：返回 `[]`。
  - 不存在客户：明确语义（404 或空列表，按产品决定）。
- **回归 / 契约测试**
  - 显式断言返回字段集合，避免 `SELECT *` 漂移。
- **失败路径**
  - DB 连接失败、超时：返回受控错误，不泄露 SQL/堆栈。
- **性能**
  - 大量 notes（如 10k+）下的响应时间与是否使用索引（EXPLAIN）。
