# Code Review: `get_customer_notes`

## 假设
- 未提供 `db` 对象的具体类型，假设为标准 DB-API 兼容连接（如 sqlite3/psycopg2 等）。
- 假设 HTTP 层未对 `customer_id` 做强类型校验/转换。
- 未见调用方代码与表结构，相关推断会标注不确定。

## 严重问题（Severity 排序）

### 🔴 Critical — SQL 注入漏洞
```python
query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"
```
- `customer_id` 来自 HTTP query 参数，**直接用 f-string 拼接进 SQL**，是典型的 SQL Injection。
- 攻击者可通过 `?customer_id=1 OR 1=1` 拖库，或通过 `; DROP TABLE ...`、UNION 注入等手段读取/破坏数据。
- 即便"内部 CRM"也不可豁免：内部系统常因弱认证/越权/SSRF 成为横向移动目标。
- 修复方向（不在此直接改代码）：使用参数化查询（`?` / `%s` / 命名参数），由驱动负责转义；同时在入口层将 `customer_id` 强制转换为整数并校验范围。

### 🟠 High — 缺乏输入校验与类型强制
- 函数未校验 `customer_id` 是否为整数、是否为正、是否非空。
- 若 `customer_id` 为 `None` 或字符串，SQL 会构造出语法错误或语义错误（如 `... = None`）。
- 即使改为参数化查询，仍应在边界进行类型/范围校验，防止异常上抛到用户层泄露堆栈。

### 🟠 High — 越权访问风险（IDOR）
- 函数仅按 `customer_id` 查询，未校验当前调用者是否有权查看该客户的备注。
- HTTP 参数完全可控的 ID 查询是 IDOR 的高发场景。
- 不确定：未见认证/授权中间件代码，无法断言是否在上层已处理；**建议明确确认**。

### 🟡 Medium — `SELECT *` 的可维护性与隐私风险
- `SELECT *` 会返回所有列，可能包含未来新增的敏感字段（如内部标记、PII），并会通过 API 泄露。
- 列顺序/数量变化也会让上层 `dict(row)` 的契约隐式改变，破坏向后兼容。
- 建议：显式列出需要的列。

### 🟡 Medium — 缺少分页/数量限制
- 未做 `LIMIT`/分页：单个客户备注若数量很大，会造成内存峰值和响应延迟，构成潜在 DoS。
- 建议增加 `LIMIT` 与可选的 `OFFSET`/游标，以及稳定的 `ORDER BY`。

### 🟡 Medium — 错误处理与资源管理缺失
- `db.execute(...).fetchall()` 未做异常捕获；DB 异常会直接冒泡。
- 若 `db.execute` 返回的是游标/结果集，未显式关闭可能在某些驱动下泄漏资源。
- 不确定：依赖 `db` 的实际接口，需结合实际驱动确认。

### 🟢 Low — `dict(row)` 的可移植性
- `dict(row)` 仅在 row 对象支持映射协议（如 `sqlite3.Row` 配置了 `row_factory`，或 SQLAlchemy `Row`）时有效。
- 对原生 tuple 行会抛出 `ValueError`。建议显式使用列名映射或 ORM/`RowMapping`。

### 🟢 Low — 缺少日志与可观测性
- 没有结构化日志（查询耗时、行数、客户 ID 脱敏）。一旦发生注入或越权，溯源困难。

## 正确性风险小结
- 注入导致结果可被操纵。
- 类型不当导致查询语义偏差或异常。
- 行对象到 `dict` 的转换依赖驱动配置，存在隐式契约。

## 测试建议（当前无测试，建议补齐）

### 单元测试（使用内存 DB 或 mock 游标）
1. **正常路径**：给定整数 `customer_id`，返回对应行列表；空结果返回 `[]`。
2. **类型边界**：
   - `customer_id` 为字符串数字、负数、0、超大整数。
   - `customer_id` 为 `None`、空串、浮点 → 期望明确拒绝或抛出受控异常。
3. **SQL 注入回归测试**：
   - 传入 `"1 OR 1=1"`、`"1; DROP TABLE notes--"`、`"1 UNION SELECT ..."`，断言：
     - 不会返回他人备注；
     - 不会执行第二条语句；
     - 在参数化方案下应抛出类型/绑定错误或安全地返回空。
4. **行映射**：mock 一个不支持 `dict(row)` 的 row 对象，验证错误被妥善处理或正确转换。
5. **大结果集**：模拟数万行，验证存在 `LIMIT` 或流式处理，不会无限增长内存。
6. **数据库异常**：`db.execute` 抛出 `OperationalError` 时，函数行为是否符合契约（重抛/记录/降级）。

### 集成/安全测试
7. **授权测试**：调用方为不同租户/用户时，访问他人 `customer_id`，应被拒绝（需在上层鉴权点测试）。
8. **Fuzz 测试**：对 `customer_id` 进行模糊输入（含 Unicode、控制字符、SQL 元字符），监测异常分类与日志。
9. **静态检查 / SAST**：把该函数加入 Bandit/Semgrep 规则集（拼接 SQL 检测），CI 中阻断同类写法。

### 性能/回归
10. **基准测试**：单次/批量查询的 P95 延迟与内存占用，作为后续优化基线。
