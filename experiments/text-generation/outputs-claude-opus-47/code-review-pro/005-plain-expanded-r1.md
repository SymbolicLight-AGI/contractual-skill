# 代码评审：用户导出功能 PR

## 总体判断
**不可合并 (Block)**。该 PR 存在严重的权限绕过和隐私泄露风险，必须修复后才能进入下一轮评审。

---

## 主要发现（按严重度排序）

### 1. 【严重 / Critical】后端缺少角色校验 —— 权限绕过
- **现象**：仅前端按钮做管理员判断，后端 API 未校验调用者角色。
- **影响**：任何已登录用户（甚至未授权用户，取决于鉴权中间件）可直接构造 HTTP 请求调用导出接口，绕过前端限制，批量拉取全量用户 PII。属于典型的 **Broken Access Control (OWASP A01)**。
- **要求**：后端必须在接口入口处显式校验管理员角色（基于服务端会话/JWT claim），并对失败请求返回 403，而不是依赖前端。

### 2. 【严重 / Critical】PII 批量泄露风险
- **现象**：导出字段包含姓名、手机号、邮箱、最近订单金额，且无数量上限。
- **影响**：
  - 一次请求可能拖走全量用户数据，构成数据外泄事件。
  - 手机号、邮箱属于个人敏感信息，多数司法管辖区（GDPR / 国内《个人信息保护法》）要求最小化、合法性基础、留痕。
  - 订单金额属于商业敏感信息。
- **要求**：
  - 强制服务端分页 / 单次条数上限。
  - 评估是否需要字段级脱敏（如手机号中间四位掩码）。
  - 评估是否需要数据导出审批流。

### 3. 【严重 / High】缺少审计日志
- **现象**：导出操作未记录审计日志。
- **影响**：发生数据泄露时无法溯源（谁、何时、导出多少、过滤条件、IP）。不符合常见合规要求与内部安全基线。
- **要求**：写入不可篡改的审计日志，至少包含：操作者 ID、时间、请求参数、返回行数、客户端 IP/UA。

### 4. 【中 / Medium】缺少速率限制与异常告警
- **现象**：无频次/总量限流，没有异常导出告警。
- **影响**：即便加了角色校验，管理员账号被盗后仍可被用于批量爬取，且无告警。
- **要求**：加 rate-limit + 大批量导出触发安全告警/二次确认。

### 5. 【中 / Medium】缺少测试
- **现象**：PR 未提及任何测试。
- **建议测试用例**：
  1. 非管理员直接调用 API → 期望 403。
  2. 未登录调用 API → 期望 401。
  3. 超过单次导出上限 → 期望分页或拒绝。
  4. 成功导出 → 审计日志被写入且字段完整。
  5. 前端隐藏按钮但绕过 UI 直接调用 API → 期望被后端拦截（回归测试）。
  6. 字段脱敏策略（若引入）单元测试。

## 不确定性 / 信息不足
以下事项变更摘要未说明，需要确认后再评估：
- 当前接口的鉴权中间件是否至少要求登录态？（影响风险等级是否升至"任何匿名用户可下载"）
- 导出格式（CSV/Excel/JSON）以及是否生成临时下载链接？临时链接是否带签名与过期时间？
- 数据是否包含已注销 / 已删除用户？（涉及"被遗忘权"合规）
- 是否已有公司级数据导出审批流程未被遵循？
- 法务/合规是否已对该字段集做过评审？

未在变更内确认前，不应假设上述任何一项已被妥善处理。

## 结论
在后端角色校验、审计日志、导出上限、对应测试四项至少同时落地之前，**不建议合并**。安全与产品负责人需共同确认字段范围与合规基础。
