# Code Review: `get_customer_notes`

## 总体判断
**不可合并 (Block)**。函数存在严重的 SQL 注入漏洞，且来源于外部 HTTP 输入，风险等级为 Critical。此外还有正确性、可维护性与测试覆盖的问题。

---

## 主要发现

### 1. 【Critical / 安全】SQL 注入漏洞
- `customer_id` 通过 f-string 直接拼接进 SQL 字符串，且根据背景该值来自 HTTP query 参数，属于不受信输入。
- 攻击者可构造如 `1 OR 1=1`、`1; DROP TABLE notes--`、UNION 注入等 payload，导致：
  - 越权读取其他客户备注（横向数据泄露，CRM 数据属隐私敏感）。
  - 数据破坏或拖库。
  - 基于报错/盲注的进一步信息泄露。
- **影响**：客户隐私泄露、数据完整性丧失、合规风险（视所在司法管辖区可能触发数据保护法）。
- **修复方向**（仅建议，不直接改代码）：使用参数化查询（占位符 `?` 或 `%s`，取决于驱动），不要依赖手动转义；并在上游对 `customer_id` 做类型校验（应为整数/UUID）。

### 2. 【High / 安全 & 性能】`SELECT *`
- 返回所有列可能包含内部字段（如软删除标记、内部备注作者、隐私字段），随接口直接暴露给 HTTP 调用方，存在过度暴露风险。
- schema 变化时返回结构隐式改变，难以维护，且容易破坏下游。
- 建议显式列出所需字段。

### 3. 【Medium / 正确性】缺少授权 / 越权检查
- 函数没有校验当前调用者是否有权访问该 `customer_id` 的备注。即便修复了 SQL 注入，攻击者仍可通过遍历 `customer_id` 拉取所有客户的备注（IDOR）。
- 背景说是“内部 CRM”，但仍需基于调用者身份的访问控制；不应假设“内部=可信”。
- **Handoff**：建议由后端/权限模块负责人确认授权层位置（中间件、服务层还是该函数内）。

### 4. 【Medium / 正确性】输入校验缺失
- 未校验 `customer_id` 是否为期望类型（int/UUID）、是否非空。
- 当 `customer_id` 为 `None` 或非法字符串时行为未定义，可能抛出底层数据库异常并向上泄露堆栈/SQL 片段。

### 5. 【Low / 可维护性】无分页 / 无排序
- `notes` 表若数据量大，一次性 `fetchall()` 可能导致内存与延迟问题。
- 没有 `ORDER BY` 导致返回顺序不稳定，前端展示与测试断言都会受影响。

### 6. 【Low / 可观测性】缺少错误处理与日志
- DB 异常会原样冒泡，需确认上层是否有统一处理；同时缺少结构化日志（不要记录敏感字段值，但应记录 customer_id 与调用者身份用于审计）。

---

## 风险影响汇总
- **机密性**：高（SQL 注入 + IDOR + SELECT *）。
- **完整性**：高（注入可写）。
- **可用性**：中（无分页可能引发大查询）。
- **合规/隐私**：中–高（CRM 客户数据）。

---

## 测试建议

修复后建议至少覆盖：

1. **安全 / 注入回归测试**
   - 传入 `"1 OR 1=1"`、`"1; DROP TABLE notes--"`、`"1 UNION SELECT ..."`：应被拒绝（参数化下应抛类型错误或返回空）。
   - 传入非整数字符串、超长字符串、空字符串、`None`：应返回明确的 4xx 或受控异常，而非 500/SQL 报错。

2. **授权测试**
   - 调用者 A 查询自己拥有权限的 `customer_id`：成功。
   - 调用者 A 查询不属于其权限范围的 `customer_id`：拒绝（403/空，依设计而定）。

3. **正确性测试**
   - 存在 N 条备注时返回 N 条，字段集合等于显式定义的白名单。
   - 客户存在但无备注：返回 `[]`。
   - 客户不存在：返回 `[]` 或 404（取决于产品定义，需明确）。

4. **边界 / 性能测试**
   - 大量备注（如 10k+）：验证分页/限制行为。
   - 排序稳定性：相同输入产生相同顺序。

5. **集成测试**
   - 从 HTTP 入口到 DB 的端到端测试，确保参数化在整个调用链都生效（不仅是单元层）。

## 不确定性 / 信息缺口
- 使用的 DB 驱动未知（sqlite3 / psycopg2 / SQLAlchemy Core？），影响参数化占位符语法与 `dict(row)` 是否可行。
- `db.execute` 是否在事务/连接池上下文中、是否自动提交未知。
- 是否已有上游中间件做认证/授权未知。
- `customer_id` 的规范类型（整型自增 vs UUID）未知。
- 产品对“客户不存在”与“无备注”的语义是否区分未知。

---

## 需要的 Handoff
- **安全负责人**：确认 SQL 注入与 IDOR 修复方案及回归测试纳入安全基线。
- **后端负责人**：决定授权检查位置（中间件 vs 服务层）、输入校验层、分页约定。
- **产品负责人**：定义“客户不存在 vs 无备注”的返回语义，以及哪些 note 字段可暴露给前端。
- **QA / 测试负责人**：补齐当前缺失的单元 + 集成 + 安全回归测试。
