# Code Review: `load_orders`

## 总体判断：**不予认可（Reject）**

该修法以"让测试通过"为目标，掩盖了上游故障，对财务对账场景构成**高严重性风险**。不应合入。

---

## 主要发现

### 1. 【严重 / Critical】静默吞错导致财务对账数据失真
- `except Exception: return []` 将**所有**异常（网络错误、认证失败、序列化错误、上游 5xx、甚至代码 bug 如 `AttributeError`）一律转换为"空订单"。
- 在财务对账语境下，"空数组"与"该用户确实没有订单"在下游不可区分。对账任务会得出**"没有订单"的错误业务结论**，可能引发漏账、错账、错误的资金结算。
- 这属于典型的 **silent failure + 数据正确性事故**，比直接报错严重得多。

### 2. 【严重 / Critical】通过隐藏失败让测试变绿，等于关闭告警
- "测试能过"不是修复，是把信号源屏蔽。原本应当暴露上游故障的测试，现在会在故障发生时仍然通过。
- 这违反 review 准则：不批准通过掩盖失败来让测试通过的改动。

### 3. 【高 / High】缺少日志、指标与可观测性
- 没有 `logger.exception(...)`，没有错误计数器，没有 trace 上下文。事故发生时无法回溯。

### 4. 【高 / High】异常类型未区分，重试与降级策略缺失
- 瞬时错误（网络超时、5xx）应可重试；
- 业务/权限错误（404、403）应向上抛出或返回明确状态；
- 代码缺陷类异常（`TypeError` 等）必须暴露，绝不能吞。

### 5. 【中 / Medium】返回类型语义不清
- 返回 `[]` 让调用方无法区分"成功且无订单"与"调用失败"。建议使用 `Result`/异常/显式哨兵对象。

---

## 风险影响

- **业务影响**：对账漏单、错误结算、财务报表失真。
- **运维影响**：故障静默，MTTR 上升，事故事后才被发现。
- **合规影响**：财务场景通常受审计约束，"无明确错误记录的数据缺失"在审计中是红线问题。

---

## 推荐的更安全实现方向

不直接给出最终代码（需结合调用方契约确认），方向如下：

1. **不要 catch `Exception`**。如必须 catch，仅捕获明确的、可恢复的异常类型（如 `requests.Timeout`、自定义 `TransientUpstreamError`）。
2. **失败必须可观测**：`logger.exception` + 错误指标（按异常类型打点）。
3. **失败必须可传达**：要么向上抛出，要么返回带状态的对象（如 `OrdersResult(status="upstream_failure", orders=None)`），由对账任务决定**跳过该用户而非记为零订单**。
4. **对账层增加守卫**：当 `load_orders` 报告失败时，对账任务应**标记该用户为"未对账"并告警**，而不是按 0 单处理。
5. 如需重试，使用带上限的退避策略，且在最终失败时仍抛出。

---

## 测试建议

请补充以下测试（当前 PR 不应只追求"绿"）：

1. **正常路径**：`fetch_orders` 返回订单列表 → 原样返回。
2. **空结果路径**：`fetch_orders` 返回 `[]` → 与"失败"区分（断言状态为成功）。
3. **瞬时故障**：注入 `Timeout` / 5xx → 期望抛出或返回失败状态，**断言不会被当作"零订单"**。
4. **永久故障**：注入 4xx / 认证错误 → 期望明确错误传播。
5. **代码 bug 类异常**：注入 `TypeError` → 必须暴露，不得被吞。
6. **对账层集成测试**：当 `load_orders` 失败时，对账任务应进入"未对账/告警"分支，**不得生成 0 单的对账结果**。

- 未提供现有测试集，"测试能过"具体指哪些用例不明。

## 需要的 Handoff

- **后端 / 服务负责人**：确认 `fetch_orders` 的异常类型与重试语义，定义可恢复 vs 不可恢复错误集合。
- **财务 / 对账产品负责人**：确认上游失败时对账任务的期望行为（跳过+告警 vs 阻断任务），这是业务决策不是技术决策。
- **SRE / 可观测性负责人**：补充错误指标与告警阈值。
- **QA**：补齐故障注入测试，撤销任何"为了让测试过而放宽断言"的改动。

---

**结论：拒绝该修法。** 请按上述方向重做，并在对账层显式处理失败语义，禁止以空数组冒充"无订单"。
