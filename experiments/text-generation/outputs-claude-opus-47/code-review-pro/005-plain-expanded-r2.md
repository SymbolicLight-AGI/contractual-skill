# Code Review: 用户数据导出功能

## 总体判断

**不可合并 / Block merge.** 该 PR 存在严重的权限绕过和隐私泄露风险，属于高危变更，必须修复后才能进入下一轮 review。

---

## 主要发现 (按严重度排序)

### 🔴 Critical — 后端未做角色校验 (越权 / Broken Access Control)
- 前端按钮的管理员判断**不构成安全边界**。任何用户（含匿名或低权限账户）只要直接调用导出 API，即可拉取全量用户姓名、手机号、邮箱、订单金额。
- 这属于 OWASP A01:2021 Broken Access Control，等同于一个公开的 PII 接口。
- **影响**：全量用户 PII + 交易金额泄露，可能触发监管事件（GDPR / 《个人信息保护法》/ 数据安全法），需上报安全与合规。

### 🔴 Critical — 隐私 / PII 批量导出无控制
- 导出字段包含手机号、邮箱、订单金额，属于个人敏感信息。
- 无导出数量上限，单次可拉取全表，构成"批量爬取"风险。
- 没有字段最小化（是否真的需要完整手机号？可否脱敏 / 仅末四位？）。
- 没有用途声明 / 二次确认 / 审批流。

### 🟠 High — 缺少审计日志
- 无"谁、在什么时间、导出了什么范围、多少条记录"的审计记录。
- 一旦发生泄露事件，**无法溯源、无法举证、无法满足合规审计要求**。
- 审计日志本身也应避免明文记录被导出的 PII 内容（只记录范围/计数/操作者）。

### 🟠 High — 缺少速率限制 / 配额
- 无每用户、每 IP、每时间窗的导出频次或行数上限。
- 即使加上鉴权，被泄露的管理员凭证也可一次性脱库。

### 🟡 Medium — 测试缺口
该 PR 显然缺少以下测试，必须补齐：
1. **授权测试**：匿名用户、普通用户调用导出 API → 期望 401/403。
2. **角色测试**：仅 admin（及/或更细粒度的"数据导出"权限）可成功。
3. **审计测试**：成功与失败的导出均写入审计日志，且日志不含完整 PII。
4. **配额测试**：超过单次行数上限 / 频次上限 → 拒绝。
5. **字段测试**：返回字段与脱敏策略符合约定（防止字段悄悄增加）。
6. **回归测试**：前端隐藏按钮的同时，API 直连仍被拒。

## 风险影响汇总

| 维度 | 风险 |
|---|---|
| 安全 | 任意用户可越权导出全量用户 PII |
| 隐私/合规 | 批量 PII 出境/泄露风险，可能违反 PIPL/GDPR |
| 可运营性 | 无审计 → 事件无法溯源 |
| 可滥用性 | 无配额 → 凭证泄露即等于脱库 |
| 测试 | 关键授权路径无测试覆盖 |

## 必需的修复方向 (建议，不直接改码)

1. 后端 API 强制做角色 / 权限校验，前端判断仅作为 UX。
2. 引入审计日志：操作者、时间、筛选条件、返回行数、请求 ID；日志不含明文 PII。
3. 单次导出行数上限 + 用户级速率限制；大批量走异步任务 + 审批。
4. 字段最小化与脱敏（手机号/邮箱按需脱敏）。
5. 补齐上文列出的 6 类测试。
