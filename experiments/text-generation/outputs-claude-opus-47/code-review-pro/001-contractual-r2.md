# 代码审查报告 — `get_customer_notes`

## 总体判断

该函数存在**严重安全缺陷**（SQL 注入），且缺少输入验证、错误处理与测试覆盖。在 CRM 场景下，`customer_id` 来自 HTTP query 参数属于不可信输入，当前实现可被远程利用以泄露或破坏数据库数据。**建议阻止合入，需安全团队介入**。

---

## 主要问题

### 1. SQL 注入（严重 / Critical）
- **Finding**：`query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"` 通过 f-string 将外部输入直接拼接进 SQL 语句。
- 由于 `customer_id` 来自 HTTP query 参数，攻击者可构造 `1 OR 1=1`、`1; DROP TABLE notes--`、UNION 注入等 payload。
- 即使数据库驱动不支持多语句执行，UNION/布尔/时间盲注仍可用于数据泄露。

### 2. 缺少输入验证与类型约束
- **Finding**：未对 `customer_id` 做类型校验（应为整数或受控格式）、未做空值/越权检查。
- 进入函数的任何字符串都会被直接拼入 SQL。

### 3. 缺少授权/访问控制证据
- **Uncertainty**：从片段无法确认调用方是否做了「当前登录用户是否有权读取该 `customer_id` 的备注」的鉴权。
- **Risk**：若上游也未校验，将形成 IDOR（横向越权），任何已登录用户可遍历其他客户的备注。

### 4. `SELECT *` 过度返回
- **Finding**：返回所有列。
- **Risk**：`notes` 表若包含内部字段（作者、内部标记、敏感标签、PII）会被原样返回到 HTTP 响应，造成数据过度暴露。同时使列结构变更后接口行为隐式漂移。

### 5. 错误处理与资源管理缺失
- **Finding**：无 try/except，无对 `db.execute` 失败、连接超时、`fetchall` 大结果集的处理。
- **Risk**：异常会冒泡到 HTTP 层，可能泄露堆栈/SQL 片段；大客户备注无分页会导致内存与延迟问题。

### 6. 隐式信任 `dict(row)` 的可序列化性
- **Uncertainty**：依赖驱动行对象支持 `dict(row)`；某些驱动需要显式 mapping。
- **Risk**：DateTime/Decimal/bytes 等字段可能导致下游 JSON 序列化失败。

### 7. 日志与可观测性
- **Uncertainty**：未见日志。需确认调用链是否会把拼接后的 SQL 写入日志——若是，将把注入 payload 与潜在 PII 写入日志系统。

---

## 风险与影响

- **数据泄露**：通过 SQL 注入读取 `notes` 之外的任意表（用户、凭据、订单）。
- **数据完整性**：依驱动配置可能存在写入/删除风险。
- **合规风险**：CRM 备注通常含 PII；越权或注入泄露可能触发数据保护合规事件。
- **可用性**：恶意构造可触发慢查询/全表扫描，造成 DoS。
- **接口契约漂移**：`SELECT *` 使列变更隐式影响外部消费者。

---

## 测试建议

参数化与安全测试（Test advice）：

1. **SQL 注入回归测试**：
   - 输入 `1 OR 1=1`、`1; SELECT 1`、`1 UNION SELECT ...`、`'; DROP TABLE notes;--` 等 payload，断言：要么抛出验证错误，要么仅命中 `customer_id = <原值>`，绝不返回额外行。
2. **类型与边界**：负数、0、超大整数、浮点、空串、None、非 ASCII、超长字符串 → 应返回 4xx 或参数错误。
3. **正向用例**：存在备注 / 无备注 / 多条备注的客户，断言字段集合与排序。
4. **授权测试**：用户 A 请求用户 B 的 `customer_id`，断言 403/404；未登录请求 401。
5. **字段白名单测试**：响应仅包含约定字段，新增列不会自动外泄。
6. **错误路径**：DB 不可用、查询超时、连接断开 → 返回受控错误，不泄露 SQL 或堆栈。
7. **大结果集**：构造 N 条备注，验证分页/上限行为。
8. **序列化**：含 datetime/decimal/null 的行能正确转 JSON。
9. **日志断言**：日志中不出现原始 SQL 与 PII 明文。

## 不确定性

- 上层是否已做鉴权（IDOR 风险无法仅凭片段确认）。
- `db.execute` 使用的驱动/ORM 是否支持参数化占位符语法（`?` vs `%s` vs `:id`）——修复方式取决于此，但「必须参数化」的结论不变。
- `notes` 表实际列结构与敏感字段范围未知。
- HTTP 层是否已将 `customer_id` 转成 int；若已转，注入面收窄但**不可作为唯一防线**。
- 是否有审计日志/速率限制等纵深防御。

## 交接建议

- **Security**：SQL 注入与潜在 IDOR 必须由安全评审签字；需评估是否已在生产环境暴露及是否需要事件响应。
- **Backend owner**：改用参数化查询、列白名单、显式类型校验、错误处理与分页。
- **Product owner**：确认接口应返回的字段集合与访问范围（谁可以读谁的备注）。
- **QA/test owner**：补齐上述安全/授权/失败路径回归测试，并加入注入 payload 字典作为常态化用例。
