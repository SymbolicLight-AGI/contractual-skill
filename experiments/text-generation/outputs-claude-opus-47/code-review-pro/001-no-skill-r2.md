# Code Review: `get_customer_notes`

## 主要问题

### 1. 安全：SQL 注入（严重，阻断合并）
- `customer_id` 来自 HTTP query 参数，属于不受信任的输入，却被通过 f-string 直接拼入 SQL。
- 攻击者可构造如 `1 OR 1=1`、`1; DROP TABLE notes--` 等载荷，导致数据泄露、篡改或破坏。
- 即使前端做了校验，后端也必须使用参数化查询（占位符 + 参数绑定）或经过严格类型转换/白名单校验。
- 这是该函数最关键的问题，其他改动都应在修复此项之后再讨论。

### 2. 正确性与健壮性
- **缺少类型校验**：未将 `customer_id` 转换为整数或校验范围/格式；非数字输入会导致 SQL 语法错误或异常向上抛出。
- **缺少授权检查**：函数本身未确认调用者是否有权查看该 `customer_id` 的备注。即便用于"内部 CRM"，仍建议在调用层或本函数内做权限校验，避免 IDOR（越权访问）。
- **`SELECT *` 的脆弱性**：表结构变化（新增敏感列，如内部评级、PII）会被无差别返回，可能导致信息泄露并使 API 响应不稳定。建议显式列出字段。
- **未处理空结果与异常**：未区分"客户不存在"与"客户存在但无备注"，调用方难以做正确的 HTTP 状态码映射（404 vs 200 []）。
- **未分页/未排序**：备注可能很多，一次性 `fetchall()` 存在内存和延迟风险；同时缺少 `ORDER BY` 会导致顺序不确定。
- **资源管理**：依赖 `db.execute(...)` 的实现，未见游标关闭或事务边界；若底层不是上下文管理，长期运行可能泄漏连接。
- **`dict(row)` 的可移植性**：是否可行取决于驱动（如 sqlite3 需要设置 `row_factory`，DB-API 默认 row 不一定可直接 `dict()`）。不确定当前 `db` 的实现细节。

### 3. 可观测性与日志
- 缺少结构化日志（请求方、customer_id、行数、耗时）。
- 若记录日志，需注意不要把备注内容（可能含 PII）直接落盘。

### 4. 接口契约
- 返回类型未注解，文档缺失：调用方无法明确字段集合和可能的异常。
- 没有定义错误语义（参数非法、客户不存在、DB 故障的区分）。

## 测试建议

### 单元测试（使用 mock 或内存数据库）
1. **SQL 注入回归测试**：传入 `"1 OR 1=1"`、`"1; DROP TABLE notes--"`、`"' OR ''='"` 等，断言：
   - 不会返回非目标客户的数据；
   - 不会执行多语句；
   - 抛出可预期的校验异常或返回空结果。
2. **类型校验测试**：传入字符串、负数、`None`、超大整数、浮点数，验证错误处理路径。
3. **正常路径**：已存在的 `customer_id`，返回行数、字段、顺序符合预期。
4. **空结果**：客户存在但无备注，返回 `[]`。
5. **不存在的客户**：返回 `[]` 或抛出特定异常（取决于约定），需在测试中固化契约。
6. **字段稳定性**：断言返回字典的 key 集合，防止 `SELECT *` 因表结构变更悄悄改变响应。
7. **大结果集**：N 条（例如 10k）备注下的耗时与内存上限（可作为性能基线）。

### 集成测试
8. 使用真实数据库（或容器化测试库）跑端到端：连接、参数绑定、字符集（UTF-8 备注、emoji）正确。
9. 并发调用：验证连接池/事务无泄漏。

### 安全测试
10. 在 CI 中引入静态分析（如 Bandit、Semgrep 规则 `python.lang.security.audit.formatted-sql-query`）以阻止此类 f-string SQL。
11. DAST/Fuzz：对暴露该函数的 HTTP 端点跑 SQL 注入扫描（sqlmap 或自研 fuzz）。

### 授权测试
12. 模拟低权限用户请求他人 `customer_id`，断言被拒绝（若授权应在本层处理）。
