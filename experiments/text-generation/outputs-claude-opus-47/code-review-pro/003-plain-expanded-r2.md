# Code Review: `load_orders` 异常处理修改

## 总体判断

**不认可此修法。** 该改动以掩盖失败的方式让测试通过，对财务对账场景构成高风险，违反"不得通过隐藏错误使测试通过"的审查原则。**Request changes.**

---

## 主要发现

### 1. 【严重 / Critical】静默吞错导致财务对账数据错误
- **位置**：`load_orders` 中 `except Exception: return []`
- **问题**：上游 `client.fetch_orders` 出现任何异常（网络超时、认证失败、5xx、序列化错误、甚至代码 bug 引发的 `AttributeError`）都会被吞掉并返回 `[]`。
- **业务影响**：在财务对账上下文中，"接口失败"与"该用户确实没有订单"被混为一谈。对账任务会显示"没有订单"，可能导致：
  - 漏记收入 / 漏对账
  - 错误的财务报表与合规风险
  - 下游自动化（退款、结算、风控）基于错误的空集做决策
- **数据丢失风险**：是。这是典型的 silent data loss。

### 2. 【严重 / Critical】用"返回成功样态"来通过测试
- 通过把异常转成空数组让测试变绿，等同于把失败信号从系统中抹除。这是测试反模式：测试不再验证真实行为，而是验证"任何情况下都不抛错"。
- 这违反了 review 规则中明确禁止的"approve fixes that make tests pass by masking failures"。

### 3. 【中 / Major】过宽的 `except Exception`
- 即便保留兜底，也不应捕获所有 `Exception`。会掩盖编程错误（`TypeError`、`KeyError`）、配置错误、鉴权异常等需要立即暴露的问题。

### 4. 【中 / Major】缺少可观测性
- 无日志、无指标、无追踪。即便上游持续失败，运维与财务都无从感知。

### 5. 【低-中 / Minor】缺少类型与契约说明
- 返回值类型未注明，调用方无法区分"空"与"错"。建议用显式结果类型（如 `Result`/`Either`），或抛出领域异常。

---

## 风险影响汇总

| 维度 | 影响 |
|---|---|
| 正确性 | 高 — 业务结果错误且不可察觉 |
| 安全 / 合规 | 中-高 — 财务对账错误可能涉及审计与合规 |
| 可观测性 | 高 — 失败完全不可见 |
| 可维护性 | 中 — 宽泛 catch 会掩盖未来 bug |

---

## 推荐的更安全做法

不要兜底成 `[]`。让失败可见、可分类、可重试：

```python
class OrderFetchError(Exception):
    """领域层错误，调用方必须显式处理。"""

def load_orders(client, user_id):
    try:
        return client.fetch_orders(user_id)
    except (TimeoutError, ConnectionError) as e:
        logger.warning("fetch_orders transient failure", extra={"user_id": user_id, "err": str(e)})
        raise OrderFetchError("transient") from e
    except AuthError as e:  # 视客户端 SDK 实际异常类型而定
        logger.error("fetch_orders auth failure", extra={"user_id": user_id})
        raise OrderFetchError("auth") from e
```

调用方（对账任务）必须：
- 区分"空订单"与"获取失败"；
- 失败时**不要**将该用户标记为"无订单"，而是标记为"待重试 / 待人工核查"；
- 配合重试（指数退避）+ 死信队列 / 告警。

---

## 测试建议

应至少覆盖以下用例（当前修法会让这些都"通过"但语义错误）：

1. **正常路径**：客户端返回订单列表 → 原样返回。
2. **真实空集**：客户端返回 `[]` → 返回 `[]`，对账任务标记"无订单"。
3. **瞬时失败**（TimeoutError / 5xx）→ 抛 `OrderFetchError`，对账任务**不得**把该用户当作"无订单"。
4. **鉴权失败**（401/403）→ 抛 `OrderFetchError("auth")`，触发告警。
5. **协议/反序列化错误** → 抛错，记录原始响应摘要（不含 PII / 不含完整凭据）。
6. **对账层集成测试**：模拟上游失败，断言对账报告中该用户被标为"获取失败"而非"零订单"。
7. **回归测试**：禁止存在"任何异常 → 返回 []"的测试断言。

- 未知 `client.fetch_orders` 的具体异常类型与是否内建重试 → 上述异常分类需按实际 SDK 调整。
- 未知调用方（对账任务）是否区分"空"和"错"的语义 → 若不区分，问题更严重，需要同步修改调用方。
- 未知是否存在 SLA / 重试 / 告警基础设施。
- 未知"测试就能过"指的是哪些测试 —— 如果是单元测试在 mock 抛错时期望 `[]`，那这些测试本身需要被修正。
