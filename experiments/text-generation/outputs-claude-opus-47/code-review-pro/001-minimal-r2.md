## 主要问题（按严重度排序）

### 🔴 Critical — SQL Injection
- `query = f"SELECT * FROM notes WHERE customer_id = {customer_id}"` 直接将来自 HTTP query 参数的 `customer_id` 拼接进 SQL。
- 攻击者可注入任意 SQL（例如 `1 OR 1=1`、`1; DROP TABLE notes--`，或基于 UNION 的数据外泄）。
- 即便 `customer_id` 通常是数字，HTTP 参数是字符串，未做类型强制 → 风险确凿。
- **应采用参数化查询**（占位符由 DB 驱动绑定），而非字符串格式化或手动转义。

### 🔴 High — 缺少授权 / 越权访问 (IDOR)
- 函数仅根据 `customer_id` 拉取数据，没有任何当前用户与 `customer_id` 的归属/权限校验。
- 内部 CRM 仍可能存在越权风险：任何登录用户可遍历其他客户 ID 读取备注。
- 需在调用层或本函数中加入访问控制（如租户/角色检查）。

### 🟠 Medium — `SELECT *`
- 选择全部列会泄露未来新增的敏感字段（如内部标签、PII），并导致接口契约脆弱。
- 应显式列出所需列。

### 🟠 Medium — 输入校验缺失
- 未校验 `customer_id` 类型、范围、是否存在。
- 缺失时（`None`、空字符串）会产生 SQL 语法错误并可能把内部异常透出到 HTTP 响应。

### 🟡 Low/Medium — 无分页 / 无排序
- `notes` 可能随时间增长，无 `LIMIT`/`OFFSET` 与确定性 `ORDER BY` 会导致：
  - 响应体过大、内存与延迟问题；
  - 返回顺序不确定，影响前端展示与一致性测试。

### 🟡 Low — 错误处理与日志
- 未捕获/记录 DB 异常。攻击者通过差异性错误信息可进行盲注或信息收集。
- 应在调用边界统一处理异常，避免把堆栈/SQL 错误返给客户端。

### 🟡 Low — 资源 / 事务管理
- 未知 `db.execute` 是否托管连接/游标生命周期。若非 ORM 管理，需确保连接归还与异常下的清理。

### 🟡 Low — 返回类型与契约
- `dict(row)` 依赖驱动是否支持行映射（如 `sqlite3.Row`）。若驱动返回元组将抛错。
- 函数无类型注解、无 docstring，调用方对返回结构无契约。

### 🟡 Info — 可观测性
- 缺少结构化日志/指标（查询耗时、返回行数），不利于后续性能与安全审计。

## 测试建议（当前 0 测试）

### 单元测试（使用 mock/fake DB）
1. **SQL 参数化验证**：断言传给 `db.execute` 的是带占位符的 SQL，且参数以绑定形式传入；含特殊字符的输入不应改变 SQL 文本。
2. **注入 payload 测试**：传入 `"1 OR 1=1"`、`"1; DROP TABLE notes"`、`"' UNION SELECT ..."` 等，断言：
   - 不会生成多条语句；
   - 返回为空或抛出受控的校验错误。
3. **输入校验**：`None`、空字符串、负数、超长字符串、非数字、极大整数边界。
4. **返回结构**：空结果返回 `[]`；多行结果保持字段一致；列名稳定。
5. **异常路径**：DB 抛错时函数行为（透传 vs 包装），确保不泄露内部细节。

### 集成测试（针对真实 schema 的临时 DB）
6. 准备种子数据，校验过滤正确性（仅返回匹配 `customer_id` 的行）。
7. 大数据量下的分页/性能基线（若引入 `LIMIT`）。
8. 索引使用：用 `EXPLAIN` 断言走 `customer_id` 索引（如果有）。

### 安全/授权测试
9. 越权场景：用户 A 请求用户 B 的 `customer_id`，应被拒绝（需在路由/服务层加权限后测试）。
10. 模糊测试（fuzzing）`customer_id` 参数，确保不出现 500 泄露 SQL 错误。

### 回归/契约测试
11. 固化返回字段集合，防止 `SELECT *` 带来的隐式契约漂移（建议在改为显式列后加 schema 断言）。
