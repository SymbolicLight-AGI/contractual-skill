# 任务结果 finance-contract-005

## 判断结论

**不能直接走合同。** 当前材料存在范围过宽、阶段不清、接口未盘点、验收标准缺失等多项重大风险，必须经多角色复核后才能进入签约流程。

## 已知事实

- 项目为集团级知识库 + 多 Agent 工作流。
- 范围覆盖：1 个集团总部知识库、5 个业务 Agent、对接 OA/CRM/售后工单 3 个系统，试点后推广至 8 家分公司。
- 客户要求：单一合同覆盖全部阶段、报价含所有系统接口、验收标准"后续再定"。

## 财务与合同风险

- **范围无边界**：一份合同覆盖试点 + 8 家分公司推广，存在无限交付与无限服务暴露。
- **接口未盘点**：OA/CRM/售后工单接口数量、版本、是否含第三方均未明确，"报价含所有接口"等于开口承诺。
- **验收标准缺失**：验收"后续再定"将直接导致回款无法触发、争议高、应收账款滞留。
- **付款里程碑未约定**：未见阶段付款、试点验收与推广验收的划分，回款风险高。
- **多分公司主体不清**：8 家分公司是否独立签约、独立付款、独立验收未定，可能引发主体与发票风险。
- **多 Agent 与知识库的内容/数据归属、保密、合规边界未约定**：涉及法务与风险评审。

## 需确认的问题

- 合同是采用"主合同 + 分阶段 SOW"还是单一合同？试点与推广能否拆分？
- 5 个 Agent 的功能清单、模型/算力归属、迭代次数上限？
- 三大系统接口的数量、协议、对端是否需第三方厂商配合及其费用承担方？
- 验收标准能否在签约前形成可量化条款（功能、性能、数据、稳定性）？
- 推广至 8 家分公司的触发条件、单价、工作量是否单独计价？
- 付款里程碑、发票主体、账期、逾期条款？
- 数据安全、知识库内容合规、模型输出责任如何划分？

## 必须复核的角色与事项

- **售前**：澄清范围边界、Agent 功能清单、推广范围与触发条件，重新形成可签约的范围说明。
- **交付**：评估 5 个 Agent + 3 系统接口 + 8 家分公司推广的工作量、人月与排期，确认接口可行性与第三方依赖。
- **财务**：核定报价口径（是否含全部接口/推广）、付款里程碑、发票主体、账期与回款风险，禁止在验收标准缺失前确认报价。
- **法务**：审阅"一份合同覆盖全部阶段"的结构风险，建议主合同 + SOW；补充验收、变更、知识产权、数据与保密条款。
- **风险审查**：评估无限服务、验收空白、跨分公司主体、第三方系统对接的综合敞口，给出是否准入及前置条件。

## 处置建议

- 当前**不批准**直接走合同，也不批准"报价含所有接口"与"验收后续再定"两项让步。
- 建议改为：主合同 + 试点 SOW 先行，推广阶段以补充 SOW 触发；验收标准须在签约前量化。
- 所有让步（单一合同、接口全包、验收延后）须由售前、交付、财务、法务、风险五方书面会签后方可推进。

## 不确定性标注

- 客户是否接受拆分为主合同 + SOW、8 家分公司的签约主体与付款主体、第三方接口范围与费用承担方，均未在材料中给出，需向客户确认。
