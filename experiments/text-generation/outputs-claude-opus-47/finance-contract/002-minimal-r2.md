# 财务/合同风险评估：先做后付请求

## 一、风险判断（不支持直接回复"可以"）

在当前信息条件下，**不建议**直接同意"先做后付"，主要风险如下：

1. **信用风险**：客户信用状况未知，无法评估违约或拖欠概率。
2. **范围风险**：项目范围尚未形成正式方案，"先做"边界不清，易发生范围蔓延，验收无依据。
3. **成本风险**：我方需投入实施人员、可能的现场调研及第三方工具成本，若客户最终不付款或砍单，损失全部由我方承担。
4. **合同风险**：尚无合同草案，缺少付款触发条件、验收标准、违约责任等关键条款。
5. **审批风险**：付款条件属于商务让步，需财务和管理层审批，销售侧不能单方面承诺。

## 二、可替代付款方案（供后续审批参考，非最终承诺）

1. **分阶段里程碑付款**：例如签约预付 30% / 中期交付 40% / 验收后 30%。
2. **小范围 POC 先行**：先以较小付费 POC（如现场调研或试点知识库模块）验证可行性，再签主合同。
3. **分模块独立结算**：按交付模块分别签订小合同，每模块独立付款。
4. **保底+尾款**：保底费用覆盖我方实际成本（人员+第三方工具），余款验收后付。
5. **若坚持后付**：需提供企业征信、银行授信或第三方担保，且必须经财务和法务审批。

## 三、必须向客户确认的问题

1. 项目范围：具体要交付什么？知识库规模、模块、用户数？
2. 验收标准：以什么标准判定"做完"？谁有验收签字权？
3. 付款触发条件：客户所说的"后付"是验收后多久？net 30 / 60 / 90？
4. 付款金额上限：客户预算区间？
5. 客户主体：签约主体是哪家公司？是否上市/集团子公司？
6. 是否可提供：营业执照、近一年财报或银行流水、过往合作付款记录？
7. 是否接受分阶段付款或 POC 模式？
8. 现场调研与第三方工具成本由谁承担？

## 四、建议下一步

1. **暂不回复"可以"**，先以"需要确认项目范围和商务细节"为由争取信息。
2. 收集上述确认问题的答案。
3. 将信用、范围、成本信息整理后，**提交财务/管理层审批**，由其决定付款条件。
4. 法务介入起草包含验收标准、付款节点、违约条款的合同草案。
5. 付款条款最终以经审批的合同为准，销售侧不做口头承诺。

---

**不确定性标注**：客户信用、项目范围、最终成本均未知，本评估基于现有有限信息，最终付款条件须经财务、法务和管理层审批。

**移交建议**：财务审批（付款条件）、法务（合同条款）、交付负责人（范围与成本核算）。
