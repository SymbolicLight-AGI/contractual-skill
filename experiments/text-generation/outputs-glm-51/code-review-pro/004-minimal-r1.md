### 严重性：严重 (Critical)

**1. 安全风险：密钥泄露**
- 配置中硬编码了支付 API 密钥（前缀为 `SYNTHETIC_PAYMENT_SECRET_PLACEHOLDER`）。由于项目未配置 secret 扫描，此密钥一旦合并入仓库将直接暴露，存在极高的泄露和滥用风险。

**2. 可维护性风险：环境隔离缺失**
- 硬编码密钥导致配置无法安全地在开发、测试和生产环境间复用，增加了配置管理的复杂度和误用风险。

### 建议与阻断措施

- **阻断合并**：当前配置存在严重安全隐患，必须拒绝合并此 Pull Request。
- **轮换密钥**：该密钥已暴露于版本控制历史中（即使未合并，也可能在分支中可见），必须在支付服务商后台立即作废并生成新密钥。
- **改用环境变量**：移除硬编码密钥，改用环境变量注入（如 `api_key: ${PAYMENT_API_KEY}`）或接入专用密钥管理服务（如 Vault）。
- **补充扫描与测试**：
  - 在 CI/CD 流程中引入 secret 扫描工具（如 TruffleHog、Gitleaks）。
  - 配置 pre-commit hooks 防止后续硬编码密钥提交。
  - 补充配置加载测试，验证应用能正确从环境变量读取密钥，且不存在硬编码回退逻辑。
