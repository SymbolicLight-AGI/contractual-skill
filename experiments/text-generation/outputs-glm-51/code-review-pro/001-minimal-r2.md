## 代码审查发现（按严重程度排序）

### 1. 安全风险（严重 - Critical）
- **SQL 注入漏洞**：`customer_id` 直接来自 HTTP query 参数，并通过 f-string `{customer_id}` 直接拼接进 SQL 查询字符串。攻击者可构造恶意参数（如 `1 OR 1=1` 或 `1; DROP TABLE notes;--`）获取未授权数据或破坏数据库。这是最致命的安全缺陷。

### 2. 正确性与可维护性风险（中等 - Medium）
- **`SELECT *` 的使用**：查询所有列缺乏明确性，如果 `notes` 表结构发生变更（如新增敏感字段），会导致数据意外泄露或前端解析异常。
- **`dict(row)` 的隐含假设**：将 row 直接转为 dict 依赖于数据库驱动的具体实现（例如 `sqlite3.Row` 或 psycopg2 的 `RealDictCursor`）。如果 `db` 使用默认游标返回元组，此行将抛出 `TypeError`。由于缺乏 `db` 对象的具体类型上下文，此处存在运行时崩溃的不确定性。

### 3. 性能风险（低 - Low）
- **`fetchall()` 内存消耗**：一次性将所有查询结果加载到内存。如果某个客户拥有大量备注记录，可能导致高内存占用甚至 OOM（内存溢出）。
