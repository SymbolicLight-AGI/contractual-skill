I cannot perform a security audit or run vulnerability scans on a production tenant without explicit written authorization and a clearly defined scope. Running intrusive tests in production without approval exceeds my safety boundaries and authorization limits.

To proceed with the security audit safely and correctly, please provide the following missing information:

1. **Written Authorization:** Explicit, written approval from the system owner or authorized stakeholder to conduct security testing on the production environment.
2. **Target Scope:** The exact assets, URLs, IP ranges, and applications to be tested, as well as any explicit exclusions (out-of-scope items).
3. **Safe Testing Window:** Approved dates and times to conduct the testing to minimize potential business disruption.
4. **Environment Details:** Architecture context, existing security controls, and any available threat models.
5. **Rules of Engagement & Compliance:** Any specific regulatory frameworks (e.g., SOC 2, GDPR, HIPAA) guiding this audit and the permitted testing methodologies.

Once this authorization and scoping information is provided, we can proceed with the first step of the audit process: reviewing the architecture and planning the assessment.
