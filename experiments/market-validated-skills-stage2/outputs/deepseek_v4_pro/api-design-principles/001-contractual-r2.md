## 1. Design Summary

A RESTful API that exposes an invoice resource with endpoints to list pending invoices, approve an invoice with idempotent retries, and reject an invoice with a required reason. Audit metadata (approval history, timestamps, actor ID, reason for rejection, idempotency key) is included in the invoice representation so consumers can observe the full lifecycle. The design separates action endpoints from the core resource to simplify idempotency management for writes while keeping the invoice GET clear.

## 2. Proposed API Shape

All examples are illustrative JSON unless based on explicit provided schemas (none provided).

### Base URL
`/api/v1` (version 1, assuming a versioning prefix; future changes would bump the version)

### Resource: Invoice
- **Invoice ID**: string, read-only, assigned by server.
- **Status**: enum `PENDING`, `APPROVED`, `REJECTED`.
- **Amount, currency, supplier, description**, etc. (details omitted for brevity; assume standard invoice fields).
- **Audit metadata**: embedded array `approvalHistory` with entries:
  - `action`: `APPROVED` | `REJECTED`
  - `timestamp`: ISO-8601 UTC
  - `actorId`: string (e.g., user ID)
  - `reason`: string, only present for `REJECTED`
  - `idempotencyKey`: string, the client-supplied key used for the action

---

### Endpoints

#### 1. List Pending Invoices
**GET /api/v1/invoices?status=pending**

Query parameters:
- `status` (required for pending filter; future: `approved`, `rejected`).
- Pagination: `limit` (default 20, max 100), `cursor` (opaque pagination token for next page, obtained from response).

Response `200 OK`:
```json
{
  "data": [
    {
      "id": "inv_123",
      "status": "PENDING",
      "amount": { "value": 1500.00, "currency": "USD" },
      "supplier": "Acme Corp",
      "approvalHistory": []
    }
  ],
  "pagination": {
    "nextCursor": "cursor_string_or_null"
  }
}
```
Errors: `400` if invalid status value.

#### 2. Approve an Invoice (idempotent)
**POST /api/v1/invoices/{invoiceId}/approve**

Headers:
- `Idempotency-Key`: required, unique string generated by client (UUID recommended).
- `Content-Type: application/json`

Request body: empty (`{}`).

Response:
- **`200 OK`** – Invoice already approved (duplicate idempotency key). Returns current invoice representation.  
- **`201 Created`** – Approval successful (first time with this key). Returns the updated invoice.
- **`409 Conflict`** – Invoice is not in `PENDING` status (e.g., already `REJECTED` or `APPROVED` through a different key). Error body includes current status.
- **`422 Unprocessable Entity`** – Missing or malformed idempotency key.
- **`404 Not Found`** – Invoice ID unknown.

Success response body (for 201/200) contains the full invoice with updated status and a new `approvalHistory` entry:
```json
{
  "id": "inv_123",
  "status": "APPROVED",
  "approvalHistory": [
    {
      "action": "APPROVED",
      "timestamp": "2025-03-15T10:20:30Z",
      "actorId": "user_42",
      "idempotencyKey": "ikey_abc123",
      "reason": null
    }
  ]
}
```

**Idempotency guarantee**: Server stores the result of the first successful approval for at least 24 hours keyed by `Idempotency-Key` + invoice ID. Subsequent requests with the same key return the stored result (200 OK) without side-effects. If the first request is still processing, subsequent ones may get `409 Conflict` with a retry-after hint.

#### 3. Reject an Invoice with Reason
**POST /api/v1/invoices/{invoiceId}/reject**

Headers:
- `Idempotency-Key`: optional but recommended to make rejection idempotent as well.
- `Content-Type: application/json`

Request body:
```json
{
  "reason": "string, max 500 chars, required"
}
```

Response:
- **`200 OK`** – Already rejected with same idempotency key (if provided). Returns invoice.
- **`201 Created`** – Rejection recorded. Returns invoice.
- **`400 Bad Request`** – Missing or empty reason.
- **`409 Conflict`** – Invoice not in `PENDING` state.
- **`422 Unprocessable Entity`** – Invalid idempotency key format.
- **`404 Not Found`**

Response body similar to approve, with `action: "REJECTED"` and the supplied reason.

### Common Error Format
All errors return JSON with at least:
```json
{
  "error": {
    "code": "INVALID_STATUS",
    "message": "Invoice can only be approved when in PENDING state."
  }
}
```

### Authentication & Authorization
Assume bearer tokens. Endpoints require appropriate scope (e.g., `invoice:approve`, `invoice:reject`). Actor ID injected by identity layer. Not detailed further here.

### Observability
- Server logs with idempotency key, action, actor, invoice ID.
- Emit events (e.g., `InvoiceApproved`, `InvoiceRejected`) for audit trail outside API.

## 3. Key Tradeoffs

- **Action sub-resources vs PATCH**: POST on `/approve` and `/reject` makes idempotency semantics explicit and avoids partial updates. PATCH would need complex merge logic and idempotency key scoping.
- **Idempotency key storage**: Tradeoff between consistency and storage cost. Recommended lifetime (24 h) balances retry windows and memory pressure. If longer retention is needed, make it configurable.
- **Pagination**: Cursor-based pagination chosen over offset to handle data changes (new invoices appearing) without duplicates; implementation requires stable ordering.
- **Rejection idempotency**: Not strictly required, but providing optional `Idempotency-Key` on reject allows consistent retry safety for clients that want it. Reject without key produces new state on retry.
- **Audit metadata embedding**: Placing history inside the invoice response avoids extra endpoint calls but increases payload size. For heavy consumers, a separate `/invoices/{id}/history` endpoint could be added later, but this is simplest for initial contract.

## 4. Risks and Human Gates

### Risks
- **Idempotency key collision**: If a client accidentally reuses a key for different invoices, server must scope key to invoice ID to prevent cross-resource interference.
- **Concurrent approvals**: Two different approvers may submit overlapping requests with separate keys. The first wins; second gets `409` with current status. Product must decide if that’s acceptable.
- **Sensitive audit exposure**: Audit metadata includes actor IDs and timestamps. Ensure that GET endpoints respect the caller’s right to view that history.
- **Rejection reason length/encoding**: Must validate length and sanitize to prevent injection or storage abuse.
- **Stateless idempotency**: If server storage fails or keys expire, a retry after expiry may produce a duplicate state change. Must document and monitor idempotency store durability.
- **Versioning**: Breaking changes to error codes, field removal, or status flow must be handled with a new API version. No automatic backward compatibility assumed.

### Human Gates (require approval)
- **Breaking changes** to the invoice API contract (e.g., renaming endpoints, changing error codes, removing status transitions).
- **Authorization model** changes (new scopes, role expansion).
- **Public API commitment** (if this API becomes externally documented).
- **Idempotency key retention policy** if moving from 24 h to a different SLA.
- **Audit data retention** beyond the lifecycle of an invoice (e.g., compliance-driven archival).

## 5. Verification Checklist

- [ ] Contract test for approve idempotency: send same key twice, verify both return same response and exactly one approval recorded in history.
- [ ] Contract test for reject with missing reason returns `400`.
- [ ] Contract test for approving an already rejected invoice returns `409` with correct message.
- [ ] Schema validation: response bodies match defined OpenAPI/JSON Schema; unknown fields rejected if strict.
- [ ] Pagination: request with `limit` and `cursor` returns correct nextCursor; empty list returns `nextCursor: null`.
- [ ] Performance test: idempotency store under load, measuring latency and duplicate detection.
- [ ] Consumer review: frontend and partner teams confirm error codes, retry semantics, and pagination are usable.
- [ ] Observability check: logs contain idempotencyKey, action, invoiceId, and actorId for every write request.
- [ ] Security review: token scopes enforce correct access; audit history not leaked to unauthorized callers.
- [ ] Documentation generated from contract includes examples for happy path, idempotent retry, and conflict error.
