## 1. Design summary

Design a REST API around the `invoice` resource and its approval lifecycle.

Supplied requirements:

- List pending invoices.
- Approve one invoice.
- Reject one invoice with a reason.
- Support idempotent approval retries.
- Expose audit metadata.
- No code required.

Assumptions:

- Consumers are authenticated users or services with permission to approve/reject invoices.
- Invoice approval state is modeled as a lifecycle status, e.g. `pending_approval`, `approved`, `rejected`.
- Audit metadata can be exposed safely if it does not include sensitive internal-only fields.
- Approval and rejection are customer-visible state changes and should be explicitly authorized.

Recommended API style: REST, versioned under `/v1`.

# List pending invoices

Lists invoices pending approval.

| Parameter | Type | Required | Description |
|---|---:|---:|---|

| `cursor` | string | optional | Cursor for pagination. |

# Approve an invoice

### `POST /v1/invoices/{invoiceId}/approval`

Approves a pending invoice.

Use `Idempotency-Key` to safely retry approval requests.

#### Headers

| Header | Required | Description |
|---|---:|---|
| `Authorization` | yes | Bearer token or equivalent auth mechanism. |
| `Idempotency-Key` | strongly recommended / required for retries | Unique key generated by the client for this approval attempt. |
| `Content-Type` | yes | `application/json`. |

Recommendation: require `Idempotency-Key` for this write endpoint to guarantee safe approval retries.

#### Request body

```json
{
  "comment": "Approved for payment."
}
```

| Field | Type | Required | Description |
|---|---:|---:|---|
| `comment` | string | no | Optional approver comment. |

#### Example request

```http
POST /v1/invoices/inv_123/approval
Idempotency-Key: 8f6f4d9c-8261-4b35-94ab-93f04f44d1a7
Content-Type: application/json
```

```json
{
  "comment": "Approved for payment."
}
```

#### Example response: `200 OK`

```json
{
  "id": "inv_123",
  "status": "approved",
  "amount": {
    "currency": "USD",
    "value": "1250.00"
  },
  "vendor": {
    "id": "ven_456",
    "name": "Example Vendor"
  },
  "dueDate": "2026-06-15",
  "createdAt": "2026-05-01T10:30:00Z",
  "updatedAt": "2026-05-24T09:00:00Z",
  "audit": {
    "submittedAt": "2026-05-20T14:45:00Z",
    "submittedBy": {
      "id": "usr_111",
      "displayName": "Alex Morgan"
    },
    "approvedAt": "2026-05-24T09:00:00Z",
    "approvedBy": {
      "id": "usr_222",
      "displayName": "Sam Rivera"
    },
    "rejectedAt": null,
    "rejectedBy": null,
    "rejectionReason": null
  }
}
```

#### Idempotency behavior

For `POST /v1/invoices/{invoiceId}/approval`:

- If the same `Idempotency-Key` is retried with the same request body, return the original successful response.
- If the same `Idempotency-Key` is reused with a different request body, return `409 Conflict`.
- If the invoice is already approved by the same completed idempotent request, return the original `200 OK`.
- If the invoice was already approved by a different request, return either:
  - `200 OK` with the approved invoice if approval is considered naturally idempotent, or
  - `409 Conflict` if the API must distinguish duplicate/conflicting approvals.

Recommendation: use `200 OK` for already-approved invoices when the resulting state is the same and no conflicting business rule is violated.

## Error model

Use a consistent JSON error shape.

```json
{
  "error": {
    "code": "invoice_not_pending_approval",
    "message": "Invoice cannot be approved because it is not pending approval.",
    "details": {
      "invoiceId": "inv_123",
      "currentStatus": "rejected"
    },
    "requestId": "req_abc123"
  }
}
```

### Recommended status codes

| Status | Scenario |
|---:|---|
| `400 Bad Request` | Invalid request body, invalid query parameter, malformed cursor. |
| `401 Unauthorized` | Missing or invalid authentication. |
| `403 Forbidden` | Authenticated caller lacks permission to view, approve, or reject invoice. |
| `404 Not Found` | Invoice does not exist or caller cannot access it. |
| `409 Conflict` | State conflict, e.g. reject an already approved invoice, reused idempotency key with different payload. |
| `422 Unprocessable Entity` | Semantically invalid request, e.g. missing rejection reason, reason too long. |
| `429 Too Many Requests` | Rate limit exceeded. |
| `500 Internal Server Error` | Unexpected server error. |

### Example validation error

```json
{
  "error": {
    "code": "rejection_reason_required",
    "message": "A rejection reason is required.",
    "details": {
      "field": "reason"
    },
    "requestId": "req_abc123"
  }
}
```

## 3. Key tradeoffs

### Action endpoints vs status update endpoint

Recommended:

```http
POST /v1/invoices/{invoiceId}/approval
POST /v1/invoices/{invoiceId}/rejection
```

Alternative:

```http
PATCH /v1/invoices/{invoiceId}
```

With body:

```json
{
  "status": "approved"
}
```

Tradeoff:

- Action endpoints are clearer for business workflows like approval and rejection.
- `PATCH` is more generic but can blur business rules and lifecycle transitions.
- Because approval and rejection have different validation rules, especially rejection reason and idempotency behavior, action endpoints are preferable.

### `200 OK` vs `201 Created` for approval

Recommended: `200 OK`.

Reason:

- Approval changes the state of an existing invoice.
- No separate public approval resource is required by the stated requirements.

If approval records become first-class resources later, consider:

```http
POST /v1/invoices/{invoiceId}/approvals
```

returning `201 Created`.

### Cursor pagination vs offset pagination

Recommended: cursor pagination.

Reason:

- Pending approval lists can change frequently as invoices are approved/rejected.
- Cursor pagination is more stable under concurrent changes than offset pagination.

## 4. Risks and human gates

### Security and privacy risks

- Audit metadata may expose names, user IDs, timestamps, or rejection reasons.
- Rejection reasons may contain sensitive business information.
- Authorization must distinguish:
  - who can list pending invoices,
  - who can approve,
  - who can reject,
  - who can view audit metadata.

Security review required before public exposure of audit fields.

### Compatibility risks

- Status names such as `pending_approval`, `approved`, and `rejected` become part of the public contract.
- Error codes become contractually important for clients.
- Changing audit field names or meanings later may be breaking.

Human approval required for public API commitments and versioning decisions.

### Operational risks

- Write endpoints need request tracing through `requestId`.
- Approval retries must be idempotent to avoid duplicate state transitions or duplicate downstream payment actions.
- Concurrent approve/reject requests need deterministic conflict behavior.

### Human gates

Require approval from:

- Product: approval lifecycle semantics and visible audit fields.
- Security: permissions and sensitive data exposure.
- API/platform owners: endpoint naming, idempotency policy, error format.
- QA: contract and state-transition test coverage.
- Documentation: developer-facing examples and retry guidance.

## 5. Verification checklist

### Contract behavior

- [ ] `GET /v1/invoices?status=pending_approval` returns only invoices visible to the caller.
- [ ] Pagination works with `limit` and `cursor`.
- [ ] Pending list response includes required audit metadata.
- [ ] `POST /v1/invoices/{invoiceId}/approval` approves a pending invoice.
- [ ] `POST /v1/invoices/{invoiceId}/rejection` rejects a pending invoice.
- [ ] Rejection requires a non-empty `reason`.

### Idempotency

- [ ] Retrying approval with the same `Idempotency-Key` and same body returns the same result.
- [ ] Reusing an `Idempotency-Key` with a different body returns `409 Conflict`.
- [ ] Concurrent approval requests produce one deterministic final state.
- [ ] Concurrent approve and reject requests produce a clear conflict outcome.

### Error handling

- [ ] Invalid invoice ID returns `404 Not Found` or access-safe equivalent.
- [ ] Unauthorized caller receives `401 Unauthorized`.
- [ ] Authenticated but unpermitted caller receives `403 Forbidden`.
- [ ] Invalid lifecycle transition returns `409 Conflict`.
- [ ] Validation failures return structured errors with stable `error.code`.

### Security and audit

- [ ] Audit fields are reviewed for sensitive data exposure.
- [ ] All write operations record actor, timestamp, resulting state, and request ID.
- [ ] Logs avoid storing sensitive rejection details unless approved.
- [ ] Permissions are verified separately for listing, approving, rejecting, and audit visibility.

### Compatibility

- [ ] Public enum values are documented.
- [ ] Error codes are documented.
- [ ] Future changes are additive where possible.
- [ ] Breaking changes require a new version or approved migration plan.
