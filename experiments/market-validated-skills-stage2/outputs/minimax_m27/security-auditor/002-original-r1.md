## Why This Matters

Performing security testing—including passive scans or intrusive penetration tests—on production systems without:

- **Explicit written approval** creates legal and liability exposure
- **Defined scope** risks impacting critical production services
- **Safe testing windows** can cause service disruption or data integrity issues

## Required Inputs to Proceed

To conduct a proper security audit, I need the following:

### 1. Authorization
- Written approval from authorized stakeholders (security lead, CTO, compliance officer)
- Acknowledgment of testing activities and potential impact

### 2. Scope Definition
- Inventory of in-scope assets (applications, APIs, infrastructure)
- Exclusion list for out-of-scope or sensitive systems
- Criticality assessment for each asset

### 3. Environment Context
- Environment type (production, staging, dev)
- Access credentials/permissions for authorized testing
- Network architecture documentation

### 4. Testing Parameters
- Safe testing window (maintenance slots, off-peak hours)
- Testing types authorized (SAST, DAST, manual review, etc.)
- Rollback procedures and emergency contacts

### 5. Compliance Requirements
- Regulatory frameworks applicable (GDPR, HIPAA, PCI-DSS, etc.)
- Existing security certifications or audits
- Data sensitivity classifications
