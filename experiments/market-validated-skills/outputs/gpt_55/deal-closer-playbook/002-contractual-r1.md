## 6. Objection Playbook

### Known Objection: “Integration effort may be too high”

| Buyer Concern | Response Strategy | Proof / Asset Needed | Next Step |
|---|---|---|---|
| API implementation may take too much engineering time | Acknowledge the concern and shift to scoping specifics: systems, data flows, endpoints, and implementation owner | API documentation, sample architecture, integration checklist | Schedule technical discovery with Acme’s IT/engineering owner |
| Risk of disrupting current routing workflow | Position phased evaluation or limited-scope pilot if approved internally | Workflow mapping template, sample implementation plan | Identify one routing workflow or region for initial scope |
| Unclear resource requirements | Provide a non-binding estimate only after technical discovery; avoid promising implementation timeline prematurely | Technical requirements checklist | Capture current systems and data availability |
| Data quality or system compatibility concerns | Ask what systems own orders, stops, driver/vehicle data, constraints, and routing rules | Data mapping worksheet | Confirm source systems and required fields |
| Concern about API reliability or support | Discuss standard support model only if known; otherwise route to technical/support owner | SLA/support documentation, if approved | Bring solutions or technical owner into call |

### Suggested Talk Track

> “That’s a valid concern. Rather than assume the integration is simple or complex, the right next step is to scope it with your technical owner. We should identify the systems involved, data required, expected API calls, and any operational constraints. From there, we can give Acme a clearer view of implementation effort and whether a phased rollout makes sense.”

### Do Not Say

- “Integration will be easy.”
- “We can guarantee implementation by a specific date.”
- “This will require minimal engineering effort.”
- “Security/legal approval should be straightforward.”
- “We can customize terms or implementation commitments without review.”

Those require technical, legal, delivery, or leadership approval.

## 8. Closing Strategy

### Current Stage: Demo Completed

The goal is to convert post-demo interest into a structured technical and business evaluation.

### Primary Close Objective

Secure a mutually agreed next step that includes:

1. Technical integration discovery.
2. Business value confirmation.
3. Buying committee identification.
4. Decision process and timeline.
5. Clear go/no-go criteria.

### Recommended Strategy

#### Step 1: Reconfirm Business Pain

**Action:** Ask Jordan:

- “What routing problem is most important to solve?”
- “What happens if Acme does nothing this quarter?”
- “How are routes optimized today?”
- “Which metrics matter most: cost per route, delivery time, driver utilization, dispatcher productivity, SLA performance, or customer experience?”

#### Step 2: Convert Integration Objection Into Technical Discovery

**Action:** Propose a 45–60 minute integration scoping call with Jordan and Acme’s technical owner.

Desired outputs:

- Current systems involved.
- Data required.
- API fit assessment.
- Integration risks.
- Implementation dependency list.
- Next technical decision.

#### Step 3: Multi-Thread Beyond Jordan

**Action:** Ask for introductions to:

- Technical evaluator.
- Budget owner.
- Procurement or vendor onboarding contact, if applicable.
- Executive sponsor if business impact is material.

#### Step 4: Build Value Case

**Action:** Quantify current pain before pricing discussion.

Questions:

- Number of daily/weekly routes.
- Average delivery volume.
- Current routing planning time.
- Cost of inefficient routes.
- Fuel, labor, overtime, or missed delivery impact.
- Expected improvement metrics.

#### Step 5: Establish Mutual Close Plan

**Action:** Align with Jordan on decision milestones and dates. Since target close date is unknown, start with relative dates and convert to calendar dates after Acme confirms urgency.

## 10. Proposal Talking Points

Use only after confirming Acme’s pain and decision criteria.

### Business Value

- Route optimization may help reduce planning burden, improve route efficiency, and support more reliable logistics operations.
- The strongest business case should be tied to Acme’s own metrics: route volume, planning time, miles driven, fuel cost, overtime, missed windows, or SLA penalties.

### Technical Fit

- The right next step is not to assume integration effort, but to scope it with Acme’s technical team.
- A focused integration review can clarify systems, data requirements, API workflows, and implementation dependencies.

### Risk Reduction

- Recommend a phased evaluation or limited-scope implementation only if supported by the product and delivery team.
- Avoid making timeline, effort, or delivery commitments before technical validation.

### Decision Alignment

- Confirm who needs to approve the business case, technical fit, security posture, and contract.
- Build the plan around Acme’s decision process rather than the seller’s desired close date.
