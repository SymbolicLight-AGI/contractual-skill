# 5. Primary Objection Response

## Objection: “Integration effort may be too high”

### Why They Say This

The buyer likely worries that the API will require too much engineering time, disrupt current dispatch workflows, or create dependency on technical resources they do not control.

This may also be a proxy objection for:

- “We are interested, but implementation feels risky.”
- “Operations wants this, but engineering may not prioritize it.”
- “We do not know how this fits into our current TMS or dispatch stack.”
- “We have been burned by API integrations before.”

### Response Framework

1. **Acknowledge**  
   “That is a fair concern. Route optimization is only valuable if it can fit into your current workflow without creating a heavy lift.”

2. **Reframe**  
   “Rather than treating this as a full implementation decision today, let’s first scope the smallest useful integration path.”

3. **Evidence**  
   Use API documentation, sandbox access, sample payloads, integration checklist, and a technical working session to reduce uncertainty.

4. **Confirm**  
   “If we can show your technical team that the first integration can be completed with limited lift, would that address the main concern?”

### Example Response

“Jordan, that is a completely fair concern. Route optimization only makes sense if the integration effort does not outweigh the operational benefit. Rather than asking Acme to commit before the technical path is clear, I suggest we schedule a 45-minute integration review with whoever owns your dispatch or routing systems. We can walk through the API docs, expected inputs and outputs, authentication, sample requests, and the lightest possible implementation path. If we can confirm the first version is straightforward, then we can move to business case and timeline. If we find it is heavier than expected, we will know that early and can adjust the plan.”

### If They Push Back

If Jordan says engineering does not have bandwidth:

“Understood. In that case, let’s make the technical review even lighter. We can send an integration brief first and ask your technical lead to mark the top three concerns asynchronously. Then we can use the live session only to address those issues. Would that be easier for your team?”
