## 6. Objection Playbook

### Known Objection: “Integration effort may be too high”

| Objection | Likely Meaning | Response Strategy | Proof / Asset Needed | Next Step |
|---|---|---|---|---|
| Integration effort may be too high | Acme is worried about engineering time, system compatibility, data requirements, or disruption. | Acknowledge the concern, reduce ambiguity, and propose a scoped technical review. | API docs, integration architecture overview, sample payloads, implementation checklist, sandbox access if available. | Schedule a technical integration workshop with Acme’s IT/engineering owner. |

### Suggested Internal Talk Track

> “That’s a valid concern. Before we assume the integration is heavy, let’s map the actual systems, data inputs, and workflow. We can use a technical workshop to identify required endpoints, data format, implementation steps, and resource needs. From there, Acme can make a more informed decision on effort versus operational benefit.”

### Discovery Questions For This Objection

1. What transportation, dispatch, order management, or fleet systems would need to connect to the API?
2. Who owns those systems internally?
3. What data is currently available for routes, stops, vehicles, constraints, and delivery windows?
4. Is Acme looking for a full production integration immediately or a limited pilot first?
5. What integration approaches are acceptable: API-to-API, batch upload, middleware, sandbox prototype, or phased rollout?
6. What level of engineering effort would Acme consider reasonable?
7. Has Acme integrated similar APIs before?
8. What would make the integration feel low-risk?

### Anticipated Objections

| Potential Objection | Response Strategy |
|---|---|
| “We do not have engineering bandwidth.” | Explore phased pilot, lower-scope integration, or technical validation first. Do not promise delivery timelines without approval. |
| “We need to see ROI before committing.” | Build a value model using Acme’s current routing costs, planning time, mileage, delivery volume, and late-delivery metrics. |
| “We may build this internally.” | Compare time-to-value, maintenance burden, algorithm quality, and opportunity cost, using only validated product capabilities. |
| “Security or data privacy may be an issue.” | Route to security/legal review. Do not make security commitments without the appropriate owner. |
| “Procurement will take time.” | Identify vendor onboarding and contract steps early; create a mutual close plan with procurement milestones. |

## 10. Proposal Talking Points

Use these internally and in customer-facing discussions only after confirming product accuracy and approval boundaries.

### Recommended Themes

1. **Reduce integration uncertainty**
   - “The next step is to map the actual integration effort rather than assume it is high.”

2. **Tie API value to operational outcomes**
   - “We should quantify how route optimization impacts planning time, mileage, capacity, on-time delivery, or cost per route.”

3. **Start with a controlled validation**
   - “A scoped technical validation can help Acme understand effort and value before committing to broader rollout.”

4. **Create a shared decision path**
   - “If integration, value, budget, and procurement all need approval, we should make those steps visible now.”

### Avoid Saying

- “The integration will be easy.”
- “We can guarantee implementation by a specific date.”
- “Legal/security will approve this.”
- “We can discount to close.”
- “This will definitely produce a specific ROI.”

These require evidence, internal approval, or customer validation.
