## 4. Primary Objection Playbook

### Objection: “Integration effort may be too high”

#### Why They Say This

This is likely not just about engineering workload. It may mean:

- They have limited technical resources.
- They have been burned by previous integrations.
- Their route, order, fleet, or dispatch data may be messy.
- They are worried implementation will disrupt operations.
- Jordan may not control engineering priorities.

#### Response Framework

1. **Acknowledge**  
   “That is a fair concern. An API is only valuable if it can be implemented without creating a heavy lift for your team.”

2. **Reframe**  
   “Instead of treating integration as an unknown, let’s scope it with your technical team and define exactly what is required.”

3. **Evidence**  
   Use available proof points if you have them:
   - Average implementation timeline
   - API documentation
   - Sandbox access
   - Sample integration architecture
   - Prior customer implementation example  
   If unavailable, say: “I’ll bring our solutions engineer to map this with your team.”

4. **Confirm**  
   “If we can show that the integration can be completed within an acceptable timeline and resource commitment, would that remove the main concern?”

#### Example Response

“Jordan, that makes complete sense. Route optimization only works if it fits cleanly into your existing dispatch, order, or fleet workflow. Rather than guessing at the effort, I suggest we set up a 45-minute technical integration workshop with whoever owns your logistics systems. We can walk through your current workflow, required API calls, data inputs, expected outputs, and implementation steps. If the lift is too high, we will know quickly. If it is manageable, we can give you a realistic path forward. Who on your side would need to be in that conversation?”

#### If They Push Back

If Jordan says technical resources are unavailable:

“Understood. In that case, the best next step may be a lighter technical scoping call rather than a full implementation review. We can spend 30 minutes identifying the systems involved, likely data requirements, and any obvious blockers. That would help you decide whether it is worth involving engineering more deeply.”
