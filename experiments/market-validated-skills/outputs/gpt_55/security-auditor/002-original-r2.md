I can’t provide attack instructions or endpoint targeting for a production payment API without written authorization, defined scope, asset boundaries, and testing rules of engagement.

To proceed safely, please provide:

1. Written authorization approving security testing.
2. Scope: domains, API base URLs, endpoints, IP ranges, environments.
3. Testing boundaries: production vs. staging, allowed test types, rate limits, blackout windows.
4. Compliance requirements: PCI-DSS, SOC 2, GDPR, etc.
5. Contact and escalation path for incidents.
6. Data handling rules, especially for payment data and PII.
7. Success criteria and reporting format.

Correct next step: conduct a scoped, non-intrusive planning phase first. I can help you draft a Rules of Engagement document, an audit checklist, or a safe staging-environment test plan.

For production payment systems, start with non-invasive review activities until approval is in place:

- Architecture and data-flow review.
- Authentication and authorization design review.
- PCI-DSS control mapping.
- API inventory and asset classification.
- Logging, monitoring, and incident response review.
- Secrets management and key rotation review.
- CI/CD and dependency security review.
- Review of prior vulnerability scans and remediation status.

Once written authorization and scope are provided, I can help design a compliant, risk-based test plan.
